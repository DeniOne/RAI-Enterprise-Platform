# Ручной `end-to-end` тест: хозяйство «Казьминский» до техкарты

## Цель

Подтвердить, что система реально может довести **новое хозяйство, которого нет в базе**, до стадии готовой техкарты под `озимый рапс 2026/2027` по крупному полю `KZM-01` порядка `1000 га`.

## Участники сценария

| Роль | Кто это |
|---|---|
| Внутренний оператор | сотрудник RAI, который может работать с регистрацией и planning-контуром |
| Внешний гл. агроном хозяйства | человек со стороны хозяйства во `front-office` |
| Система | planning, `front-office`, `clarification`, `TechMap generation` |

## Шаг 1. Зарегистрировать контрагента

### Действие

Внутренний оператор использует
[05_chat_inputs_legal_and_setup.md](/root/RAI_EP/apps/api/test/real-farms/kazminskiy/05_chat_inputs_legal_and_setup.md)
для регистрации `party`.

### Ожидаемый результат

- в системе создан новый контрагент;
- хозяйство не смешано с уже существующими записями;
- карточка содержит корректное наименование и адрес.

### Blocker

- контрагент не создаётся;
- хозяйство ошибочно связывается с чужой записью;
- система не принимает реальные реквизиты.

## Шаг 2. Создать `CRM account`

### Действие

Внутренний оператор создаёт `account` хозяйства.

### Ожидаемый результат

- хозяйство получает рабочий `account`;
- дальше к нему можно привязать поле, сезон и пользователя `front-office`.

### Blocker

- `account` не создаётся;
- `party` и `account` остаются не связанными.

## Шаг 3. Создать поле `KZM-01`

### Действие

Внутренний оператор создаёт поле:

- код `KZM-01`
- площадь `1006 га`
- тип `CHERNOZEM`

Опора на
[02_field_passport_kzm-01.md](/root/RAI_EP/apps/api/test/real-farms/kazminskiy/02_field_passport_kzm-01.md).

### Ожидаемый результат

- поле создано и привязано к хозяйству;
- площадь и тип поля совпадают с тестовым пакетом.

### Blocker

- поле не создаётся;
- поле создаётся вне `account`;
- площадь теряется или искажается.

## Шаг 4. Создать сезон `2026/2027`

### Действие

Внутренний оператор создаёт сезон для `KZM-01`.

### Ожидаемый результат

- сезон существует;
- поле и сезон связаны;
- planning-контур готов к `HarvestPlan`.

### Blocker

- сезон нельзя связать с полем;
- сезон не виден в planning-контуре.

## Шаг 5. Пригласить внешнего гл. агронома хозяйства во `front-office`

### Действие

Внутренний оператор:

- создаёт контакт;
- создаёт приглашение;
- передаёт ссылку активации и `Telegram`-контур, если нужен.

### Ожидаемый результат

- внешний пользователь активирует доступ;
- у него появляется `front-office` контур именно своего хозяйства.

### Blocker

- приглашение не создаётся;
- пользователь не может активировать доступ;
- пользователь попадает не в свой внешний контур.

## Шаг 6. Первичный контекст от внешнего гл. агронома хозяйства

### Действие

Внешний пользователь отправляет стартовое сообщение из
[06_chat_inputs_agronomist_context.md](/root/RAI_EP/apps/api/test/real-farms/kazminskiy/06_chat_inputs_agronomist_context.md).

### Ожидаемый результат

- система принимает сообщение;
- создаётся thread / intake / draft;
- контекст не теряется;
- поле, сезон и культура могут быть связаны с диалогом.

### Blocker

- сообщение не маршрутизируется;
- система не видит хозяйство;
- диалог создаётся без usable context.

## Шаг 7. Дозапрос недостающих данных

### Действие

Система и внутренний контур запрашивают gaps, а внешний гл. агроном отвечает короткими сообщениями.

### Ожидаемый результат

- данные добираются по одному gap за раз;
- контекст становится богаче;
- система движется к readiness, а не гоняет пользователя по кругу.

### Blocker

- gaps не объясняются;
- система запрашивает бессвязный набор сведений;
- после ответов readiness не улучшается.

## Шаг 8. Создать `HarvestPlan`

### Действие

Внутренний planning-контур создаёт `HarvestPlan` для поля и сезона.

### Ожидаемый результат

- `HarvestPlan` создан;
- план привязан к нужному хозяйству и сезону;
- можно запускать генерацию `TechMap`.

### Blocker

- план не создаётся;
- сезон или хозяйство не подтягиваются;
- контекст не переносится в planning-контур.

## Шаг 9. Запустить генерацию техкарты

### Действие

Внутренний planning-контур запускает генерацию техкарты под озимый рапс.

### Ожидаемый результат

- создаётся `TechMap`;
- ветка — озимая;
- explainability показывает логику ветки и admission;
- обязательные `ControlPoint` по осенней розетке и перезимовке присутствуют.

### Blocker

- карта не создаётся;
- карта уходит не в ту культуру/ветку;
- карта игнорирует масштаб и ограничения поля.

## Шаг 10. Проверить explainability

### Действие

На detail-странице техкарты проверить:

- `generationStrategy`
- `cropForm`
- `canonicalBranch`
- `FieldAdmissionResult`
- `mandatory blocks`
- `shadow parity`
- `ControlPoint`

### Ожидаемый результат

- карта выглядит канонической, а не случайной;
- gaps и допущения читаются;
- нет ложной “идеальности”.

### Blocker

- explainability пустая;
- не видно, почему выбрана озимая ветка;
- не видно, какие данные были критичны.

## Критерии успеха всего прогона

Сценарий считается успешным, если:

1. новое хозяйство реально заведено в системе;
2. внешний гл. агроном хозяйства реально вошёл во `front-office`;
3. система реально приняла стартовый контекст;
4. gaps можно добрать без тотальной анкеты;
5. `HarvestPlan` и `TechMap` реально созданы;
6. техкарта выглядит правдоподобной для `Казьминского`, большого поля и озимого рапса.

## Что фиксировать по итогам ручного теста

| Что фиксировать | Пример |
|---|---|
| Где path прошёл | регистрация, invite, intake, generation |
| Где path сломался | например, gap-loop не довёл до readiness |
| Что было неочевидно пользователю | формулировка запроса, экран, статус |
| Что потребовало ручного вмешательства | привязка поля, сезона, контакта |
| Что надо чинить после теста | конкретный экран, маршрут или runtime-gap |
