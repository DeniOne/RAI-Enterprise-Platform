# Subsystem: Audit & Events 🛡️

> **Статус:** Канон | **Версия:** 1.0 | **Расположение:** Business Core

---

## 1. Назначение
Подсистема Аудита обеспечивает 100% прослеживаемость всех действий в системе. Это критически важно для безопасности (кто удалил поле?) и для агрономической точности (когда реально была закрыта задача?).

---

## 2. Архитектура "Отказоустойчивого Аудита"

В RAI EP реализована многоуровневая система сохранения событий:

1. **Memory Buffer**: Мгновенная фиксация события в коде.
2. **Primary Storage**: Запись в таблицу `AuditLog` (PostgreSQL) в рамках транзакции.
3. **Emergency Storage**: Если БД недоступна или транзакция упала, событие сохраняется в таблицу `AuditFailure` или Redis Buffer.
4. **Retry Logic**: Система делает до 3 попыток пересохранения "проваленных" событий.

---

## 3. Структура события (Event Schema)

```typescript
interface AuditEvent {
  id: string;          // UUID
  timestamp: Date;     // Когда произошло
  userId: string;      // Кто сделал
  companyId: string;   // В рамках какой компании
  action: string;      // Тип действия (CREATED, UPDATED, DELETED, COMPLETED)
  entityType: string;  // К чему относится (Season, Field, Task)
  entityId: string;    // ID объекта
  payload: Json;       // Что именно изменилось (дифф или снапшот)
  metadata: Json;      // Контекст (IP, браузер, версия бота)
}
```

---

## 4. Ключевые типы событий

- **SECURITY.LOGIN**: Вход в систему.
- **AGRO.SEASON_CLOSED**: Финализация сезона (триггер для снапшота).
- **AGRO.FIELD_GEOMETRY_CHANGED**: Изменение границ поля (критично для площадей).
- **CORE.ACCESS_DENIED**: Попытка несанкционированного доступа (403).

---

## 5. Использование данных
Аудит используется не только для "разборок", но и для:
- Построения отчетов по активности персонала.
- Восстановления данных в случае ошибки пользователя.
- Обучения ИИ на основе "правильных" действий агрономов.
