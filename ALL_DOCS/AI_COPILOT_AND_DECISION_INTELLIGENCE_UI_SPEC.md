# AI Copilot and Decision Intelligence UI Spec

## 1. Цель

Система должна убрать конфликт `AI vs UI`.

Правило:
- пользователь работает в доменном экране
- AI помогает в контексте экрана
- чат не заменяет рабочие интерфейсы

Это означает:
- AI Dock становится `copilot`, а не "второй продукт"
- память показывается только там, где она повышает доверие к ответу
- экспертные агенты вызываются контекстно
- `Стратегия -> Прогнозы` становится рабочим местом решений, а не лентой ответов

## 2. UX-принципы

### 2.1. No Technical Noise

Пользователю нельзя показывать технические термины по умолчанию:
- `engram`
- `episodic recall`
- `memory tier`
- `prompt assembly`

Вместо этого интерфейс показывает смысл:
- "Учтен похожий кейс прошлого сезона"
- "Учтены текущие отклонения"
- "Есть рискованный исторический паттерн"

### 2.2. Progressive Disclosure

Первый слой UI короткий и прикладной.
Раскрытие деталей - только по клику, и только в тех ролях, где это полезно.

### 2.3. Context Beats Chat

Кнопка вызова AI должна находиться рядом с задачей:
- на техкарте
- на карточке отклонения
- в стратегии
- в прогнозах

Не нужно заставлять пользователя идти в отдельный чат ради каждой операции.

### 2.4. Explain, Then Escalate

Сначала система объясняет ответ коротко.
Если уровень уверенности низкий или риск высокий, система предлагает escalation:
- `Запросить экспертное заключение`
- `Передать на проверку`
- `Сформировать сценарий`

## 3. Информационная архитектура

### 3.1. Sidebar

Изменение верхнего уровня не требуется.

Используем текущий канон:
- `Управление Урожаем`
- `Стратегия`
- `Экономика`
- `Финансы`

Изменение внутри раздела `Стратегия`:
- `Стратегический обзор`
- `Портфель планов`
- `Карта рисков`
- `Прогнозы`
- `Журнал стратегических решений`

`Прогнозы` - отдельный рабочий экран для cross-domain decision intelligence.

## 4. AI Dock

### 4.1. Роль

AI Dock = постоянный copilot-слой.

Он должен:
- отвечать кратко
- объяснять, почему ответ дан
- подсказывать следующий шаг
- предлагать escalation к expert-agent, если задача выходит за пределы обычного совета

Он не должен:
- быть единственным способом работы с системой
- дублировать стратегические и табличные экраны
- показывать внутреннюю телеметрию памяти всем пользователям

### 4.2. Минимальное объяснение памяти

Под ответом допускается только одна короткая строка:
- `Учтен похожий кейс прошлого сезона`
- `Учтены активные отклонения по полю`
- `Учтены предпочтения и политика компании`

Поведение:
- строка показывается только если память реально повлияла на ответ
- раскрытие деталей доступно по `Почему этот ответ?`
- privileged-view может показать source type и confidence

### 4.3. Suggested Actions

Под ответом AI Dock может показывать не более 3 действий:
- `Открыть техкарту`
- `Запросить экспертное заключение`
- `Построить прогноз`

Если confidence низкий или risk tier высокий, `Запросить экспертное заключение` поднимается на первое место.

## 5. Mega-Agronomist / `chief_agronomist`

### 5.1. Правило вызова

`chief_agronomist` не должен жить как отдельный постоянный чат.

Он вызывается контекстно:
- из техкарты: `Запросить экспертное заключение`
- из отклонения/алерта: `Эскалировать к Мега-Агроному`
- из AI Dock: suggested action, если риск высокий или уверенность низкая

### 5.2. Формат окна

Первый релиз:
- `drawer` или `takeover panel`
- не отдельный split-screen workspace

Структура окна:
- Header: поле, культура, сезон, причина вызова
- Block 1: `Вердикт`
- Block 2: `Что делать сейчас`
- Block 3: `Альтернативы`
- Block 4: `На чем основано`
- Footer actions:
  - `Принять`
  - `Передать человеку`
  - `Создать задачу`

### 5.3. Почему это не overkill

Это не еще один чат.
Это специализированный экспертный review surface для high-stakes кейсов.

## 6. Strategy -> Forecasts

### 6.1. Назначение

`Стратегия -> Прогнозы` - это единый рабочий экран прогнозирования и выбора решений по всем аспектам бизнеса:
- агрономия
- экономика
- финансы
- риски
- операции
- портфельные решения

### 6.2. Основные режимы экрана

- `Прогноз`
- `Факторы`
- `Сценарии`
- `Оптимизация`
- `Риски`

### 6.3. Базовая компоновка

Верхняя панель:
- хозяйство
- поле/кластер
- культура/бизнес-направление
- сезон/период
- горизонт прогноза
- сценарий

Центральная область:
- `Базовый сценарий`
- `Измененный сценарий`
- `Рекомендация`

Правая панель:
- assistant panel от `data_scientist`
- объяснение расчета
- сбор допущений
- генерация отчета

Нижняя область:
- диапазон неопределенности
- ключевые драйверы
- ограничения
- trace / evidence summary

### 6.4. Что пользователь должен увидеть без прокрутки

- ключевой KPI
- диапазон
- риск-tier
- recommended action
- главный tradeoff

### 6.5. Что не надо делать

Не надо превращать экран в:
- ленту "умных мыслей"
- бесконечный список графиков
- интерфейс для data scientist-engineer

Это экран для руководителя и аналитика принятия решения.

## 7. Control Tower

### 7.1. Memory Fabric

Для ops/admin в `Control Tower` добавляется отдельный блок `Memory Fabric`.

Показывает:
- active layers
- recall latency
- episode count
- engram count
- hot alert recall
- consolidation freshness
- pruning/archive status
- trust score

Это не пользовательская функция. Это operational surface.

### 7.2. Memory Lane in Trace

В forensic trace добавляется шаг памяти:
- что было доступно к recall
- что было реально использовано
- что было отброшено
- почему ответ был escalated или downgraded

Это только для ops/admin и не является MVP-обязательством.

## 8. Agent Registry

Agent Registry становится не просто списком ролей, а `runtime contract inspector`.

Для каждого агента показываются:
- execution path
- autonomy mode
- capabilities
- tools
- memory policy
- output contract
- governance policy
- visibility and tenant scope

Правило показа:
- базовый summary по умолчанию
- детали только в раскрывающихся секциях

## 9. Visibility Matrix

| Surface | Business user | Manager | Executive | Ops/Admin |

| AI Dock memory hint | Yes | Yes | Yes | Yes |
