# PLAN — UX шлифовка Dock/Focus, хоткеев и стабильности
Дата: 2026-03-02  
Статус: draft  
Decision-ID: AG-UX-POLISH-001  

## Результат (какой артефакт получим)
- Исполнимый план для `P2.3`: довести текущий web-чат и смежный core-flow до более управляемого UX-состояния через минимальные режимы `Dock/Focus`, базовые хоткеи и устранение конкретных источников нестабильности.
- Зафиксированный frontend-scope без визуального расползания: изменения только в `apps/web`, только вокруг чата/виджетов/core-flow, без “редизайна ради редизайна”.
- Явное соответствие `AG-UX-POLISH-001`: только после подтверждённого `P0/P1`, без тяжёлых анимаций, без регрессий `чат -> виджеты -> Draft→Commit`.

## Границы (что входит / что НЕ входит)
- Входит: анализ текущего UX-контура `apps/web` для AI-чата, виджетов и связанных панелей, чтобы выделить 3-5 реально дорогих UX-узких мест.
- Входит: минимальные состояния `Dock/Focus` для чата/виджетов, если они опираются на уже существующий layout/store и не требуют архитектурного переписывания shell.
- Входит: минимальный набор хоткеев для ускорения работы:
  - фокус ввода;
  - отправка;
  - toggle панелей / режимов `Dock/Focus`.
- Входит: устранение top-3 причин нестабильности в целевом контуре, если они воспроизводимы по коду/прогону и лежат в рамках чата/виджетов/core-flow.
- Входит: smoke-проверка `чат + Dock/Focus + виджеты + Draft→Commit` без регрессий.
- Не входит: тяжёлые анимации, визуальные эксперименты, переписывание дизайн-системы, расширение scope на весь frontend, новые backend API ради UX-полировки.
- Не входит: изменение tenant-модели, любые payload с `companyId` из UI, или поломка существующего канона `Geist` / light theme / `font-medium` вместо bold.

## Риски (что может пойти не так)
- Есть риск ложной “полировки”, когда под видом UX-улучшений начнётся переработка layout и визуального языка вне scope.
- Есть риск регрессии core-flow: хоткеи и состояния `Dock/Focus` могут конфликтовать с текущей логикой чата, виджетов и Draft→Commit.
- Есть риск нестабильности state-management, если новые UI-состояния будут размазаны по нескольким store/компонентам без единой точки управления.
- Есть риск нарушения UI canon: особенно через тяжёлые эффекты, тёмные паттерны, лишнюю визуальную плотность и несогласованную типографику.
- Есть риск, что “top-3 причины нестабильности” окажутся backend/transport-проблемами, а не UX-проблемами; в таком случае их нужно зафиксировать как blocker, а не чинить вне scope.

## План работ (коротко, исполнимо)
- [ ] Подтвердить фактические точки UX-боли в `apps/web`: AI-chat panel, widgets rail, связанные переключатели/панели и сценарий `Draft→Commit`.
- [ ] Зафиксировать 3-5 самых дорогих UX-узких мест по текущему коду и воспроизводимым сценариям, не выдумывая новые проблемы.
- [ ] Спроектировать минимальную модель состояний `Dock/Focus`: где хранится состояние, какие режимы допустимы, как они влияют на layout без визуальной перегрузки.
- [ ] Спроектировать минимальный набор хоткеев с безопасным поведением и без конфликта с вводом текста и браузерными дефолтами.
- [ ] После `ACCEPTED` реализовать `Dock/Focus` и хоткеи в существующем web-контуре, не ломая `widgets` и основной чатовый поток.
- [ ] После `ACCEPTED` устранить top-3 причины нестабильности внутри целевого UI-контура либо явно зафиксировать blocker, если проблема вне frontend-scope.
- [ ] После `ACCEPTED` выполнить smoke-проверку `чат (ввод/отправка) -> toggle Dock/Focus -> widgets render -> Draft→Commit` и собрать evidence для review packet.

## DoD
- [ ] Режимы `Dock/Focus` работают в целевом чат-контуре без визуальной деградации.
- [ ] Хоткеи работают и не ломают ввод/отправку сообщений.
- [ ] Core-flow `чат, виджеты, Draft→Commit` проходит smoke без регрессий.
- [ ] Нет тяжёлых анимаций и нарушений UI canon.
- [ ] Зафиксированы и устранены top-3 UX-причины нестабильности в пределах scope либо явно описаны blocker-ограничения.
