# PLAN — Каноническая схема widgets[] и renderer в web-чате
Дата: 2026-03-02  
Статус: draft  
Decision-ID: AG-WIDGETS-SCHEMA-RENDERER-001  

## Результат (какой артефакт получим)
- Исполнимый план для P1.2: ввести версионируемую каноническую схему `widgets[]` в `apps/api` и отрисовку минимум 2 типов виджетов справа в `apps/web` для ответов `POST /api/rai/chat`.
- Зафиксированный scope e2e: API DTO/типы, web store без `any` в критичном контуре виджетов, renderer по `type`, fallback для неизвестного типа.
- Явное соответствие `AG-WIDGETS-SCHEMA-RENDERER-001`: tenant-safe данные виджетов, no eval, минимальные 2 виджета без переписывания всего чата.

## Границы (что входит / что НЕ входит)
- Входит: проектирование discriminated union для `widgets[]` с явной версией (`schemaVersion` и/или `type + version`), обновление `RaiChatResponseDto`, возврат 2 канонических типов виджетов из `apps/api/src/modules/rai-chat/*`.
- Входит: обновление `apps/web/lib/stores/ai-chat-store.ts` и связанных компонентов чата так, чтобы `widgets` стали первичным typed-контрактом, а правая панель рендерила минимум `DeviationList` и `TaskBacklog`.
- Входит: безопасный fallback на неизвестный тип виджета без падения UI.
- Не входит: полный рефакторинг всего AI-chat UX, массовое добавление всех будущих виджетов, выполнение кода из payload, перенос tenant identity из пользовательского запроса.
- Не входит: расширение backend за пределы `rai-chat` и минимально необходимого источника данных для 2 виджетов; при отсутствии готовых доменных данных допустим контролируемый mock/proof path, если это не нарушает Decision scope.

## Риски (что может пойти не так)
- В текущем `apps/web/lib/stores/ai-chat-store.ts` `widgets` временно мапятся в `suggestedActions`, а `suggestedActions` типизированы через `Record<string, any>[]`; это может потребовать аккуратного разрыва обратной совместимости между store и компонентами.
- В `apps/api` сейчас есть только заглушка `Last24hChanges`, поэтому для `DeviationList` и `TaskBacklog` нужно либо найти уже существующие доверенные источники данных, либо зафиксировать ограниченный mock/proof path без архитектурного разрастания.
- Есть риск смешения UI-схемы и доменной логики: payload виджета должен быть только структурированными данными для рендера, без вычислений на стороне клиента.
- Есть tenant/security-риск, если payload виджета начнёт отражать `companyId` из пользовательского ввода, а не из доверенного серверного контекста.
- Есть риск дрейфа схемы между `apps/api` и `apps/web`, если не вынести единый явный discriminated union или не синхронизировать контракты вручную.

## План работ (коротко, исполнимо)
- [ ] Просмотреть текущие точки интеграции `rai-chat` в `apps/api` и `apps/web`: DTO ответа, zustand store, компоненты панели/сообщений, чтобы зафиксировать место для typed widgets renderer.
- [ ] Спроектировать каноническую схему `widgets[]` как discriminated union с явной версией и typed payload для минимум 2 типов: `DeviationList` и `TaskBacklog`.
- [ ] Определить серверный источник данных для 2 виджетов: использовать существующие доверенные данные из `apps/api` либо минимальный proof/mock path, если реального источника нет и это не нарушает scope решения.
- [ ] Обновить `apps/api/src/modules/rai-chat/*`: убрать `any` из widgets-контракта, формировать `widgets[]` по канонической схеме и не включать tenant identity из пользовательского payload.
- [ ] Обновить `apps/web/lib/stores/ai-chat-store.ts`: заменить временное отображение `data.widgets -> suggestedActions`, ввести typed хранение `widgets` и сохранить совместимость истории сообщений.
- [ ] Реализовать renderer в web-чате по `type` с минимум 2 компонентами виджетов справа и безопасным fallback для неизвестного типа.
- [ ] Подготовить проверки e2e уровня промта: API-ответ содержит канонические `widgets[]`; web renderer показывает оба типа; неизвестный тип не ломает UI.
- [ ] После реализации собрать review packet с diff, логами прогонов и кратким описанием/скрином сценария `чат -> 2 виджета справа`.

## DoD
- [ ] `widgets[]` в `rai-chat` имеют явную версию и discriminated union без `any` в критичном контуре.
- [ ] Минимум 2 виджета (`DeviationList`, `TaskBacklog`) проходят e2e путь `API -> web renderer`.
- [ ] Неизвестный `type` не ломает UI и обрабатывается fallback-логикой.
- [ ] `companyId` не передаётся в payload виджета из пользовательского ввода и берётся только из доверенного backend context.
