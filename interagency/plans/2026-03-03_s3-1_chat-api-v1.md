Decision-ID: AG-CHAT-API-V1-001

## Результат
- Контракт `POST /api/rai/chat` будет формализован до `v1`: `RaiChatResponseDto` начнёт явно поддерживать `toolCalls[]` и `openUiToken`.
- `RaiChatService` будет возвращать выполненные typed tool calls в ответе без изменения tenant-модели и без затрагивания `workspaceContext`.
- Появится тестовое подтверждение, что `toolCalls`, `traceId` и `threadId` присутствуют в ответе при сценарии с вызовом инструментов.

## Границы
- Входит: изменения только в `apps/api/src/modules/rai-chat/dto/rai-chat.dto.ts`, `apps/api/src/modules/rai-chat/rai-chat.service.ts`, `apps/api/src/modules/rai-chat/rai-chat.service.spec.ts`.
- Входит: минимальная подготовка поля `openUiToken` как опционального поля контракта без сложной бизнес-логики его генерации.
- Не входит: изменение `workspaceContext`, изменение tenant isolation, изменение `companyId` flow, добавление новых контроллеров, изменение transport path `POST /api/rai/chat`.
- Не входит: string-execution, произвольные tool payloads вне уже существующего typed tooling.

## Риски
- Есть риск расхождения между “выполненными инструментами” и “запрошенными инструментами”, если в ответ попадут не реально исполненные tool calls, а исходный request payload.
- Есть риск сломать DTO-валидацию, если `toolCalls` будут оформлены без корректных декораторов `class-validator` и `class-transformer`.
- Есть риск скрытого расширения scope, если начать придумывать логику генерации `openUiToken`; по плану это поле должно остаться пустым, но типизированным.
