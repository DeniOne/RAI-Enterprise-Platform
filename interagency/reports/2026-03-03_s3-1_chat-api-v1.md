# REPORT — S3.1 Chat API v1
Дата: 2026-03-03  
Статус: final  

## Что было целью
- Формализовать `v1`-контракт `POST /api/rai/chat`.
- Добавить в `RaiChatResponseDto` поля `toolCalls` и `openUiToken`.
- Подтвердить тестом, что ответ возвращает выполненные typed tool calls вместе с `traceId` и `threadId`.

## Что сделано (факты)
- В [rai-chat.dto.ts](/root/RAI_EP/apps/api/src/modules/rai-chat/dto/rai-chat.dto.ts) расширен `RaiChatResponseDto`:
  - добавлено поле `toolCalls?: RaiToolCallDto[]` с декораторами `class-validator` и `class-transformer`;
  - добавлено поле `openUiToken?: string` с ограничением длины.
- В [rai-chat.service.ts](/root/RAI_EP/apps/api/src/modules/rai-chat/rai-chat.service.ts) ответ `handleChat(...)` теперь включает список фактически выполненных инструментов, собранный из `executedTools`.
- `openUiToken` добавлен как опциональное поле контракта без отдельной логики активации.
- В [rai-chat.service.spec.ts](/root/RAI_EP/apps/api/src/modules/rai-chat/rai-chat.service.spec.ts) обновлён тест на формальный ответ: проверяются `toolCalls`, `traceId`, `threadId`, `openUiToken`.

## Что сломалось / что не получилось
- Первый прогон теста упал из-за неверного ожидания в assertion: фактический результат `workspace_snapshot` содержит `route`, `lastUserAction`, `hasSelection`, а не придуманные поля. Тест приведён к реальному typed contract.
- Manual check не требовался и не выполнялся.

## Что сделано дополнительно
- В ответ возвращаются именно фактически исполненные инструменты, а не сырой входной `request.toolCalls`.
- Для примитивных non-object результатов предусмотрена безопасная упаковка в `{ result: ... }`, чтобы не ломать `payload: Record<string, unknown>`.
