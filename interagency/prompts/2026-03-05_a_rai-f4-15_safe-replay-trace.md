# PROMPT — Safe Replay Trace (Phase 4.15)
Дата: 2026-03-05
Статус: active
Приоритет: P2

## Цель
Реализовать механизм "Safe Replay Trace" — возможность повторно прогнать конкретный сеанс (traceId) в изолированном READ-ONLY режиме. Это необходимо для дебага инцидентов и проверки, как изменённый промпт или новая модель справится с тем же самым контекстом, не создавая сайд-эффектов (например, не отправляя реальное письмо и не списывая деньги).

## Контекст
- **Связанные документы:**
  - `docs/00_STRATEGY/STAGE 2/A_RAI_IMPLEMENTATION_CHECKLIST.md` (Phase 4, пункт 4.3: Safe Replay Trace).
- У нас есть `AiAuditEntry`, где сохранены все шаги и входящие данные (включая payload). 
- Идея в том, чтобы запустить SupervisorAgent'а с флагом `isReplay=true`, при котором все инструменты, меняющие состояние (WRITE), либо игнорируются, либо мокаются успехом.

## Ограничения
- Изоляция обязательна: никаких реальных мутаций в режиме Replay. RiskPolicy/ToolsRegistry должны блокировать WRITE-операции.
- Replay должен быть доступен только администраторам системы (или через специальный эндпоинт).

## Задачи (что сделать)
- [ ] Добавить поддержку `replayMode: boolean` в `WorkspaceContext` или аргументы `SupervisorAgent.execute`.
- [ ] Доработать `RaiToolsRegistry` (или базовый класс Tool) так, чтобы при флаге `replayMode` инструменты с `actionType === 'WRITE'` (или схожие) не выполнялись физически, а возвращали заглушку (mock success).
- [ ] Создать API-эндпоинт (например, `POST /rai/explainability/trace/:traceId/replay`) в `ExplainabilityPanelController` (или `RaiChatController`), который:
  1. Вытаскивает исходный инпут из базы (первый `AiAuditEntry` трейса).
  2. Запускает оркестратор в replay режиме.
  3. Возвращает новый `traceId` реплея и результат.
- [ ] Покрыть логику запуска реплея и блокировки мутаций unit-тестами.

## Definition of Done (DoD)
- [ ] Механизм Safe Replay реализован: WRITE-операции блокируются/мокаются.
- [ ] Эндпоинт для запуска реплея по старому `traceId` работает.
