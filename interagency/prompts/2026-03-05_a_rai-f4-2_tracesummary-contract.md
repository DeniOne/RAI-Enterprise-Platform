# PROMPT — TraceSummary Data Contract v1 (Phase 4.2)
Дата: 2026-03-05
Статус: active
Приоритет: P0

## Цель
Зафиксировать и реализовать схему `TraceSummary Data Contract v1` в БД (Prisma) и DTO. Это фундаментальный слой телеметрии для AI: сбор токенов, времени, версий промптов/моделей/инструментов и заполнителей под метрики качества (evidenceCoverage, invalid%, BS%).

## Контекст
- **Почему это важно сейчас:** Без нормального контракта данных дашборды, расчет честности (BS%) и биллинг будут сосать хуй. Это единый источник правды для метрик качества ответов AI-роя.
- **На какие документы опираемся:**
  - `docs/00_STRATEGY/STAGE 2/A_RAI_IMPLEMENTATION_CHECKLIST.md` (Phase 4, пункт 2).

## Ограничения (жёстко)
- **Security / tenant isolation:** Все записи телеметрии обязаны иметь `companyId`. Ничего глобального.
- **Backward Compatibility:** Обновление схемы БД не должно сломать существующие трейсы (`AiAuditEntry` / `traceId`). Если нужно — расширяем текущую модель или создаем связанную `TraceSummary` (one-to-one к `AiAuditEntry`).
- **Сайд-эффекты:** Эта задача — строго модели данных и сбор метрик на лету. Никаких изменений бизнес-логики агентов.

## Задачи (что сделать)
- [ ] Определить модель `TraceSummary` в Prisma схеме (связь с `traceId` / `companyId`).
- [ ] Добавить обязательные поля телеметрии: `totalTokens`, `promptTokens`, `completionTokens`, `durationMs`.
- [ ] Добавить поля версионирования: `modelId`, `promptVersion`, `toolsVersion`, `policyId`.
- [ ] Добавить placeholder-поля качества (будут обновляться асинхронно): `evidenceCoveragePct`, `invalidClaimsPct`, `bsScorePct`.
- [ ] Создать Zod DTO `TraceSummaryDto` для API-ответов.
- [ ] Написать сервис или функцию для аккумулирования этих данных в конце обработки агента.
- [ ] Сгенерировать миграции Prisma (`db push` / `migrate dev` — в зависимости от вашего локального пайплайна, обычно `db push`).
