# Концепция: Agent-First Architecture & Superbase (Agent_RAI) 🤖🏛️

## 1. Введение
В 2026 году традиционные CRM-интерфейсы с сотнями полей становятся обузой. Проект RAI_EP переходит на **Agent-First** модель, где основным способом взаимодействия является чат с ИИ-агентом (**Agent_RAI**), а графический интерфейс (UI) становится вспомогательным, ситуативным и реактивным.

---

## 2. Ключевые компоненты

### 2.1 Agent_RAI (Оркестратор-Мозг)
`Agent_RAI` — это не просто чат-бот, это центральный управляющий узел системы.
- **Интерпретация**: Понимает намерения пользователя (интенты) через естественный язык.
- **Recall (Память)**: Обращается к системе Энграмм для получения контекста прошлых успехов и ошибок.
- **Advisory (Валидация)**: Проверяет каждое действие через Shadow Advisory Service на соответствие рискам.
- **Execution (Исполнение)**: Напрямую вызывает доменные сервисы для проведения операций (регистрация, планирование, аудит).

### 2.2 Superbase (Гибкое Хранилище)
Традиционные жесткие схемы БД заменяются на модель "Супербазы".
- **UniversalEntity**: Универсальная модель данных, использующая `JSONB` для хранения атрибутов. 
- **Динамическая схема**: Агент может добавлять новые реквизиты сущностям (например, новые параметры почвы или специфические реквизиты контрагента) без проведения миграций БД.
- **Метаданные**: Каждое поле в Супербазе имеет метаданные о провенансе (кто и когда заполнил, уровень уверенности AI).

### 2.3 Система Энграмм (Когнитивная память)
Энграммы — это "снимки опыта", которые позволяют агенту:
- **Positive (+)**: Рекомендовать проверенные решения.
- **Negative (-)**: Блокировать действия, которые ранее приводили к сбоям.
- **Ситуативность**: Агент "вспоминает" нужную информацию именно в момент совершения действия.

### 2.4 Situational UI (Реактивный фронтенд)
Фронтенд перестает быть набором статичных страниц.
- **UI по требованию**: Агент отправляет сигнал фронтенду отрисовать конкретный виджет (карту, мини-форму, график).
- **Фронт-офис**: Основной упор делается на операционную работу "в поле" через мобильные и чат-интерфейсы.
- **Сквозная навигация**: Переход от чата к детальному UI происходит мгновенно по ссылке-токену.

### 2.5 Legal & Compliance (Юридический комплаенс)
Агент интегрируется с внешними базами данных для анализа рисков:
- **Базы судов/приставов**: Проверка наличия активных арбитражных дел и исполнительных производств.
- **Риск-профиль**: Формирование оценки благонадежности, которая напрямую влияет на условия генерируемого договора (например, требование 100% предоплаты при высоких рисках).
- **Авто-дисквалификация**: Блокировка регистрации при обнаружении критических стоп-факторов (банкротство, ликвидация).

---

## 3. Сценарий работы: Регистрация Контрагента

1. **User**: "Зарегистрируй новое ООО 'АгроСвет' в Краснодаре, ИНН 1234567890".
2. **Agent_RAI**:
   - Вызывает `CounterpartyLookupProvider` для обогащения данных.
   - **Legal Check**: Пробивает по базам ФССП и Арбитража. Находит 3 активных иска по неисполнению обязательств.
   - Проверяет `ShadowAdvisory`: выдает статус `REVIEW` из-за судебных рисков.
   - **Agent_RAI**: "Контрагент ООО 'АгроСвет' найден. Внимание: обнаружено 3 судебных иска на 5.4 млн руб. Рекомендую работать только по предоплате. Регистрируем с этим условием?"
3. **User**: "Да, регистрируй".
4. **Agent_RAI**: Сохраняет данные в `UniversalEntity` с пометкой о высоком риске и спец-условием для договоров.

---

## 4. Инфраструктурные изменения

- **Backend**: Внедрение `AgentModule`, расширение `MemoryModule`.
- **Database**: Миграция на `JSONB` структуры для `UniversalEntity`.
- **Frontend**: Создание реестра ситуативных компонентов (`apps/web/src/components/situational/*`).
- **Telegram**: Расширение хендлеров для поддержки сложных диалоговых деревьев.

---

## 5. Преимущества
1. **Скорость**: Регистрация сущностей в 5-10 раз быстрее, чем через заполнение форм.
2. **Гибкость**: Система адаптируется под любые новые требования без переписывания кода.
3. **Безопасность**: Каждое действие ИИ-агента логируется и проверяется системой рисков.
