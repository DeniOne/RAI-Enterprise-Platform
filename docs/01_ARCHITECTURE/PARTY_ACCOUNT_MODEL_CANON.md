# Канон модели Party vs Account

## 1. Назначение

Этот документ фиксирует единую модель сосуществования сущностей `Party` и `Account` в RAI EP на текущем этапе продукта.

Цель: исключить двусмысленность в UX, агентных командах и backend-интеграциях.

## 2. Канонические роли сущностей

### `Party` (таблица `commerce_parties`)

- Юридически значимый контрагент.
- Контур: `Commerce / Party Management`.
- Хранит регуляторно-юридическую информацию:
  - наименование,
  - юрисдикцию,
  - regulatory profile,
  - `registrationData`,
  - граф связей `PartyRelation`,
  - права на активы через `AssetPartyRole`.
- Используется как источник истины для договоров, комплаенса, ownership-структуры.

### `Account` (таблица `accounts`)

- Операционный CRM/агро-контур "хозяйства/рабочего аккаунта".
- Контур: `CRM + Consulting execution`.
- Хранит операционный профиль:
  - имя хозяйства,
  - ИНН (если есть),
  - статусы/риск,
  - holding,
  - CRM-активности (`contacts/interactions/obligations`).
- Является обязательной связкой для текущих агро-процессов:
  - `Field.clientId -> Account.id`,
  - `HarvestPlan.accountId -> Account.id`,
  - часть front-office привязок.

## 3. Модель сосуществования (текущее состояние)

1. `Party` и `Account` существуют **параллельно**, но больше не живут как несвязанные двойники.
2. В основной схеме сохраняется запрет на hard relation `Account -> Party` через Prisma `@relation`.
3. Для runtime введён стабильный soft-link `accounts.partyId` + индекс `accounts(companyId, partyId)`.
4. `Party` является master для юридических/регистрационных данных, а `Account` — operational projection для CRM/агро-контура.
5. Прикладная логика больше не должна резолвить связь только по ИНН/названию, если `partyId` уже установлен.
6. Runtime CRM допускает автопровижининг `Account` из существующего `Party`, но после провижининга связь должна закрепляться через `partyId`.

## 4. Обязательные правила потока

1. Для регистрации контрагента и юридических данных создаётся/обновляется `Party`.
2. После создания/обновления `Party` выполняется обязательная проекция в `Account`:
   - проставляется `account.partyId`,
   - синхронизируются `inn / jurisdiction / status / type`,
   - контакты из `Party.registrationData.contacts` и `meta.managerName` проецируются в CRM `Contact`.
3. Для операционного CRM и техкарты используется `Account`.
4. Изменения master-полей из CRM (`name -> shortName`, `inn`, `jurisdiction`) должны писаться обратно в связанный `Party`.
5. В реальном E2E потоке `контрагент -> поле -> сезон -> план -> техкарта` переход должен быть:
   - `Party` создан/найден,
   - `Account` создан/разрешён и связан через `partyId`,
   - далее все агро-сущности работают через `accountId`.

## 5. Что запрещено

1. Считать `Party` и `Account` взаимозаменяемыми.
2. Держать связь между ними только на эвристиках по ИНН/названию, если `partyId` уже известен.
3. Писать в UI/агенте, что "карточка контрагента не найдена", если `Party` есть, но отсутствует `Account`.
4. Строить техкарту напрямую от `Party` без этапа разрешения `Account`, пока агро-схема завязана на `accountId`.

## 6. Норматив для UI и агента

1. В UI явно разделяются термины:
   - `Контрагент (юрлицо)` = `Party`,
   - `Хозяйство / операционный аккаунт` = `Account`.
2. Команды агента на открытие карточки по названию контрагента обязаны:
   - сначала искать CRM `Account`,
   - при отсутствии резолвить по `Party`,
   - при необходимости создавать `Account`,
   - закреплять связку через `partyId`,
   - после этого открывать workspace.
3. Ответы на вопросы про директора/реквизиты должны сначала выдавать факт в чате, а окно результата использовать как следующий шаг, а не как замену ответа.

## 7. Ожидаемый эффект

- У пользователя исчезают ложные ошибки "карточка не найдена" при наличии контрагента.
- Юридические реквизиты и директор перестают расходиться между `Party` и CRM workspace.
- Команда работает по одному контракту терминов.
- E2E цепочка подготовки техкарты становится предсказуемой и повторяемой.
