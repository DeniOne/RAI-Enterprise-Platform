# MEMORY STORAGE CANON (S5.2)

## 0. Статус документа
Данный документ фиксирует архитектурные стандарты хранения памяти агента, уровни хранения (S/M/L Tiers) и структуру данных "Carcass + Flex".
- Обязателен для любых реализаций `MemoryAdapter` и работы с долговременной памятью.
- Нарушение = PROCESS HALT.

---

## 1. Трехуровневая модель памяти (Tiered Storage)

Архитектура памяти агента строится на консолидации данных с течением времени, чтобы избежать "token inflation" и разрастания базы.

### S-Tier: Short-term (Raw Logs)
- **Сущность**: Необработанные взаимодействия (User Message + Agent Response + Tool Calls).
- **Назначение**: Контекст текущей сессии чата, точное восстановление истории диалога.
- **Хранение**: Векторная БД или реляционная таблица логов (`MemoryInteraction`).
- **Retention (Политика)**: Ограниченный срок жизни (например, 6-12 месяцев или N сообщений на сессию). Старые логи удаляются или отправляются в холодный архив.

### M-Tier: Middle-term (Episodes)
- **Сущность**: Консолидированные выжимки из S-Tier.
- **Назначение**: Семантический поиск прошлых решений, фактов и контекста ("Мы это уже обсуждали в прошлом месяце: решили сделать так-то").
- **Хранение**: Векторная БД (`MemoryEpisode`).
- **Retention (Политика)**: Хранятся бессрочно ("вечная память"), но могут инвалидироваться при явном конфликте фактов или по запросу пользователя ("забудь это").

### L-Tier: Long-term (Profile & Knowledge)
- **Сущность**: Структурированные профили (пользователя, системы, проекта) и глобальные факты.
- **Назначение**: Неявный контекст, подаваемый агенту при каждом запуске (preferences, roles, rules).
- **Хранение**: Документное/JSON-хранилище (`MemoryProfile`).
- **Retention (Политика)**: Хранятся бессрочно. Обновляются патчами (Create/Update/Delete).

## 2. Принцип Carcass + Flex для метаданных

Чтобы избежать "зоопарка таблиц" при добавлении новых фичей памяти, используется единая модель расширения.

### Обязательный каркас (Carcass)
Фиксированные колонки в реляционных/векторных таблицах:
- `id` (PK)
- `companyId` (Tenant Isolation, обязательно)
- `userId` / `sessionId` (Скоуп)
- `content` (Сам текст для эмбеддингов/поиска)
- `embedding` (Вектор)
- `createdAt` / `updatedAt` / `updatedBy`

### Гибкая часть (Flex JSONB)
Любые специфичные метаданные хранятся в поле `attrs` (JSONB) с обязательной валидацией по `schemaKey`.

**Обязательные поля внутри Flex-структуры**:
- `schemaKey`: Строковой идентификатор схемы (например, `"memory.episode.v1"`).
- `provenance`: Источник знания (откуда агент это узнал: `"user-chat"`, `"system-inference"`, `"external-signal"`).
- `confidence`: Оценка уверенности агента в факте (0.0 - 1.0).

*Пример*:
```json
{
  "schemaKey": "memory.episode.v1",
  "provenance": "user-chat",
  "confidence": 0.95,
  "topic": "architecture-canon",
  "tags": ["governance", "security"]
}
```

## 3. Правила консолидации (Сжатие)

1. **Continuous Consolidation**: Сырые логи (S-Tier) не хранятся вечно. Фоновый процесс (или асинхронный воркер после завершения сессии) анализирует новые взаимодействия и формирует из них Эпизоды (M-Tier).
2. **Profile Updates**: Из Эпизодов (M-Tier) могут быть извлечены конкретные факты для обновления Профиля (L-Tier). Например, если пользователь сказал "Всегда используй Tabs, а не Spaces", это уходит в L-Tier.
3. **No Redundancy**: M-Tier должен содержать только *новую* или *уточняющую* информацию. Полный дубляж логов в эпизоды запрещен.

---

## 4. Правила Безопасности (Security Canon Alignment)

1. **Tenant Isolation**: Правило #1 из `SECURITY_CANON.md`. Любой retrieval или консолидация **обязаны** использовать фильтрацию по `companyId`.
2. **RBAC Scope**: Память скоупится по роли: агент не может извлечь профиль администратора, общаясь с обычным пользователем, если это нарушает access model.
3. **Audit Trail**: Прямые модификации L-Tier (профиля) или M-Tier (удаление эпизодов) всегда логируются. Принцип Auditability соблюдается безусловно.
4. **Purge Policy**: При удалении тенанта или пользователя из системы, все связанные с `companyId` / `userId` записи S/M/L уровней удаляются (каскадно или скриптом).

---

## 5. Интеграция с MemoryAdapter

Интерфейс `MemoryAdapter` абстрагирует эти слои:
- `appendInteraction` пишет в S-Tier.
- `retrieve` ищет по M-Tier (и, опционально, свежему S-Tier).
- `getProfile` / `updateProfile` работают с L-Tier. 
Агент-потребитель (например, `RaiChatService`) не должен знать о таблицах, JSONB или процессе сжатия. Он только отправляет IO-команды.
