# Front UX Map (Beta / F0) 🔒 Normative / Binding

**Статус:** 🔒 Normative / Binding  
**Тип:** Strategic, Read-only, Context-driven  
**Навигация:** ❌ без sidebar, ❌ без “разделов”

---

## 0. UX-принципы (коротко)
1. **Одна точка входа** → Global State (Общее состояние)
2. **Навигация от сигналов**, а не от меню
3. **Drill-down** вместо переходов
4. **Overlay** вместо страниц
5. **WHY > WHAT > DETAILS** (в таком порядке)

---

## 1. Entry Point — Global State View (GSV)
**ID:** GSV-01  
**Роль:** CEO / Architect / Legal / R&D

### Назначение
Мгновенно понять: что происходит, где риск, что блокирует.

### Состав экрана
- **A. Состояние системы (вверху)**
  - Общий статус: ОК / ВНИМАНИЕ / ЗАБЛОКИРОВАНО
  - Временная метка "Актуально на..."
- **B. Активные ограничения (центр)**
  - Юридические ограничения (кол-во + критичность)
  - R&D ограничения (заблокированные эксперименты)
  - Операционные сигналы (только чтение)
- **C. Эскалации (внизу)**
  - Элементы с направлением: ↑ эскалация / → стабильно / ↓ разрешение

**Интеракции:** Click на любое ограничение → Context View. Никаких кнопок действий.

---

## 2. Context View — R&D
**ID:** CTX-RD-01  
**Открывается из:** GSV → R&D Ограничение

### Назначение
Понять состояние научного контура и причины блокировок.

### Состав
- **A. Сводка экспериментов**
  - Список (компактный): Название, Состояние FSM, Статус протокола, Юр. флаг.
- **B. Визуализация состояния**
  - Таймлайн FSM (только чтение). Подсветка текущего стейта.
- **C. Причины блокировки**
  - Ссылки на LegalRequirement, Проблемы протокола, Блокировки замеров.

**Интеракции:** Click на эксперимент → Detail Overlay (Эксперимент).

---

## 3. Detail Overlay — Эксперимент
**ID:** OV-RD-EXP-01  
**Тип:** Overlay / Side Panel (не страница)

### Назначение
Дать полную картину эксперимента без возможности вмешательства.

### Секции
- **Заголовок:** Название, ID, Текущее состояние FSM.
- **Протокол:** Версия, Статус (УТВЕРЖДЕН / ЧЕРНОВИК), Дата утверждения.
- **Замеры:** Количество, Блокировка: ДА / НЕТ.
- **Юридический импакт:** Целевые LegalRequirements, ComplianceStatus.
- **Заключение (если есть):** Резюме, Применимость, Ограничения.

**Интеракции:** Close overlay → возврат в Context View.

---

## 4. Context View — Legal / Compliance
**ID:** CTX-LGL-01  
**Открывается из:** GSV → Юридическое ограничение

### Назначение
Понять, что именно требует право и где нарушение/риск.

### Состав
- **A. Список требований**
  - Резюме, Цель (ЭКСПЕРИМЕНТ / ПРОТОКОЛ / …), ComplianceStatus, Версия проверки.
- **B. Обязательства**
  - Описание, Срок, Статус.
- **C. Потенциальные санкции**
  - Тип, Сумма (если есть), Уровень влияния.

**Интеракции:** Click на требование → Detail Overlay (Требование).

---

## 5. Detail Overlay — Юридическое требование
**ID:** OV-LGL-REQ-01

### Секции
- **Ссылка на норму:** Юридический документ, Параграф, Версия.
- **Интерпретация:** Резюме требования.
- **Комплаенс:** Статус, Уверенность, Версия проверки.
- **Затронутые объекты:** Эксперименты / Протоколы.

---

## 6. Context View — Риск-мониторинг
**ID:** CTX-RISK-01  
(Даже без полноценного Risk Engine)

### Назначение
Осознать суммарную экспозицию риска.

### Состав
- **Общий индикатор риска** (НИЗКИЙ / СРЕДНИЙ / ВЫСОКИЙ)
- **Источники риска:** Юридический, R&D, Операционный.
- **Направление:** эскалация / стабильно.

⚠️ **Без чисел, без расчётов — индикативно.**

---

## 7. Навигационная логика (строго)
```
Вход (Login)
 → Общее состояние (Global State View)
   → (клик по ограничению)
     → Контекстный вид (Context View)
       → (клик по элементу)
         → Детальный оверлей (Detail Overlay)
```
❌ **нет глубины > 3**  
❌ **нет “потерянных” экранов**  
❌ **нет меню**  
✅ **всегда можно вернуться в Global**

---

## 8. Пустые и граничные состояния
- **Empty State:** “Активных ограничений нет”, “Все эксперименты соответствуют нормам”.
- **Blocked State:** Красный маркер, обязательное объяснение (WHY).
- **Data Delay:** Явный индикатор: “Данные актуальны на...”.

---

## 9. Запрещённые UX-паттерны (ещё раз)
❌ **Иконки редактирования**  
❌ **Кнопки действий (Actions)**  
❌ **Мастера (Wizards)**  
❌ **Inline-контролы**  
❌ **Массовое выделение**

---

## 10. Definition of UX Done (Beta)
UX считается корректным, если:
1. Пользователь не может изменить состояние системы.
2. Любой статус имеет объяснение.
3. Навигация невозможна без контекста.
4. Фронт можно удалить — и система не пострадает.

---

## Итоговая фиксация
> **Front UX Map (Beta):**  
> Это карта экранов осознанности, а не интерфейс управления, без меню, без действий, с обязательным слоем объяснений.
