## Как читать документ

- `Вариант A` — быстрый и дешёвый: закрыть явные риски без большой перестройки.

### 7.1) Честный статус DB-программы (чтобы не обманывать себя)

Что уже реально сделано:

- есть логическая фрагментация схемы, ownership-манифесты и рабочие enforce-гейты;
- есть tenant-wave для `FrontOfficeThread` family;
- есть индексный governance и evidence-подход.

Что не закрыто на дату 2026-03-19:

- физического split БД нет и не должен был быть на этом этапе (это осознанное решение, не баг процесса);
- основная схема всё ещё крупная (`6084` строк, `195` моделей);
- `companyId` остаётся доминирующим, а `tenantId` покрывает ограниченный набор моделей;
- Phase 1/4/5/6/7 содержит незакрытые execution-хвосты (backfill/e2e, projection jobs, enum conversions, index removal wave, закрытие Wave 1 + core aggregate migration).

Практический вывод:

- попытка не провалена;
- сделан сильный governance-фундамент;
- не завершён operational finish, из-за чего эффект “оптимизировали БД” снаружи пока слабый.

| Срок | Что делаем | Результат |

Definition of Done для “DB реально усилили”:

- нет `TBD/PENDING` в wave-closeout документах;
- минимум 1 core aggregate прошёл full migration wave;
- минимум 1 projection job работает в прод-контуре;
- выполнена первая removal волна индексов по evidence;
- phase-status файлы синхронизированы с фактом, а не с намерением.

## Рекомендуемый комбинированный курс (практический)

- Ближайшие 30 дней: взять `A` для Security/Backend/CI + `B` для Frontend/Docs.
- 30-60 дней: перейти на `B` в Architecture, AI, Agent, Database, Code Quality.
- 60-90 дней: точечно запускать `C` только там, где подтверждена окупаемость (AI routing, progressive delivery, DR).

Итоговая стратегия: **не «большой рефакторинг сразу», а жёсткий risk-first hardening + поэтапная институционализация практик**.
