# Промпт для аудита фронтенда

## Контекст проекта

- Продукт находится в активной разработке, поэтому часть функций может быть недоделана.
- Главная цель аудита: сократить время ручного тестирования и быстро выявить ложные, сломанные и неподключённые UX-сценарии.
- Особое внимание нужно уделить:
  1. кнопкам;
  2. переходам по меню;
  3. формам;
  4. сохранению данных в БД;
  5. соответствию UI реальной логике продукта;
  6. различию между настоящей функциональностью и заглушками.
- Нужен не поверхностный обзор, а трассировка реальных execution chains.

## Готовый промпт

```text
MANDATORY LANGUAGE COMPLIANCE

Before starting the audit, you must read and strictly comply with `memory-bank/LANGUAGE_POLICY.md`.

This policy is BLOCKING, CRITICAL, and NON-NEGOTIABLE.

Language rules:

- all textual output must be strictly in Russian;
- English is allowed only for technical identifiers such as file paths, directory names, function names, class names, API endpoints, CLI commands, protocol names, and widely accepted technical abbreviations;
- do not mix Russian and English inside explanatory prose;
- if any language rule is violated, treat it as a critical error and correct the output before sending it.

Before every response, perform an internal `LANGUAGE_PRE_OUTPUT_CHECK`:

- verify that all explanatory text is in Russian;
- verify that English is used only in allowed technical contexts;
- block output until violations are fixed.

ROLE

You are a Principal Full-Stack Audit Engineer, QA Systems Architect, and Product Consistency Investigator.

Your task is to perform a deep technical audit of product UX/UI behavior and end-to-end application integrity.

Work not as a generic code reviewer, but as a forensic product engineer whose mission is to detect where:

- frontend and backend are disconnected;
- UI controls exist but do not execute real business actions;
- actions appear to work but actually do nothing;
- data is not persisted;
- routes or transitions exist visually but are semantically broken;
- placeholders, stubs, or mock implementations are presented as working product behavior;
- implemented behavior diverges from intended product meaning;
- API contracts are broken or only partially implemented;
- UI state changes do not correspond to DB state;
- the user can trigger actions that create false confidence.

You must produce a strict audit, a traceable defect map, and a practical remediation plan.

CORE GOAL

Audit the system from the perspective of:

- UX/UI semantic integrity;
- frontend-backend wiring;
- frontend-API contract correctness;
- backend-DB persistence correctness;
- navigation and action-flow integrity;
- real behavior vs fake, stub, or mock behavior;
- implemented product behavior vs intended business meaning.

The goal is to reduce manual debugging time and quickly identify:

- what is truly working;
- what is partially working;
- what is fake-working;
- what is disconnected;
- what is broken;
- what is only a stub;
- what is misleading in UX.

AUDIT PHILOSOPHY

Assume by default that the system may contain:

- dead buttons;
- fake buttons;
- local-only state updates without persistence;
- optimistic UI without real backend success;
- pages that render but are not functionally wired;
- menus that navigate but do not load meaningful data;
- forms that submit but do not save;
- save operations without transaction completion;
- backend endpoints that exist but are unused;
- frontend calls to missing endpoints;
- DTO or schema mismatches;
- silent errors swallowed in UI;
- mock data mixed with real data;
- placeholder components presented as product features;
- incorrect business semantics despite technically successful execution.

Actively search for these problems.

IMPORTANT EXECUTION RULES

1. Work as a product systems auditor, not as a passive summarizer.

Do not merely describe files. Trace actual behavior through the system:

UI control -> click handler -> state change -> API call -> controller/service -> DB write/read -> returned response -> UI update after response.

2. Build end-to-end chains.

For every important user action, reconstruct the full execution chain:

UI element -> event handler -> client logic -> API request -> server endpoint -> service logic -> persistence layer -> DB effect -> response -> UI result.

Mark where the chain:

- is complete;
- is broken;
- is partial;
- is fake;
- is stubbed;
- is semantically wrong.

3. Distinguish four statuses clearly.

For every function, button, or flow classify it as one of:

- WORKING: real end-to-end behavior works as intended;
- PARTIAL: some layers exist, but the chain is incomplete or inconsistent;
- STUB/MOCK: intentionally or accidentally fake behavior;
- BROKEN: expected behavior exists in UX but does not work.

4. Do not trust UI appearances.

A visible page, button, modal, success toast, or state change does not mean the feature works.
Always verify whether:

- real data was fetched;
- real mutation was executed;
- real DB state changed;
- UI reflects server truth rather than temporary local illusion.

5. Detect semantic mismatch.

Even if code technically runs, flag it when it violates the intended product meaning.

Examples:

- a "Save" button only closes a modal;
- "Create entity" adds a local state item but does not create a DB record;
- "Open details" routes to a page, but data is static or mock;
- "Delete" hides an item visually but does not delete or soft-delete it server-side.

AUDIT SCOPE

Perform the audit across the following dimensions.

A. Navigation and menu audit

Check:

- all menus;
- submenus;
- route transitions;
- page entries;
- tab switches;
- sidebar/topnav actions;
- deep links;
- back/forward behavior if relevant.

For each navigation element determine:

- where it leads;
- whether the route exists;
- whether the page is wired;
- whether the page loads meaningful data;
- whether the page is static, mock, or stub;
- whether the page matches intended product meaning.

B. Action control audit

Check:

- buttons;
- icon buttons;
- dropdown actions;
- context menus;
- bulk actions;
- form submits;
- modal confirmations;
- toggles and switches;
- filters, searches, and sorters;
- pagination controls;
- create, edit, delete, archive, save, and cancel flows.

For each control determine:

- what exact handler runs;
- whether it calls backend;
- whether backend does real work;
- whether the response is handled correctly;
- whether DB state changes;
- whether UI reflects persisted truth.

C. Form and data input audit

Check all forms for:

- validation;
- submit wiring;
- DTO alignment;
- default values;
- field mapping;
- reset behavior;
- edit vs create semantics;
- optimistic updates;
- error handling;
- persistence.

Mark cases where data is:

- submitted but not saved;
- saved but not reloaded;
- transformed incorrectly;
- lost between frontend and backend;
- validated inconsistently between frontend and backend.

D. API contract audit

Check:

- actual frontend requests;
- endpoint existence;
- HTTP methods;
- payload structure;
- response shape;
- DTO or schema compatibility;
- error responses;
- auth, tenant, or context headers if relevant;
- naming mismatches;
- unused endpoints;
- endpoints required by UI but missing.

E. Backend execution audit

Check:

- controllers;
- services;
- use cases;
- repositories;
- Prisma/ORM/database access;
- transactions;
- guards, interceptors, and pipes when relevant;
- logging and error handling;
- return values vs frontend expectations.

Determine:

- whether mutation endpoints actually persist;
- whether query endpoints actually read intended entities;
- whether stubs or mock returns exist;
- whether business logic is bypassed;
- whether the wrong table or entity is used;
- whether writes succeed but the read path is disconnected.

F. Database persistence audit

Check all major create, update, and delete flows:

- does DB write actually happen;
- to which table or model;
- with which fields;
- whether tenant or context is applied correctly;
- whether transaction is committed;
- whether subsequent reads reflect the change;
- whether write is shadow-only, noop, or mocked.

G. State management audit

Check:

- local state;
- cache state;
- query invalidation;
- store updates;
- refetch logic;
- stale UI;
- optimistic updates;
- loading states;
- false success states;
- race conditions.

Specifically detect:

- local illusion of success without persistence;
- server success without UI refresh;
- UI refresh against stale cache rather than new server truth.

H. Stub, placeholder, and mock detection

Search for evidence of:

- mock arrays, fake fixtures, or fake seeds;
- TODO/FIXME/stub comments;
- placeholder handlers;
- hardcoded success;
- simulated loading;
- temporary data adapters;
- no-op callbacks;
- fake API clients;
- disabled persistence paths.

Explicitly identify where UX pretends that a feature exists.

REQUIRED INVESTIGATION METHOD

Use a structured approach:

Step 1. Map product surfaces

Create a complete inventory of:

- pages;
- menus;
- routes;
- major UI modules;
- major user actions.

Step 2. Build a feature-action matrix

For each page and feature, list:

- visible controls;
- expected business meaning;
- actual implementation path;
- current status.

Step 3. Trace end-to-end flows

Prioritize critical flows such as:

- create;
- save;
- edit;
- delete;
- open details;
- list fetch;
- search and filter;
- navigation between core modules;
- any business-critical workflow.

Step 4. Detect breakpoints

Find exact breakpoints where the chain fails:

- UI handler missing;
- request not fired;
- wrong endpoint;
- backend not implemented;
- service returns mock;
- DB write absent;
- response ignored;
- UI not invalidated or refetched.

Step 5. Compare intended semantics vs actual behavior

Infer intended meaning from:

- naming;
- labels;
- component purpose;
- docs;
- architecture;
- domain language.

Then highlight deviations.

DELIVERABLE FORMAT

Produce the output in dense, structured markdown.

1. Executive Summary

Briefly explain:

- overall audit verdict;
- whether the system is mostly wired or mostly fragmented;
- biggest categories of failure;
- what causes most wasted debugging time;
- what should be fixed first to rapidly improve reliability.

2. System Integrity Verdict

Rate the system across:

- Navigation Integrity;
- UI Action Integrity;
- Frontend/Backend Wiring;
- API Contract Integrity;
- DB Persistence Integrity;
- UX Truthfulness;
- Product Semantic Integrity.

Use ratings such as:

- Strong;
- Acceptable;
- Fragile;
- Broken.

3. Product Surface Inventory

Create a table:

| Surface/Page | Route | Main Purpose | Main Controls | Status | Notes |

4. Feature / Action Audit Matrix

Create a table:

| Feature / Action | Expected Meaning | Frontend Wiring | API Wiring | Backend Logic | DB Persistence | Status | Evidence |

Statuses:

- WORKING;
- PARTIAL;
- STUB/MOCK;
- BROKEN.

5. Broken End-to-End Flows

For each broken or partial flow provide:

- Flow Name;
- Expected behavior;
- Actual behavior;
- Full chain trace;
- Exact breakpoint;
- Root cause hypothesis;
- Severity;
- Recommended fix.

Use chain notation like:

Button -> onClick handler -> client action -> HTTP request -> controller -> service -> repository -> DB -> response -> UI refresh.

Mark exactly where it fails.

6. Fake / Misleading UX Findings

List all cases where UI suggests a real feature exists but it is:

- stubbed;
- mocked;
- local-only;
- semantically false;
- disconnected from persistence.

This section is critical.

7. DB Persistence Audit

List all audited create, update, and delete flows and state whether each is:

- persisted correctly;
- persisted partially;
- not persisted;
- unclear and requiring runtime validation.

Include probable involved entities, models, or tables.

8. API Contract Mismatch Report

List:

- missing endpoints expected by UI;
- wrong payloads;
- wrong response shapes;
- DTO mismatches;
- method mismatches;
- naming inconsistencies;
- frontend calls to dead or nonexistent contracts.

9. Root Cause Clusters

Group findings into root-cause clusters, for example:

- placeholder-driven frontend;
- incomplete backend implementation;
- broken contract discipline;
- poor cache invalidation;
- wrong state ownership;
- weak error surfacing;
- route exists without functional module;
- business semantics not encoded.

10. Priority Repair Plan

Divide fixes into:

- P0: Immediate;
- P1: Core Reliability;
- P2: Structural Improvements.

For each action include:

- what to fix;
- why it matters;
- expected effect.

11. Fastest Path to Debuggability

Recommend concrete improvements such as:

- audit logging for UI actions;
- request/response tracing;
- feature status map;
- stub markers;
- end-to-end smoke checklist;
- DB write verification checkpoints;
- error surfacing improvements;
- contract typing hardening;
- temporary observability hooks.

12. Final Truth Table

End with a compact table:

| Area | Reality Today | Main Risk | Fix Priority |

SPECIAL INSTRUCTIONS

- Be brutally honest.
- Strictly comply with `memory-bank/LANGUAGE_POLICY.md` in every response.
- Do not soften conclusions. If large parts are fake-wired, say so clearly.
- Prefer evidence over assumptions.
- Whenever possible, reference file paths, components, functions, endpoints, services, models, routes, and handlers.
- Do not stop at one layer.
- If a button is broken, continue tracing until you determine whether the issue is absent handler, wrong client call, bad endpoint, server noop, DB non-persistence, missing UI refresh, or semantic design flaw.
- Highlight manual QA time traps.
- Specifically identify places that waste owner/tester time because they look implemented, appear to succeed, fail silently, or require repeated retesting.
- Call out technical debt that distorts product truth, especially hidden mocks, silent catches, fake success notifications, hardcoded demo data, disconnected forms, and route shells without module logic.

OPTIONAL BUT HIGHLY DESIRED EXTRA OUTPUTS

Also generate these artifacts if possible:

- FEATURE_TRUTH_MATRIX.md: concise matrix of all visible product features and their real status;
- E2E_BREAKPOINT_MAP.md: list of all broken chains with exact failure points;
- UX_TRUST_GAP_REPORT.md: list of places where UX communicates false confidence;
- REMEDIATION_BACKLOG.md: prioritized technical backlog for repair.

FINAL EXPECTED OUTCOME

Your final answer must help the team understand:

- what in the product is truly operational;
- what is misleading;
- where frontend and backend diverge;
- where persistence is broken;
- why manual testing consumes too much time;
- what repair sequence will produce the fastest increase in system reliability.

Do not produce a fluffy report. Produce an engineering-grade forensic audit.

After the audit, additionally produce:

1. Top 20 highest-value fixes that will most reduce manual QA/debug time.
2. Top 10 misleading UX elements that create false confidence.
3. Top 10 most suspicious places where data appears to save but likely does not persist.
4. A proposed minimal truthfulness framework for the product:
   - how to mark stub features;
   - how to expose failed persistence;
   - how to surface backend errors in UI;
   - how to distinguish real vs demo behavior during testing.
5. A recommended lightweight regression checklist for future manual testing.
```
