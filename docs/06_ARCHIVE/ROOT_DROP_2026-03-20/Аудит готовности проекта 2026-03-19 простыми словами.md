## Итог

Проект сильный по идее и объёму: много модулей, тестов, зрелая архитектурная заготовка, развитый AI/agent-контур.  
Но на дату **19 марта 2026** это ещё не “безопасный enterprise production”.

Коротко:

- готовность как платформы для pilot/ограниченного запуска: **да**;
- готовность как fully trusted production для крупных клиентов: **пока нет**;
- интегральная оценка из техотчёта: **5.2 / 10**.

Что важно: в анализ включены ваши валидные текущие изменения в `rai-chat` (forensics и tracing). Это плюс к наблюдаемости агентной системы, но не закрывает ключевые блокеры безопасности и runtime-целостности.

## Ключевые риски

1. `CRITICAL`: слабая логика аутентификации (`password123`) и `AUTH_DISABLED=true` в tracked `.env`.
2. `CRITICAL`: часть tenant-контекста приходит из тела запроса (`companyId`), а не только из доверенного server-side контекста.
3. `CRITICAL`: в strategic-расчётах найден hardcoded tenant (`"1"`).
4. `CRITICAL`: у зависимостей есть уязвимости (`27 high`, `1 critical` на момент проверки).
5. `HIGH`: AI execution path неполный (`inference` помечен как not implemented).
6. `HIGH`: ingestion-контур без полноценного retry/DLQ/enrichment.
7. `HIGH`: ключевые governance-гейты не все зелёные (`gate:invariants:enforce`, `lint:docs:matrix`).
8. `HIGH`: security workflow в CI не блокирует пайплайн (`continue-on-error: true`).
9. `HIGH`: frontend содержит hardcoded backend URL (`localhost/127.0.0.1`), повышая риск env-дрейфа.
10. `MEDIUM`: документация и фактическое состояние местами расходятся (maturity claims против runtime-сигналов).

## Почему это важно

Если запускать систему в текущем виде на широкий прод:

- можно получить инциденты доступа и cross-tenant ошибки;
- рекомендации и risk-аналитика могут строиться по неверному контексту;
- рост нагрузки усилит нестабильность в AI/data pipeline;
- команда будет терять скорость из-за рассинхрона кода, тестов и документации;
- доверие enterprise-заказчика и комплаенс-аудиторов будет под риском.

Иными словами: продукт выглядит зрелым снаружи, но внутри ещё есть несколько мест, где “временные решения” стоят слишком близко к критическим зонам.

## Что делать сначала

1. `0-2 недели (Stop-the-bleeding)`:
- убрать `password123`-схему и перевести auth на безопасную проверку паролей;
- исключить `AUTH_DISABLED` из tracked runtime-конфигурации;
- запретить tenant из client body для write-path;
- убрать hardcoded tenant из strategic-логики;
- закрыть `critical/high` зависимости.

2. `2-4 недели (Stabilize)`:
- довести до green: `gate:invariants:enforce` и `lint:docs:matrix`;
- сделать security audit в CI блокирующим;
- централизовать frontend policy для backend base URL;
- убрать legacy proxy-слой, который дублирует/размывает API-контракт.

3. `1-2 месяца (Productionize)`:
- довести inference/ingestion до полноценного боевого контура (retry, DLQ, метрики, наблюдаемость);
- декомпозировать hotspot-модули с высокой связностью;
- усилить truthfulness/validation более доказательными механизмами, а не только эвристикой.

4. `Параллельно`:
- зафиксировать единый честный maturity-статус в документации;
- сделать автоматическую проверку “план vs факт” частью release-процесса.

## Ожидаемый эффект

После выполнения шагов выше:

- резко снижается риск security-инцидентов;
- multi-tenant поведение становится предсказуемым и проверяемым;
- AI-функции становятся стабильнее в ежедневной эксплуатации;
- CI снова начинает быть источником доверия, а не формальности;
- команда быстрее выпускает изменения без накопления скрытого долга;
- проект переходит из состояния “сильный pilot” в “реально production-grade платформа”.

## Честные ограничения аудита

- Это глубокий due-diligence аудит критических контуров, но не построчная проверка каждого файла репозитория.
- UX-оценка сделана по коду, роутам и контрактам, без полевых пользовательских интервью.
- Security-выводы основаны на доступных локальных проверках и CI-сигналах; внешний pentest не выполнялся.
