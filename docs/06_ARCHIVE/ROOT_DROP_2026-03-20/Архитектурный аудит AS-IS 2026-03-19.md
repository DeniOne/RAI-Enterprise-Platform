# Архитектурный аудит системы AS-IS

Дата: 2026-03-19

## 1. Executive Summary

По факту это не микросервисная платформа и не набор независимых контуров, а большой multi-tenant модульный монолит на `NestJS` с одним главным runtime в `apps/api`, толстым `Next.js` фронтендом в `apps/web` и несколькими периферийными/частично живыми оболочками вокруг него. Главная системообразующая ось проходит через `PrismaService` с tenant-aware proxy, крупную Prisma-схему, общий `AppModule`, доменные Nest-модули и центральный AI runtime (`SupervisorAgent` -> `AgentRuntimeService` -> `AgentExecutionAdapterService`).

Архитектура не слаба целиком. У системы есть реально сильные платформенные элементы: tenant isolation на уровне ORM, outbox relay, audit/notarization, explainability/control-plane, memory layer, substantial domain logic в `tech-map`, `consulting`, `commerce`, `front-office`. Это не демо-репозиторий.

Но зрелость системы неоднородна. Рядом с production-like ядром есть заметный слой псевдо-зрелости:

- `risk` в активном API не использует реальный `packages/risk-engine`, а опирается на локальный stub `apps/api/src/shared/risk-engine.ts`.
- `strategic`, `legal`, `rd` используют частично synthetic/mock read models.
- `knowledge` читает `docs/graph.json` с файловой системы, а не живой граф.
- `inference/`, `risk-engine/`, `ingestion/`, `apps/telegram-bot/` существуют, но в текущем runtime либо не доведены, либо не являются активным центром системы.
- архитектурные HLD-документы системно обещают больше, чем подтверждает код.

Главный ограничитель роста не один, а связка из четырех факторов:

1. Слишком мягкие архитектурные границы внутри одного большого `AppModule`.
2. Центральный AI/orchestration слой как скрытая точка сверхсвязанности.
3. Переросшая единая схема БД: `6085` строк, `195` моделей, `149` enum.
4. Рассинхрон между кодом, фронтовыми контрактами, инфраструктурой и документацией.

Если продолжать наращивать функциональность без оздоровления границ, система будет расти, но цена каждого изменения будет расти быстрее самого продукта.

## 3. Архитектура системы AS-IS

### Архитектурный стиль

Фактически система представляет собой:

- большой multi-tenant modular monolith на `NestJS`;
- один основной web client на `Next.js 14 App Router`;
- несколько прилегающих sidecar/shell контуров;
- один общий PostgreSQL как главный persistent source of truth;
- Redis как runtime/cache/pub-sub/working-memory слой;
- MinIO/S3-compatible storage как object/WORM-контур;
- AI runtime, встроенный в тот же монолит, а не вынесенный в отдельный оркестратор.

Это не microservices/gRPC/NATS architecture. Прямое подтверждение:

- основной runtime собирается в одном `AppModule` (`apps/api/src/app.module.ts`);
- API поднимается одним процессом (`apps/api/src/main.ts`);
- `GraphQLModule` подключен, но активных резолверов только четыре (`rapeseed`, `season`, `task`, `technology-card`), а основная поверхность API живет в REST;
- ни gRPC, ни NATS не являются рабочими опорными механизмами текущего execution path.

### Реальные bounded contexts

Bounded contexts в коде просматриваются, но не жестко отделены:

- platform/core: tenant context, auth, prisma, outbox, audit, invariants, memory.
- AI/runtime: `rai-chat`, explainability, runtime governance, truthfulness, replay.
- agronomy core: `tech-map`, `season`, `task`, `field-registry`, `agro-events`, `consulting`.
- commerce/CRM/front-office: `commerce`, `crm`, `front-office`, `front-office-draft`.
- finance: `finance-economy`.
- auxiliary/partial domains: `legal`, `rd`, `risk`, `strategic`, `knowledge`, `knowledge-graph`, `adaptive-learning`, `vision`, `satellite`.

### Где реально проходят границы

Жесткая граница есть в двух местах:

- tenant boundary на уровне `TenantContextService` + `PrismaService` proxy;
- физическая граница между frontend и backend HTTP API.

Большинство остальных границ мягкие:

- доменные модули импортируются в один `AppModule`;
- orchestration и business logic часто живут в одном процессе и в одном dependency graph;
- AI runtime имеет доступ к нескольким доменам сразу через tool/adapter layer;
- фронтенд использует смесь rewrite, абсолютных backend URL, server-side fetch и legacy proxy routes.

### Central brain / orchestrator

Центральный оркестратор системы существует и находится в `rai-chat`:

- `RaiChatService` — тонкая оболочка.
- `SupervisorAgent` — фактический orchestrator.
- `AgentRuntimeService` — переключатель runtime-mode и governance/budget path.
- `AgentExecutionAdapterService` — главный cross-domain adapter/router.
- `AgentRegistryService` — конфигурационный каталог ролей, tool bindings, connectors и lifecycle overrides.

Это и есть “central brain”. Он реален, но архитектурно опасен, потому что одновременно выполняет routing, orchestration, fallback, validation, policy handling и частично domain dispatch.

## 5. Runtime Architecture

### Точки входа

Основные entry points:

- `apps/api/src/main.ts`: поднимает основной API на `4000`, задает prefix `/api`, включает CORS только для `http://localhost:3000`, Swagger только вне `production`.
- `apps/web`: Next.js app с rewrite `/api/:path* -> http://127.0.0.1:4000/api/:path*`.
- `apps/telegram-bot/src/main.ts`: отдельный Nest runtime для Telegram.
- `inference/app/routes.py`: `/health`, `/infer`.
- `risk-engine/app/routes.py`: `/risk/dry-run`.

### Request/response flow

Канонический product flow сегодня выглядит так:

1. Пользователь работает с `apps/web`.
2. UI вызывает backend либо через rewrite `/api`, либо напрямую на `http://localhost:4000/api`, либо через server-side fetch helpers.
3. Nest controller принимает запрос, чаще всего под JWT/dev-user security context.
4. `TenantContextInterceptor` кладет `companyId`/`tenantId` в `AsyncLocalStorage`.
5. Доменный service идет в `PrismaService`, который прозрачно заменяет model delegates на tenant-aware client.
6. Persisted truth живет в PostgreSQL.
7. При async delivery используется outbox relay и локальный event delivery path.

### Background flows

Подтвержденные background-механики:

- `OutboxRelay` — `@Cron(EVERY_SECOND)` + recovery lease + batch drain + DLQ/ordering guard.
- memory maintenance — consolidation/pruning/engram formation/remediation по cron.
- integrity drift loop — BullMQ queue + processor.
- finance reconciliation — hourly cron.
- adaptive-learning k8s job service — cron-based trigger.
- monitoring trigger в `rai-chat`.
- level-f anchoring/node watcher — cron-based services.

Это означает, что система уже не purely request/response; в ней есть заметный background runtime.

### Auth flow

Auth-картина неоднородна:

- Внутренний `AuthService` слабый: `validateUser()` проверяет literal password `"password123"`.
- `JwtAuthGuard` умеет полностью bypass JWT при `AUTH_DISABLED=true`.
- `DevModeService` может auto-create company и подставлять dev-user.
- Front-office auth заметно зрелее: invitation lifecycle, activation, binding к Telegram, password hashing (`scryptSync`), account/party link logic.

Следствие: в репозитории сосуществуют две модели зрелости auth. Внутренний auth-контур сейчас больше похож на dev/bootstrap режим, чем на enterprise auth.

### Data flow

Главный operational source of truth:

- PostgreSQL через Prisma schema из `packages/prisma-client/schema.prisma`.

Вторичные рабочие слои:

- Redis для working memory, runtime cache, pub-sub/wakeup.
- Outbox table как гарантированный async handoff.
- WORM/S3-compatible storage как audit artifact layer.

Раздвоенные/конкурирующие источники истины:

- knowledge graph как Prisma tables (`knowledgeNode`, `knowledgeEdge`) и одновременно файловый `docs/graph.json`.
- risk domain как отдельный package и одновременно local stub в API.
- AI memory как таблицы PostgreSQL/pgvector/Redis и одновременно HLD-описание более широкой memory onion, не реализованной полностью.

### AI / agent flow

Подтвержденный flow чата:

1. `RaiChatController` принимает `POST /rai/chat`.
2. `RaiChatService` делегирует в `SupervisorAgent`.
3. `SupervisorAgent` формирует `traceId/threadId`, тянет context из memory, строит execution request.
4. `AgentRuntimeService` выбирает `tool-first-legacy` или `agent-first-hybrid`.
5. `AgentExecutionAdapterService` dispatch-ит в конкретного агента или уходит в heuristic fallback.
6. `ResponseComposerService` собирает ответ.
7. `TraceSummaryService` и `SupervisorForensicsService` пишут audit/trace.
8. `TruthfulnessEngineService` асинхронно досчитывает quality.
9. Memory layer фиксирует interaction/profile update.

Это не просто “LLM wrapper”; AI runtime здесь реально встроен в продуктовую логику. Но и чистой агентной системой с жесткой доменной изоляцией его назвать нельзя.

### Memory/context flow

Реальная memory-модель:

- `L1`: Redis working memory / active alerts / hot engrams.
- `L2`: episodic memory через PostgreSQL + pgvector.
- `L4`: engrams.
- `L5`: memory profile.
- `L3/L6` по комментариям и docs планируются, но в коде не являются полноценными активными слоями.

Контекст инжектится в AI runtime через `MemoryCoordinatorService`/`MemoryFacade`, а tenant-context идет отдельно через `AsyncLocalStorage`.

## 6. Frontend ↔ Backend Contract Reality

### Что выглядит устойчивым

- Базовый Next.js shell реально большой и активно связан с backend.
- Есть typed API helpers, Zustand/XState/React Query patterns.
- В front-office контуре заметна более дисциплинированная работа с cookie/JWT и idempotency headers.
- Rewrite `/api/:path*` на backend дает единый браузерный path для части клиентов.

### Что хрупко

- `apps/web/lib/api/strategic.ts` жестко ходит на `http://localhost:4000/api` и fallback-ит в mock structure.
- `apps/web/lib/api/auth-server.ts` и `apps/web/lib/api/front-office-server.ts` также жестко завязаны на `http://localhost:4000/api`.
- `apps/web/app/dashboard/page.tsx` содержит прямые fetch на абсолютный localhost backend.
- `apps/web/app/(app)/knowledge/page.tsx` fallback-ит на `http://localhost:3001` и требует генерации `graph.json`.
- `apps/web/app/api/ai-chat/route.ts` — deprecated legacy proxy, который вручную ремапит shape ответа.
- `apps/web/middleware.ts` защищает только `/dashboard` и `/strategic`, а не всю приватную `(app)` поверхность.
- Search по `apps/web` подтверждает смешение `NEXT_PUBLIC_API_URL`, `/api`, `http://localhost:4000` и `127.0.0.1:4000`.

### Где контракт строгий, а где нет

Строже всего выглядят:

- front-office endpoints;
- AI chat endpoint с `IdempotencyInterceptor`;
- некоторые advisory/front-office flows с cookie/jwt.

Наиболее хрупкие зоны:

- strategic/read-model clients;
- knowledge page;
- legacy AI proxy;
- dashboard and debug pages;
- server-side helpers с hardcoded backend origin.

### Где пользовательский путь псевдоработает

- strategic dashboard может получить mock global state, а не реальную консистентную агрегацию;
- legal summary на фронте вообще мокаются;
- knowledge UI зависит от файла `graph.json`, а не от зрелой graph subsystem;
- deprecated `/api/ai-chat` еще поддерживается, что продлевает legacy contract вместо его закрытия.

Итог: UI богатый и большой, но backend contract не унифицирован. Это не “typed platform contract”, а смесь нескольких поколений API-подходов.

## 7. Data Architecture

### Ключевые сущности

По схеме и runtime usage основными сущностями являются:

- `Company` как tenant root;
- `User`, `Invitation`, `CounterpartyUserBinding`, `EmployeeProfile`;
- agronomy execution: `Field`, `Season`, `TechMap`, `Task`, `ExecutionRecord`, `HarvestPlan`, `HarvestResult`;
- commerce/CRM: `Account`, `Party`, `CommerceContract`, `CommerceObligation`, `Invoice`, `Payment`, `PaymentAllocation`;
- finance: `EconomicEvent`, `LedgerEntry`, `Budget*`, `CashAccount`, `AccountBalance`;
- AI/runtime: `TraceSummary`, `AiAuditEntry`, `RuntimeGovernanceEvent`, `QualityAlert`, `SystemIncident`, `EvalRun`;
- memory: `MemoryInteraction`, `MemoryEpisode`, `MemoryProfile`, `MemoryEntry`;
- knowledge: `KnowledgeNode`, `KnowledgeEdge`.

### Структура связей

Модель в целом company-scoped и tenant-first. Важная особенность: часть AI/runtime и platform-моделей уже живет в dual-key режиме (`companyId` + `tenantId`), что указывает на эволюцию схемы, но одновременно повышает сложность data governance.

### Сильные стороны data layer

- tenant scoping принудительно централизован в `PrismaService`;
- есть явный список tenant-scoped, dual-key и system models;
- outbox, audit, memory и explainability имеют persistence, а не только in-memory декларации;
- по части доменов schema действительно rich и отражает реальный продукт, а не toy-модель.

### Проблемные места

- единая schema разрослась до `6085` строк, `195` моделей, `149` enum;
- DB gate фиксирует `82` слабых single-field индекса вида `@@index([companyId])`;
- DB gate фиксирует heavy include hotspots в `commerce`, `consulting`, `integrity`, `tech-map`, `technology-card`;
- часть глобальных projection/read model paths живет через tenant-lint ignores и synthetic snapshots;
- векторный слой использует raw SQL и отдельный adapter с несовпадающим Prisma import;
- risk source of truth раздвоен между отдельным package и stub-фасадом;
- knowledge source of truth раздвоен между таблицами и файловым снапшотом.

### Consistency assumptions

Система предполагает:

- tenant-context всегда задан до обращения к tenant-scoped данным;
- outbox обрабатывает async-доставку и защищает ordering/duplicate semantics;
- некоторые FSM и invariants валидируются кодом и gate-скриптами;
- trace/audit/memory могут записываться асинхронно после ответа.

Но transaction boundaries не унифицированы по всей системе. Где-то есть явные транзакции (`audit`, `memory append`, часть outbox/event paths), где-то — обычные Prisma calls без очевидного UoW.

### Versioning / history / audit gaps

- В `tech-map`, audit, explainability и outbox versioning/history присутствуют.
- Единой подтвержденной стратегии soft delete не найдено.
- Единой политики domain-history для всех модулей не найдено.
- Стратегические/legal/risk projections не выглядят как зрелые audited read models.

## 8. Integration Architecture

### Подтвержденные внешние интеграции

- Telegram: в API-модуле `apps/api/src/modules/telegram/*` и отдельно в `apps/telegram-bot`.
- DaData: `apps/api/src/modules/commerce/services/providers/dadata.provider.ts`.
- MinIO/S3-compatible storage: `apps/api/src/shared/s3/*`, `apps/api/src/level-f/worm/worm-storage.service.ts`, `docker-compose.yml`.
- Redis/BullMQ: runtime cache, pub-sub, queue pressure, drift loop.
- LLM/provider-ready configuration: `OPENROUTER_API_KEY`, `NVIDIA_API_KEY` валидируются в `AppModule`.

### Уровень изоляции интеграций

Изоляция неоднородна:

- DaData выделен в отдельный provider class, но ходит через прямой `fetch` без подтвержденного circuit breaker и без явной idempotency модели.
- HTTP resilience модуль есть, но это общий axios-retry wrapper, а не полноценный integration fabric.
- Telegram split между API-модулем и отдельным ботом, что создает ownership ambiguity.
- WORM/S3 контур выглядит лучше формализованным, но связан с конфигом и окружением.

### Retry / timeout / fallback / idempotency

- HTTP resilience: глобальный axios retry настроен на network/5xx/429.
- Outbox имеет retry/backoff/DLQ/order guard.
- DaData provider возвращает `FOUND/NOT_FOUND/ERROR`, но явного timeout policy в самом provider не видно.
- Ingestion worker сам признает отсутствие retry policy и DLQ как TODO.
- Frontend/API частично используют idempotency keys, но не системно во всех write-paths.

### Vendor lock-in risk

- DaData привязан к RU lookup use case.
- Telegram тесно прошит в identity/invitation/front-office flows.
- AI provider surface потенциально сменяемый, но policy/runtime/config уже привязаны к конкретным моделям/именам.
- WORM/S3 контур относительно переносим, но operationally завязан на MinIO/S3 semantics.

## 9. AI / Agent / Memory Architecture

### Карта агентов

Подтвержденные канонические runtime roles:

- `agronomist`
- `economist`
- `knowledge`
- `monitoring`
- `crm_agent`
- `front_office_agent`
- `contracts_agent`
- `chief_agronomist`
- `data_scientist`

Эти роли описаны в `AgentRegistryService`, имеют tool bindings, capabilities и runtime profiles.

### Orchestration model

Реальная оркестрация — это custom Nest-based orchestration, а не LangChain/AutoGen runtime:

- supervisor orchestration;
- memory recall;
- budget/governance policy;
- adapter-based agent dispatch;
- response composition;
- trace/audit/truthfulness pipeline.

Это серьезнее, чем “просто LLM call”, но и слабее, чем жестко контрактная multi-agent platform с разделением ownership по доменам.

### Context injection

Контекст собирается из:

- `workspaceContext` с фронта;
- tenant/user context;
- memory recall;
- tool-call plan / intent classification;
- external signals/advisory feedback.

Это сильная сторона архитектуры: AI-контур реально связан с продуктовой ситуацией, а не только с текстом сообщения.

### Memory model

Реализовано:

- working memory в Redis;
- episodic memory в PostgreSQL/pgvector;
- engrams;
- memory profile;
- memory maintenance workers;
- observability по freshness/backlog.

Не реализовано как production fact:

- зрелый semantic graph memory на Neo4j/Memgraph;
- полноценный external RAG/document memory;
- подтвержденная “self-expanding schema” memory platform;
- сетевой memory layer.

### Validation / provenance / anti-hallucination

Подтвержденные механизмы:

- trace summaries;
- AI audit entries;
- truthfulness pipeline;
- runtime governance events;
- quality alerts;
- safe replay;
- explainability dashboard;
- evidence and validation payloads в agent execution result.

Это один из самых зрелых контуров системы.

### Критические слабости

- `AgentExecutionAdapterService` — огромный central adapter с heuristic routing. Это “magic black box” hotspot.
- Domain ownership агентов определен декларативно, но enforcement мягкий.
- `tool-first-legacy` и `agent-first-hybrid` сосуществуют, что продлевает архитектурную двусмысленность.
- Knowledge/risk/strategic зоны под AI orchestration частично опираются на неполные или stubbed backend read models.

## 13. Architecture Maturity Assessment

### Modularity — 6/10

Модули и домены реально выделены, packages и schema fragments существуют, ownership manifests и gates есть. Но общая модульность ослаблена единым `AppModule`, общим data layer и cross-domain orchestration.

### Clarity of boundaries — 4/10

Tenant boundary хорошая. Остальные границы мягкие. Особенно размыты frontend/backend contract, AI runtime vs domain logic, active runtime vs adjacent shells.

### Maintainability — 5/10

Сильные тестовые pockets есть (`rai-chat`, `tech-map`, `explainability`, `generative-engine`), но сложность системы уже заметно превышает простоту навигации. Документация не помогает, а часто мешает.

### Scalability — 5/10

Функционально масштабироваться можно, но организационно и архитектурно система уже упирается в schema growth, module hotspots и contract fragmentation. Это скорее “можно расти, но дорого”, чем “готово к росту”.

### Observability — 7/10

Explainability, trace summaries, runtime governance, queue pressure, Prometheus alert rules, outbox metrics и memory freshness показывают высокий уровень зрелости именно в observability discipline.

### Reliability — 5/10

Outbox, retries, invariants и reconciliation улучшают надежность. Но рядом остаются runtime stubs, integration gaps, mixed auth modes и deployment drift.

### Security hygiene — 3/10

Tenant isolation сильная, но это не перекрывает demo-password path, auth bypass, hardcoded local URLs и неполный operational hardening.

### Testability — 6/10

Важные core-модули тестируются. Но cross-module orchestration и integration/runtime drift делают end-to-end truth менее прозрачной.

### Integration resilience — 4/10

Outbox хорош, но внешние интеграции и соседние сервисы неоднородны. Retry/idempotency/fallback стандартизированы не везде.

### AI/runtime discipline — 6/10

Для AI-контура дисциплина выше среднего: traceability, replay, truthfulness, governance. Но доменные границы и фактическое качество upstream данных не дотягивают до заявленного уровня зрелости.

## 15. Final Verdict

Сегодняшняя система — это сильный, но неоднородный инженерный организм: не MVP, не игрушка, но и не та “institutional-grade microservice platform”, которую обещает часть документации. Реальный каркас системы — большой NestJS-монолит с хорошими платформенными инвариантами и зрелым AI/observability control plane, поверх которого наслоились частично доведенные домены, runtime shells и documentation drift.

На ней можно безопасно наращивать функциональность только в том случае, если следующая волна работы будет не feature-only, а boundary-first. Главный стоп-фактор — не отсутствие фич, а отсутствие честно закрепленных границ: между docs и кодом, между AI orchestration и доменами, между UI-контрактом и backend, между active runtime и соседними контурами.

Самый разумный следующий архитектурный шаг: зафиксировать канонический AS-IS runtime, закрыть ложные контуры зрелости (`risk/strategic/auth/frontend contract`), а затем начать поэтапное укрепление границ вокруг data layer и AI orchestration. Без этого дальнейший рост почти гарантированно будет происходить через усложнение, а не через усиление архитектуры.
