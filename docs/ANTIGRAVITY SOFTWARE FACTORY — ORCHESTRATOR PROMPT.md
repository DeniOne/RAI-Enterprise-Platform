## 2. ЖЕЛЕЗНЫЙ ГЕЙТ ACCEPTED (BLOCKING / NON-NEGOTIABLE)

Цель: **исключить произвол**. Переход к реализации — только после принятия плана.

### 2.1 Канонический токен принятия
Исполнение разрешено **только** если в ответе человека есть строка:

`ACCEPTED: interagency/plans/YYYY-MM-DD_<slug>.md`

Любой другой “ну ок” / “делай” / “да” **не считается**.

### 2.2 Режим PLAN-ONLY (до ACCEPTED)
Пока токена нет, имеешь право ТОЛЬКО:
- читать `interagency/prompts/*` и существующие планы/шаблоны,
- писать/обновлять **только**: `interagency/plans/YYYY-MM-DD_<slug>.md`,
- отвечать вопросами/уточнениями,
- править план (если отклонён/есть замечания).

Пока токена нет — ЗАПРЕЩЕНО:
- менять любой код в `apps/*`, `packages/*`, `libs/*`,
- запускать команды, которые меняют рабочее дерево (генерации/форматтеры/кодмоды),
- собирать “пакет на ревью”, писать отчёты, менять `interagency/INDEX.md`,
- делать commit/push (и вообще любые git-операции кроме чтения статуса/диффа, если это разрешено средой).

### 2.3 Обязательная остановка после плана
После записи плана в `interagency/plans/YYYY-MM-DD_<slug>.md` обязан вывести одну строку:

`WAITING_FOR_ACCEPTANCE: interagency/plans/YYYY-MM-DD_<slug>.md`

и **остановиться**. Никаких следующих шагов.

### 2.4 Если план отклонён
Если в ответе: `REJECTED:` / «переделай» / замечания — обновить план, снова вывести `WAITING_FOR_ACCEPTANCE: ...` и остановиться.

## 4. FOUNDATION LAW (АБСОЛЮТНЫЙ ПРИОРИТЕТ)

Перед **любым** действием обязательно:

1. Проверить соответствие:
   - CANON.md
   - ARCHITECTURAL_AXIOMS.md
   - FORBIDDEN.md
2. Проверить наличие релевантных решений в `DECISIONS.log`
3. Убедиться, что задача не нарушает Admission-принципы
4. Проверить соблюдение языковой политики
5. Проверить соответствие UI Design Canon (если задача затрагивает UI)
6. Проверить соблюдение **Multi-tenancy** (все данные должны быть изолированы через `companyId`)
7. Проверить **Infrastructure Isolation** (домен не должен зависеть от БД/Redis напрямую)
8. Убедиться в соблюдении принципа **Service = IO / Orchestrator = Brain**
9. Проверить на **CRM-блоат** (задача не должна превращать систему в CRM/ERP)

Если:
- задача противоречит FOUNDATION  
- или данных недостаточно  

→ **ПРОЦЕСС НЕМЕДЛЕННО ОСТАНАВЛИВАЕТСЯ**  
→ запросить уточнение у человека  

Фраза  
> «Недостаточно данных»  
является **корректным и обязательным ответом**.

### 4.1. Security Canon

Обязательно:
- Проверить задачу на соответствие SECURITY_CANON.md
- Убедиться в отсутствии security-антипаттернов
- Включить security-требования в ТЗ при необходимости
- Проверить наличие security review для критичных изменений

Если задача:
- Нарушает Security Canon
- Содержит потенциальные уязвимости
- Не имеет security assessment для критичных изменений

→ ПРОЦЕСС НЕМЕДЛЕННО ОСТАНАВЛИВАЕТСЯ
→ запросить security approval у человека

## 6. UI DESIGN CANON (FOUNDATION-УРОВЕНЬ)

### 6.1. Статус

UI Design Canon является **обязательным нормативным слоем** для всех задач, затрагивающих:
- пользовательские интерфейсы
- UI-компоненты
- layout, typography, визуальную иерархию
- правки существующих экранов

Нарушение UI Design Canon считается **архитектурным нарушением**.

Чётко следовать "/root/RAI_EP/docs/03_PRODUCT/UI_UX/DESIGN_SYSTEM.md"

---

### 6.2. Базовый принцип дизайна

Интерфейс:
- лёгкий
- чистый
- визуально «дышащий»
- без визуального шума

**Акцент создаётся размером и пространством, а не жирностью.**

---

### 6.3. Типографика (обязательная)

- Основной шрифт: `Geist`
- Заголовки, кнопки, метрики: `font-medium`
- Основной текст: `font-normal`
- `font-bold` и `font-semibold` **ЗАПРЕЩЕНЫ**
- Моно-шрифт — только для ID, логов, технических меток

---

### 6.4. Цветовая модель (Light)

Обязательные принципы:
- светлая тема
- отсутствие чистого чёрного
- отсутствие «тяжёлых» серых фонов

Карточки и контейнеры:
- `bg-white`
- `border-black/10`
- `rounded-2xl` или `rounded-3xl`

---

### 6.5. Компонентные ограничения

Запрещено:
- использовать устаревшие тёмные паттерны
- оставлять legacy-стили «как есть»
- смешивать визуальные стили между экранами

При модификации старых экранов действует принцип:

**Scorched Earth**  
— старый стиль полностью заменяется каноничным.

---

### 6.6. Правила для кодера (UI)

- Проверять соответствие UI Design Canon на этапе ТЗ; при некорректном UI-описании — блокировать.
- Реализовывать UI строго по канону.
- Не «улучшать дизайн от себя».
- Не переносить визуальные паттерны без фиксации.

## 8.1 INTERAGENCY WORKFLOW (PROMPTS → PLANS → REPORTS) — ОБЯЗАТЕЛЬНО

Цель: единый исполняемый контур Cursor ↔ Antigravity через общую папку `interagency/`, с жёстким разделением "сбор пакета на ревью" и "финализация".

### Каноническая папка
`interagency/` — единый источник для:
- промтов (что сделать),
- планов (как делать),
- отчётов (что реально получилось),
- шаблонов (единый формат).

Структура и правила:
- `interagency/README.md`
- `interagency/INDEX.md`
- `interagency/templates/*`

### Два обязательных промта (жёстко)
1) **Antigravity: собрать пакет для ревью**
   - `docs/ANTIGRAVITY SOFTWARE FACTORY — REVIEW PACKET PROMPT.md`
   - результат: отчёт + индекс + дифф + логи, статус в индексе = `READY_FOR_REVIEW`
   - запрет: commit/push и "DONE"

2) **Cursor: ревью и финализация**
   - `docs/CURSOR SOFTWARE FACTORY — REVIEW & FINALIZE PROMPT.md`
   - результат: ревью (APPROVED/CHANGES_REQUIRED), отметки чеклистов/memory-bank, commit, push только по явной команде USER

### Алгоритм исполнения
1) **Получение задачи**
   - задачи принимаются ТОЛЬКО из `interagency/prompts/`
   - USER кладёт промт в `interagency/prompts/YYYY-MM-DD_<slug>.md`

2) **Планирование**
   - на каждый промт создаётся план:
     - `interagency/plans/YYYY-MM-DD_<slug>.md`
     - формат: `interagency/templates/PLAN_TEMPLATE.md`

3) **Исполнение (Antigravity)**
   - реализовать задачу по плану/промту
   - затем выполнить `REVIEW PACKET PROMPT` и поставить `READY_FOR_REVIEW` в `interagency/INDEX.md`

4) **Ревью и финализация (Cursor)**
   - Cursor-агент выполняет `REVIEW & FINALIZE PROMPT`
   - только после APPROVED разрешены: чеклисты, memory-bank, commit, push (по явной команде)

### Запреты (блокирующие)
- запрещено выполнять задачи без файла в `interagency/prompts/`
- запрещено начинать без плана в `interagency/plans/`
- запрещено делать commit/push из Antigravity
- запрещено ставить `DONE` в `interagency/INDEX.md` до внешнего ревью

## 9. ПРОЦЕСС РАБОТЫ (ЛИНЕЙНЫЙ АЛГОРИТМ ДЛЯ КОДЕРА)

1. Взять задачу из `interagency/prompts/` (промт в `YYYY-MM-DD_<slug>.md`).
2. Проверить: FOUNDATION, SECURITY_CANON.md, LANGUAGE POLICY, UI Design Canon (если применимо), Decision-ID, оси Brain/IO и Multi-tenancy, отсутствие CRM-паттернов. Сформировать ТЗ с security-требованиями при необходимости.
3. Составить план → `interagency/plans/YYYY-MM-DD_<slug>.md` → вывести `WAITING_FOR_ACCEPTANCE: ...` и остановиться.
4. После строки `ACCEPTED: interagency/plans/...` в ответе — перейти к реализации.
5. Реализовать по плану, выполнить `REVIEW PACKET PROMPT`, выставить `READY_FOR_REVIEW` в `interagency/INDEX.md`.
6. Внешний ревью (Cursor): `REVIEW & FINALIZE PROMPT`. После APPROVED — обновление чеклистов, memory-bank, commit/push только по явной команде.
