# Мастер-стандарт Дизайн-системы: RAI Enterprise Platform
## Institutional Grade 10/10+ | Control Plane Canon

> [!IMPORTANT]
> Это не просто UI-кит. Это стандарт **Institutional Governance Control Plane**. Любое отклонение от данного канона приравнивается к архитектурной ошибке и подрывает доверие к системе при аудите. Интерфейс — это юридически значимая поверхность управления.

---

## 1. Философия и Позиционирование

### 1.1 Control Plane vs SaaS UI
RAI Enterprise — это не очередной SaaS. Это среда управления рисками и суверенитетом.

| Характеристика | SaaS Парадигма | Institutional Парадигма (RAI) |
| :--- | :--- | :--- |
| **Приоритет** | KPI-first (достижения) | **Risk-first** (безопасность) |
| **Стиль** | Мягкий, декоративный | **Жёсткий, структурный** |
| **Мотивация** | Вовлеченность (Engagement) | **Ответственность** (Accountability) |
| **Сложность** | Скрыта (Simplicity) | **Экспонирована** (Traceability) |

### 1.2 Принцип "Scorched Earth"
При модификации старых экранов старый стиль полностью заменяется каноничным. Никаких гибридов и legacy-стилей в одном интерфейсе.

## 2. Визуальный Канон (Foundation)

### 2.1 Типографика (Geist Canon)
**Шрифт:** `Geist Sans` (UI) / `Geist Mono` (Технические данные). Использование других шрифтов запрещено.

*   **Акцент создаётся пространством, а не жирностью.**
*   `font-medium` — для заголовков, кнопок и метрик.
*   `font-normal` — для основного текста.
*   `font-bold` и `font-semibold` — **ЗАПРЕЩЕНЫ**.
*   **Mono-font** — обязательно для ID, логов, финансовых данных, номеров счетов, БИК и ИНН.

### 2.2 Цветовая модель (Institutional Light)
*   **Canvas (Фон):** `bg-slate-50` (`#F8FAFC`).
*   **Surfaces (Карточки):** `bg-white`, `border-black/10`, `rounded-2xl` или `rounded-3xl`.
*   **Ink (Текст):** `#030213` (Смягченный черный для снижения усталости).
*   **Muted:** `#717182`.

### 2.3 Семантические статусы (Risk Triage)
| Уровень | Категория | Код | UX Поведение |
| :--- | :--- | :--- | :--- |
| **R4** | Critical | `Red-600` | Мгновенный Hard-Escalation, блокировка связанных действий. |
| **R3** | High | `Amber-600` | Красный баннер, требование немедленного Quorum (M-of-N). |
| **R2** | Moderate | `Amber-400` | Желтый индикатор, запись в CRM Deviation. |
| **R1** | Info | `Blue-500` | Синий маркер, работа AI-агента. |
| **Success** | Verified | `Emerald-500` | Данные закреплены в Ledger, транзакция исполнена. |

## 3. Архитектура взаимодействия

### 3.1 Режим View / Edit (B-Pattern)
Единый паттерн разделения режимов просмотра и редактирования:

*   **View Mode (Default):**
    *   Поля = Label + Value без рамок и input-стилей.
    *   Пустые значения отображаются как `—` (Muted).
    *   Никаких интерактивных селектов или пикеров.
*   **Edit Mode:**
    *   Появляются аккуратные input-границы (`border-black/10`).
    *   Обязательные поля помечаются красной звёздочкой `*`.
    *   Валидация (Zod) отображается под полем, не ломая layout.

### 3.2 Иерархия CTA (Call to Action)
На одной странице/экране допускается ровно ОДНО действие типа **Primary**.
*   **Primary (Black/Indigo):** Глобальное сохранение или подтверждение финализации.
*   **Secondary (Outline/White):** Кнопки "Добавить", "Отмена", "Редактировать".
*   **Ghost/Text:** Удаление, второстепенные опции.

### 3.3 Tabular Scalability (C-Pattern)
Любые списки (счета, контакты, активы) переводятся в табличный вид. Карточки для списков запрещены.
*   **Scalability:** Интерфейс должен выдерживать 20+ записей без деградации.
*   **Drawers (SidePanelForm):** Все операции создания/редактирования элементов списка происходят в боковой панели. Это сохраняет контекст основной страницы.
*   **Empty States:** Обязательная иконка + текст + кнопка Add в центре пустой таблицы.

### 3.4 Контекстные подсказки вместо поясняющих плашек
Длинные информационные плашки с очевидным поясняющим текстом в рабочих экранах запрещены.

*   **Базовое правило:** Если информация не нужна для немедленного действия, она не должна занимать постоянное место на экране.
*   **Канонический паттерн:** Короткая кнопка-подсказка `i` или `?` рядом с заголовком блока, колонки или поля.
*   **Поведение:** Подсказка открывается по наведению или фокусу и объясняет смысл простым пользовательским языком.
*   **Язык:** Без внутреннего жаргона, англицизмов и процессного формализма. Пользователь должен понимать текст без знания внутренней архитектуры.
*   **Исключение:** Постоянные баннеры допустимы только для действительно важных состояний: ошибка, риск, блокировка, эскалация, несохранённые изменения или обязательное следующее действие.
*   **Запрет:** Не дублировать очевидное описание страницы длинным абзацем под заголовком, если ту же мысль можно уместить в контекстную подсказку.

### 3.5 AI Dock и оверлэй-окна агента
Для AI-взаимодействия действует отдельный канон композиции.

*   **AI Dock слева:** только одна цельная оболочка `header -> signals -> history -> composer`.
*   **Signals:** компактный встроенный блок внутри dock. Отдельные тяжёлые карточки мини-инбокса запрещены.
*   **Правая зона:** остаётся рабочей поверхностью продукта.
*   **Окна агента:** открываются только как overlay поверх правой зоны.
*   **Structured output:** не должен жить отдельной parallel-rail архитектурой, если тот же вывод можно показать через typed overlay-window.
*   **Window modes:** использовать по тяжести задачи:
    * `inline` — краткий сигнал или insight;
    * `panel` — основной рабочий режим;
    * `takeover` — редкий глубокий режим для сравнений и крупных аналитических сценариев.

## 4. Контроль Качества и Полноты (J-Pattern)

### 4.1 Система Completeness Score
Профиль объекта считается "институционально готовым" только при 100% заполнении данных.
*   **Progress Indicator:** Визуальная полоса прогресса (напр. 9/11 полей).
*   **Completeness Banner:** Если профиль не полон, в заголовке отображается предупреждающий баннер со списком отсутствующих критических полей.
*   **Validation Gating:** Строгая валидация обязательных полей на уровне схем (`.min(1)`).

## 5. Институциональные компоненты

### 5.1 Escalation Banner
Канонический паттерн информирования об эскалации:
*   **ID:** Ссылка на тикет в Ledger.
*   **Committee:** Указание органа принятия решения (Risk/CFO).
*   **Quorum:** Прогресс подписей (напр. *2/3 Signed*).

### 5.2 Ledger Visibility
*   **Immutable Lock:** Иконка замка (`🔒`) рядом с данными, зафиксированными в Ledger (редактирование запрещено).
*   **Audit Trail:** Каждое поле в идеале должно иметь возможность просмотра истории изменений (Forensic Depth).

## 6. Брендинг и Идентификация

### 6.1 Short Name Branding
Для институциональных объектов (контрагенты, холдинги) в интерфейсе всегда отдается приоритет **Сокращенному наименованию**.
*   **Breadcrumbs & Header:** В навигации и заголовке страницы всегда выводится `shortName`. Полное юридическое имя допустимо только внутри карточки в режиме просмотра данных.
*   **Consistency:** Если `shortName` отсутствует, система должна выводить ID или заглушку, но не ломать визуальную иерархию длинным текстом.

## 7. Техническая реализация

*   **Grid:** Базовый модуль 8px.
*   **Hit Areas:** Минимальная зона клика 44x44px (`breadcrumb back`, иконки, интерактивные элементы).
*   **Focus Ring:** Четкое визуальное выделение элемента при табуляции (Accessibility).
*   **Components:** Использование `DataField` (wrapper для View/Edit) и `SidePanelForm` обязательно.
