# Чек-лист агентов: экран вызова, endpoint, trace

## 1. Цель

Единый операционный чек-лист для проверки связки:
- UI-кнопка/окно
- backend endpoint
- trace-артефакт в Control Tower / Forensics

Эффект применения: ускоряется диагностика нерабочих кнопок, быстрее локализуются разрывы UI -> API -> trace.

## 2. Общие правила проверки

1. Запускать сценарий под ролью `ADMIN` или `FOUNDER`.
2. Проверять не только HTTP 200, но и появление trace в форензике.
3. Для high-impact действий фиксировать outcome через audit trail.
4. Ошибки в UI должны показываться в интерфейсе, без browser `alert`.

## 4. Проверка экранов и кнопок (обязательный проход)

### 4.1 AI Dock

- Кнопка отправки сообщения активна только при непустом тексте.
- В ответе показывается одна строка `Почему этот ответ?` (без raw memory internals).
- Suggested actions: максимум 3.
- При экспертной эскалации открывается встроенная панель, не browser prompt.

Ожидаемый эффект: AI-поток предсказуемый, без перегруза и без скрытых модальных ошибок.

### 4.2 Экспертная эскалация (`chief_agronomist`)

- CTA виден из контекста сущности (техкарта/отклонение/поле).
- В drawer доступны действия: `Принять`, `Передать человеку`, `Создать задачу`.
- Комментарий к решению вводится в поле панели (не `window.prompt`).
- После действия фиксируется итог (`outcomeAction`, `resolvedAt`, `createdTaskId` при создании задачи).

Ожидаемый эффект: экспертный контур становится воспроизводимым и аудируемым.

### 4.3 Control Tower -> Реестр агентов

- Кнопка `Создать запрос на изменение` открывает форму без падения 500 на повторной отправке.
- Ошибки отображаются как UI-ошибки в форме, а не сырое `Request failed ...`.
- Секции `Политика памяти / Контракт ответа / Политика управления` раскрываются по требованию.

Ожидаемый эффект: конфигурация агентов проходит стабильный governance-flow.

### 4.4 Стратегия -> Прогнозы

- `Построить прогноз` недоступна без `Сезона`.
- После запуска есть `traceId`, `baseline/range/scenario delta/risk/recommendation`.
- История запусков и запись факта выполняются без блокировки всего экрана.

Ожидаемый эффект: deterministic forecasting flow проверяем end-to-end.

## 5. API и trace контрольные точки

- Chat ingress: `POST /api/rai/chat`.
- Expert review: `POST /rai-chat/expert/chief-agronomist/review`.
- Expert outcome: `POST /rai-chat/expert/reviews/:reviewId/outcome`.
- Forecast run: `POST /ofs/strategy/forecasts/run`.
- Forensics: `GET /rai/explainability/trace/:traceId/forensics`.
- Topology: `GET /rai/explainability/trace/:traceId/topology`.

## 6. Критерии “готово”

- Для каждой роли из матрицы есть рабочий UI-вход и успешный endpoint вызов.
- Для каждого high-impact потока фиксируется trace и доступен форензике.
- Пользовательский текст в agent-поверхностях соответствует `LANGUAGE_POLICY.md`.
- В рабочих экранах нет длинных поясняющих плашек, контекст даётся через локальные подсказки.

## 8. REAL E2E: от контрагента до Техкарты (без симуляции)

### 8.1 Жёсткие правила прохода

1. Работать в одном tenant и под ролью `ADMIN`/`FOUNDER`.
2. Использовать chat-first там, где есть реальные WRITE tools.
3. Для каждого шага подтверждать факт на экране и в данных (ID сущности, статус, связь).
4. Не использовать демонстрационные триггеры симуляции (`DETECT_CONFLICT`, demo-flow для подмены данных).

Ожидаемый эффект: проход воспроизводим, без подмены бизнес-логики симуляционными кнопками.

### 8.2 Алгоритм (chat приоритет + проверка экранов)

1. Команда в AI-чате: `Зарегистрируй контрагента по ИНН 7707083893`.
Эффект: создаётся/находится реальный `party` через `register_counterparty`.
Проверка экрана: `/parties` -> запись появилась в реестре контрагентов.

2. Команда в AI-чате: `Создай CRM-аккаунт "Хозяйство Волна Юг", ИНН 7707083893`.
Эффект: создаётся реальный `account` через `create_crm_account`.
Проверка экрана: `/consulting/plans` -> в селекте "Хозяйство" появляется новый аккаунт.

3. Команда в AI-чате: открыть рабочее пространство аккаунта.
Формулировка: `Открой карточку аккаунта` после выбора строки аккаунта в UI.
Эффект: чат получает контекст `accountId`, уменьшается риск ошибки на следующих CRM-командах.
Проверка экрана: панель AI показывает structured result по выбранному аккаунту.

4. Команда в AI-чате: `Добавь контакт Иван Петров, email ivan@example.com, phone +79990000000`.
Эффект: создаётся реальный `contact` через `create_crm_contact`.
Проверка экрана: CRM workspace аккаунта -> контакт присутствует.

5. Экранный шаг (реальный): создать поле через реестр полей (`POST /registry/fields`).
Эффект: появляется физический `field`, который нужен сезону и генерации техкарты.
Проверка экрана: `/assets/fields` -> поле отображается в списке.

6. Экранный шаг (реальный): создать сезон через `/seasons` с привязкой `fieldId`.
Эффект: появляется `season`, пригодный для генерации техкарты.
Проверка экрана: селект сезонов в `/consulting/plans` содержит новый сезон.

7. Экранный шаг (реальный): создать план в `/consulting/plans` c `accountId + seasonId`.
Эффект: появляется `harvestPlan` в статусе `DRAFT`.
Проверка экрана: карточка плана доступна по `/consulting/plans/:id`.

8. Команда в AI-чате (в контексте выбранных поля и сезона): `Сделай техкарту рапса`.
Эффект: chat-intent `tech_map_draft` запускает реальную генерацию карты, создаются стадии/операции/ресурсы.
Проверка экрана: открыть созданную карту в `/consulting/techmaps/:id`, счётчики стадий/операций/ресурсов > 0.

9. Экранный шаг (реальный): перевести техкарту в `REVIEW -> APPROVED -> ACTIVE`.
Эффект: карта допускается в контур исполнения.
Проверка экрана: статус `ACTIVE`, кнопка перехода в execution активна.

10. Экранный шаг (реальный): открыть `/consulting/execution`, затем `/consulting/execution/agronomist`.
Эффект: операции активной техкарты попадают в рабочую ленту исполнения.
Проверка экрана: карточки операций видимы, ошибка загрузки отсутствует.

11. Экранный шаг (реальный): нажать `Start` на операции.
Эффект: создаётся/обновляется `executionRecord`, статус переходит в `IN_PROGRESS`.
Проверка экрана: карточка операции показывает прогресс, повторный `Start` недоступен.

12. Экранный шаг (реальный): завершить операцию через `Complete`.
Эффект: `executionRecord -> DONE`, фиксируются фактические ресурсы и outbox-событие.
Проверка экрана: операция отмечена завершённой, в ленте виден факт выполнения.

Эффект исключения: исключаем ложные прогоны с `NEEDS_MORE_DATA`, фиксируем только завершённые real-результаты.
