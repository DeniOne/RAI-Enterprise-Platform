# Domain: CRM (Consulting CRM) — Consulting Control Plane

> **Contour:** 1 (Enterprise Back-Office) | **Module:** CRM/CMR  
> **Status:** ARCHITECTURAL DEFINITION  
> **Key Principle:** CMR показывает, кто принял решение, когда, почему и к чему это привело.

---

## 1. Концепция: Consulting Control Plane

CMR не является инструментом операционного управления (это задача APL/FSM). CMR — это **слой принятия решений и контроля ответственности**.

### 1.1. Отличия от классических систем
- **❌ Журнал агронома:** Нет детальных отчётов по ГСМ, запчастям или конкретным выработкам трактористов.
- **❌ ERP:** Нет бухгалтерии, складов и первичной документации.
- **✅ CMR:** Контракт → Сезон → Культура → Результат. Это фиксация *консалтинговой ценности*.

---

## 2. Роль ИИ (Agro Orchestrator) в CMR

ИИ здесь — это не просто чат-бот, а **автономный аналитик (Brain)**, который:
1.  **Inference**: Читает агроданные (NDVI, метео, почва).
2.  **Monitoring**: Следит за фазами (1–14 этапов по Рапсу).
3.  **Risk Assessment**: Поднимает алерты (гербицидный стресс, окно защиты).
4.  **Action**: Предлагает решения и сам создаёт события в ленте.

---

## 3. Контур синхронизации: Deviation Review (Технический разбор)

**Критическая дыра**: Разрыв между "ИИ обнаружил риск" и "Менеджер зафиксировал решение". Без формального разбора этот этап тонет в мессенджерах.

### 3.1. Сущность Deviation Review (Трипартитный консенсус)
Любой технический разбор, связанный с аномалией, считается **НЕЗАВЕРШЁННЫМ**, пока хозяйство явно не выразило позицию.
- **Принцип**: Молчание = событие. Отсутствие комментария = управленческое решение.
- **Trigger**: ИИ выявляет отклонение факта (APL) от эталона (АТК), превышающее порог риска.
- **Process**:
    1. **Trigger**: ИИ обнаружил отклонение.
    2. **Analysis**: Техагроном сформировал Технический разбор.
    3. **Review**: CMR отправила проект разбора хозяйству.
    4. **SLA (Silo)**: Ожидание ответа (обычно 48 часов).
    5. **Outcome**: Закрытие разбора согласно реакции (или её отсутствию).

### 3.2. Сценарии развития событий

#### Вариант А: Хозяйство участвует (Status: agreed/disagreed)
- **Результат**: Ответственность = **SHARED**.
- **Decision Log**: «Корректировка АТК согласована сторонами. Ответственность разделена: Консультант за рекомендации, Хозяйство за исполнение».

#### Вариант Б: Хозяйство молчит (Status: ignored)
Если SLA истекло без реакции:
1.  **Сценарий «Disclaimer»**: Если риски критичны — статус консалтинговой ответственности меняется на **SUSPENDED** (аннулировано).
2.  **Сценарий «Forced Adjustment»**: Если процесс нельзя остановить — ИИ применяет правки АТК принудительно, но статус ответственности = **CLIENT_ONLY**.
- **Decision Log**: «Хозяйство не предоставило позицию. Консалтинговая ответственность аннулирована/переложена на сторону клиента».

### 3.3. Модель данных DeviationReview
```typescript
{
  id: string;
  projectId: string;
  stageId: string;
  deviationSummary: string; // Что произошло
  aiImpactAssessment: string; // Оценка риска ИИ (4-7%)
  techConclusion: string; // Заключение агронома
  proposedActions: 'ADJUST_ATK' | 'NONE' | 'ESCALATE';
  
  clientResponse: {
    status: 'pending' | 'agreed' | 'disagreed' | 'ignored';
    comment?: string;
    respondedAt?: Date;
  };
  
  responsibilityMode: 
    | 'SHARED'
    | 'CLIENT_ONLY'
    | 'CONSULTANT_ONLY'
    | 'LIABILITY_SUSPENDED';

  status: 'open' | 'resolved' | 'escalated';
}
```

## 4. Стратегические Усилители (Strategic Amplifiers)

Эти слои превращают CMR из "учетной системы" в стратегический актив компании.

### 4.1. Knowledge Accretion Layer (Накопление Знаний)
Система не просто хранит лог, а формирует **Expertise Objects**.
- **Механика**: Каждое `DeviationReview` -> обобщенный кейс ("Задержка обработки при T<15°C на 5-м этапе безопасна").
- **Future Use**: ИИ использует эти кейсы для будущих рекомендаций ("Основано на 37 аналогичных кейсах").

### 4.2. Confidence & Counterfactual Index
- **Confidence Score**: ИИ маркирует каждую рекомендацию классом уверенности:
    - 🔵 **High**: Подтверждено историей и данными.
    - 🟡 **Medium**: Данные неполные.
    - 🔴 **Low**: Экстраполяция (требует внимания человека).
- **Counterfactuals**: Система хранит не только принятое решение, но и **отвергнутые альтернативы** (чтобы через год понять, "что было бы, если...").

### 4.3. Client Maturity Model (Зрелость Клиента)
Оценка готовности хозяйства к высоким технологиям.
- **Grades**: 🟢 Высокая / 🟡 Средняя / 🔴 Низкая.
- **Влияние**: Низкая зрелость -> более жесткие SLA, меньше автоматики, принудительный контроль.

### 4.4. Legal Shield Layer
Каждое решение формирует **юридический след**:
- Кто принял? На основании каких данных? При какой уверенности ИИ? Какие риски были подсвечены?
- Это превращает Decision Log в доказательную базу для арбитража.

## 5. Risk & Insurance Framework (New Core)

Мы не “делим урожай” — мы управляем рисками.

### 5.1. Типология Рисков (Risk Taxonomy)
| Тип Риска | Кто контролирует (Controllability) | Лиабилити (Liability Mode) | Страхование |
|-----------|-------------------------------------|----------------------------|-------------|
| **Agronomic** (Технология) | Consultant / Shared | 🟢 **SHARED** | Нет |
| **Climate** (Погода) | Force Majeure | 🟡 **INSURABLE** | **ДА** |
| **Operational** (Техника/Люди) | Client | 🔴 **CLIENT_ONLY** | Нет |
| **Regulatory** (Законы/Рынок) | Shared / Force Majeure | 🟢 **SHARED** / 🟡 **INSURABLE** | **ДА** |

### 5.2. Матрица Ответственности (Hard Logic)
Эта логика "зашита" в систему принятия решений:
1.  **Рекомендация согласована и выполнена** -> `SHARED`
2.  **Рекомендация согласована, но НЕ выполнена** -> `CLIENT_ONLY`
3.  **Отклонение без ответа клиента (Silence)** -> `CLIENT_ONLY` / `LIABILITY_SUSPENDED`
4.  **Ошибка в АТК (доказанная)** -> `CONSULTANT_ONLY`
5.  **Климатический форс-мажор** -> `INSURANCE_COVERAGE`

### 5.3. Insurance Layer (Страховой слой)
Страхование — это не отдельный продукт, а "плагин" для снижения рисков.
- **Trigger**: ИИ предлагает страховку, если `Controllability = Force Majeure` и `Impact = High`.
- **Value**: Андеррайтинг на основе `ATK` (Эталон) + `Decision Log` (История) + `Risk Profile`.

## 6. Сущности верхнего уровня

### 3.1. Клиент (Агрохозяйство)
*Компактная карточка.*
- **Атрибуты**: Название, Регион, Площадь зяби, Текущие культуры.
- **Статус**: Актив / Риск (если есть алерты) / Пауза.

### 3.2. Контракт / Проект
*Единица консалтингового обязательства.*
- **Поля**: Культура, Сезон, Формат (Аудит/Сопровождение), KPI (Урожайность/Маржа).

### 3.3. Сезон (Season/APL)
*Интеграция с существующей архитектурой.*
- **UI-проекция**: Текущий этап, Индикатор риска (🟢🟡🔴), Словесный прогноз (без цифр).

### 3.4. Лента решений (Decision Log) — КРИТИЧНО
*Юридическая и техническая память.*
- [Дата] | [Этап] | [Решение] | [Основание] | [Ответственный].

## 4. UI Design Canon для CMR

### Общие принципы
1.  **Scorched Earth**: Только каноничные стили (Geist, #FAFAFA, rounded-2xl).
2.  **Minimalism**: Скрывать микропроцессы. Только "Brain" выводы.
3.  **Hierarchy**: Текст и пространство вместо Bold. 

### Дерево экранов
1.  **Дашборд (Control Overview)**: Статусы проектов, критические риски, кнопка подтверждения ИИ-рекомендаций.
2.  **Реестр Клиентов**: Список хозяйств с цветовой индикацией рисков.
3.  **Карточка Проекта**: Таймлайн 14 этапов (свёрнуто), текущий статус, лента решений.
4.  **Decision Log (Global)**: Сквозной поиск по всем принятым решениям для анализа эффективности.
5.  **Risk Registry**: Список всех активных угроз по всем хозяйствам.

## 5. Что скрыто от UI (Backend Logic)
- **FSM (State Machine)**: Все 14 этапов Рапса крутятся внутри.
- **Rule Engine**: Расчёт ЭПВ, окон внесения, совместимости препаратов.
- **Economic Simulator**: Расчёт ROI и маржинальностей (выдаёт только выводы).
