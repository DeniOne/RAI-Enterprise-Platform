# A_RAI — Операционная система консалтингового бизнеса RAI

> **Версия:** 2.0 | **Дата:** 2026-03-04  
> **Статус:** System Manifesto — Top-Level Architecture Document  
> **Кодовое имя:** **A_RAI** (Agent RAI) | в разговоре: _Рэй, Рая, А-РАЙ_  
> **Переработан:** Исправлена ключевая рамка — RAI_EP управляет не фермой, а консалтинговым бизнесом

---

## Важнейший контекст: Кто мы и что мы делаем

**RAI_EP — это НЕ система управления фермой.**

RAI_EP — это **операционная система результатного агроконсалтинга**.

Наши клиенты — агрохозяйства. Наш продукт — доказанный рост урожайности и управляемая экономика производства. Наш бизнес строится на формуле:

```
Вознаграждение = Δ урожайности × Доля × Рыночная цена
```

Где `Δ = Фактический урожай − Базовая урожайность хозяйства`.

Мы зарабатываем только тогда, когда хозяйство-клиент зарабатывает больше обычного.

---

## Стратегическая проблема масштабирования

Сейчас 1 опытный агроном-консультант может качественно вести 10–15 хозяйств.

Цель RAI v3.0 — **500+ хозяйств в разных юрисдикциях**.

Это математически невозможно без **институционального AI-компонента**, который:
- сохраняет экспертизу консультанта в систему (институциональная память)
- автоматизирует рутину (анализ отклонений, мониторинг, типовые рекомендации)
- масштабирует экспертное качество без пропорционального роста штата
- не теряет контекст ни одного клиента при любой нагрузке

Это и есть **A_RAI**.

## A_RAI — интеллектуальный ассистент консалтинговой команды RAI

A_RAI — это AI-компонент, который позволяет команде RAI вести кратно больше хозяйств без потери качества консалтинга:

```
Без A_RAI:  1 консультант → 10–15 хозяйств (ограничен когнитивной нагрузкой)

С A_RAI:    1 консультант + A_RAI → 50–100 хозяйств (A_RAI берёт рутину)
            Команда RAI + A_RAI → 500+ хозяйств (институциональное масштабирование)
```

**A_RAI не заменяет агронома-консультанта. A_RAI автоматизирует рутину и масштабирует экспертизу.**

## Что A_RAI делает конкретно

### Для консультанта RAI:

| Задача | Без A_RAI | С A_RAI |
|--------|----------|---------|
| Контроль 50 хозяйств | Невозможно | A_RAI мониторит все, консультант только реагирует на S3/S4 |
| Разработка техкарты | 3–5 часов на одно хозяйство | A_RAI генерирует DRAFT, консультант правит и подписывает |
| Анализ отклонений план/факт | Ручной анализ по каждому | A_RAI автоматически детектирует, ранжирует по риску |
| Помять контекст клиента | В голове / Excel | Институциональная память: каждый сезон, каждое решение |
| Экономический анализ what-if | Часы в Excel | EconomistAgent: сценарии за секунды |
| Обнаружение риска | Постфактум, при звонке | MonitoringAgent: превентивно, до потерь |

### Для хозяйства-клиента (через консультанта):

- Получает профессиональную техкарту под **свой конкретный контекст** (не шаблон)
- Видит объяснение каждой рекомендации с источником данных и confidence
- Получает алерты о рисках до того, как они стали убытками
- Имеет прозрачную экономику: каждый рубль затрат = доказанное событие в системе

## Шесть контуров RAI — где A_RAI в каждом

По бизнес-архитектуре v2.0, RAI_EP имеет 6 контуров управления:

| Контур RAI | Роль A_RAI |
|-----------|-----------|
| **Производственный** (Season, TechMap, Task) | `AgronomAgent` генерирует DRAFT техкарты; `ControllerAgent` мониторит выполнение задач |
| **Доказательный** (Observation, IntegrityGate, Evidence) | `ControllerAgent` запрашивает Evidence; автоматически детектирует weak evidence → ChangeOrder |
| **Рисков и решений** (RiskEngine, CMR, Quorum) | `ControllerAgent` (READ-ONLY к RiskToolsRegistry); S3/S4 — Human review, не AI |
| **Финансовый** (EconomyEvent, Ledger, Budget) | `EconomistAgent` — только READ, аналитика и what-if; запись в Ledger — только Human |
| **ИИ-сопровождения** (ShadowAdvisory, Pilot, Rollout) | A_RAI работает в режиме Shadow → Pilot → Rollout с kill-switch |
| **Институциональной безопасности** (Audit, Tenant, Outbox) | A_RAI полностью изолирован по `companyId`; каждое действие — в AuditLog с traceId |

## Инварианты: что A_RAI НИКОГДА не делает

Эти инварианты вытекают из `RAI BUSINESS ARCHITECTURE v2.0` и `RAI STRATEGY v3.0`:

```
❌ A_RAI не меняет статус HarvestPlan или TechMap напрямую
   → Статусы меняет только FSM через Human-подтверждённое действие

❌ A_RAI не создаёт записи в финансовом Ledger
   → Ledger пишет только EconomyService после подтверждённого события

❌ A_RAI не активирует техкарту
   → Активация требует Integrity Admission + Human КЭП

❌ A_RAI не обходит RiskGate / IntegrityGate
   → Эти контуры стоят ПЕРЕД слоем AI, не после

❌ A_RAI не выдаёт рекомендацию без источника данных
   → Evidence Protocol обязателен: нет данных = "Недостаточно информации"

❌ A_RAI не смешивает данные разных хозяйств
   → Tenant isolation: companyId enforcement на каждом уровне

✅ A_RAI всегда: предлагает → объясняет (формула + источник) →
                 ждёт подтверждения агронома/директора → фиксирует в AuditLog
```

## Как A_RAI масштабирует консалтинг (конкретные сценарии)

### Сценарий 1: Разработка техкарты — от первого контакта до подписанного документа

#### Предсценарий 1.0 — Контекстный триаж (сбор данных об урожае)

Прежде чем A_RAI сможет сгенерировать техкарту, необходимо собрать достаточный контекст. Это самый важный этап — без него расчёты будут неточными или технически невозможными.

```
Консультант инициирует новый проект:
"Новое хозяйство: ООО Агро-Север, рапс озимый, сезон 2026"

── СЛОЙ 1: ЮРИДИЧЕСКИЙ И ОРГАНИЗАЦИОННЫЙ КОНТЕКСТ ────────────────

A_RAI → CRM-запрос / форма заполнения:
  • Юридическое лицо: наименование, ИНН, регион, УГРЮЛ-статус
  • Хозяйство: тип (ЛПХ / КФХ / АО / ООО), размер, управляющий
  • Контрагент по договору: роли, реквизиты для ConsultingRecord
  • Уровень цифровой зрелости: есть ли 1С, GPS-трекеры, лаборатория?

A_RAI (KnowledgeAgent):
→ Проверяет: это новое хозяйство или повторный клиент?
  - Если повторный → загружает из Episodic Memory историю прошлых сезонов,
    принятые/отклонённые рекомендации, итоговые урожайности, проблемные поля
  - Если новый → инициирует полный сбор контекста
→ Confidence check: если история есть — какой давности? Актуальна ли?

── СЛОЙ 2: ПРОСТРАНСТВЕННЫЙ КОНТЕКСТ (ПОЛЕ) ──────────────────────

A_RAI → запрашивает данные поля:
  • Идентификатор поля: кадастровый номер / внутренний ID
  • Площадь (га), геометрия (полигон): загрузить из ГИС или нарисовать
  • Регион, зона: влияет на RegionProfile (MARITIME / STEPPE / CONTINENTAL)
  • Рельеф: уклон, наличие депрессий, дренаж
  • Наличие ограждений: водоохранная зона, пасека в радиусе 500м, буферы

A_RAI (KnowledgeAgent):
→ Загружает RegionProfile: базовые окна BBCH, ГТК, типичные риски, вредители
→ Если геометрия известна → проверяет satellite data последние 30 дней (NDVI)
→ Выставляет флаги ограничений: "Водоохранная зона на юге — блок СЗР класс 1-2"

── СЛОЙ 3: АГРОНОМИЧЕСКИЙ КОНТЕКСТ (ИСТОРИЯ ПОЛЯ) ───────────────

A_RAI → запрашивает историю возделывания:
  • Предшественники за 3–5 лет: культура, сорт/гибрид, урожайность
    [КРИТИЧНО] Предшественник рапса — пшеница, ячмень? Подсолнечник?
    [БЛОКИРОВКА] Рапс после рапса < 4 лет → Hard Stop: риск склеротиниоза
  • История применения СЗР на предшественнике:
    [КРИТИЧНО] Были ли имидазолиноны (граминициды)? → риск фитотоксичности
  • Фитопатологический фон: Phoma lingam, Sclerotinia, Alternaria — история
  • Проблемные зоны: где по факту давало меньше? Переувлажнения, сорняки?

A_RAI (AgronomAgent):
→ Оценивает аллелопатические риски по предшественнику
→ Рассчитывает остаточный азотный фон (если есть данные)
→ Формирует предварительный риск-профиль поля: LOW / MEDIUM / HIGH

── СЛОЙ 4: ПОЧВЕННЫЙ КОНТЕКСТ ────────────────────────────────────

A_RAI → запрашивает почвенные анализы:
  • Дата последнего анализа (confidence decay: > 18 мес → понижается)
  • pH, гумус (%), грансостав (тип почвы)
  • Макроэлементы: N, P₂O₅, K₂O (мг/кг)
  • Микроэлементы для рапса: S, B, Mg, Zn (критично!)

A_RAI (AgronomAgent):
→ Если S < порога → WARN: "Сера не проанализирована. Рапс теряет до 30% урожая.
   Рекомендую: свежий анализ S/B или принять региональное среднее (conf: 0.35)"
→ Если анализ > 2 лет → WARN: confidence снижен до 0.45, нужен повторный
→ Если анализ отсутствует → запрос разрешения на импутацию из RegionProfile

── СЛОЙ 5: СЕМЕННОЙ МАТЕРИАЛ ─────────────────────────────────────

A_RAI → запрашивает параметры семян:
  • Культура, тип (озимый/яровой), сорт/гибрид
  • Поставщик, номер партии, год производства
  • Масса тысячи семян (МТС, г) — для расчёта весовой нормы высева
  • Лабораторная всхожесть партии (%) — критично для калибровки сеялки
  • Энергия прорастания (%)
  • Протравка: есть ли, чем, дата обработки

A_RAI (AgronomAgent):
→ Если МТС или всхожесть не указаны → WARN + запрашивает фото этикетки мешка
→ Предрассчитывает ориентировочную весовую норму (с confidence 0.6 до уточнения)

── СЛОЙ 6: ЦЕЛЕВЫЕ ПАРАМЕТРЫ УРОЖАЯ ──────────────────────────────

A_RAI → запрашивает намерение урожая:
  • Целевая урожайность: т/га (min / opt / max сценарий)
  • База урожайности хозяйства: среднее за 3 последних сезона
    [если нет истории → A_RAI предлагает региональный норматив как базу]
  • Качественные показатели: масличность ≥ 42%, влажность ≤ 8%
  • Бюджетный лимит: руб/га (Budget Cap для Contract Core)
  • Допустимый риск-профиль: CONSERVATIVE / BALANCED / AGGRESSIVE
  • Дата планируемого сева (или: "посчитай сам по BBCH")

A_RAI (EconomistAgent, параллельно):
→ Сразу считает экономику: при целевой урожайности и текущих ценах
→ Рассчитывает точку безубыточности и ожидаемое вознаграждение RAI (Δ-формула)

── СЛОЙ 7: РЕСУРСНЫЙ КОНТЕКСТ ────────────────────────────────────

A_RAI → запрашивает доступные ресурсы:
  • Машинный парк: самоходная + прицепная техника (тяговые классы, год)
  • Опрыскиватель: ширина захвата, производительность га/ч
  • Комбайн: ширина жатки, оснащён ли рапсовым столом?
    [КРИТИЧНО] Без рапсового стола — потери до 30% от растрескивания стручков
  • Механизаторы: количество компетентных на каждый тип агрегата
  • Склад СЗР и удобрений: есть ли, как далеко от поля?
  • Ближайшая лаборатория: для подтверждения Evidence по операциям

─────────────────────────────────────────────────────────────────
РЕЗУЛЬТАТ ТРИАЖА:

A_RAI генерирует Контекстный отчёт консультанту:

✅ Данные получены (confidence ≥ 0.7):
   Юрлицо, Поле, Предшественник (пшеница), Почва (pH 6.1, NPK)
   Семена (МТС 4.2г, всхожесть 92%), Техника, Бюджет

⚠️ Данные получены с низкой достоверностью (conf 0.3–0.7):
   Сера, Бор — нет свежего анализа (conf: 0.35). Использую регион.среднее.
   Урожайность базы — взята из регионального норматива, нет истории хозяйства.

❌ Данные отсутствуют (блокировка расчёта):
   - нет данных о СЗР на предшественнике (риск имидазолинонов)
   - нет подтверждения: рапсовый стол на комбайне есть/нет?

→ Консультант дозаполняет пробелы
→ После снятия ❌-блоков → переход к Сценарию 1.1
```

**Время триажа:** 15–30 минут (вместо 2–3 часов ручного сбора данных).

#### Основной поток 1.1 — Генерация техкарты

```
После прохождения триажа (все ❌ сняты, ⚠️ приняты агрономом):

A_RAI (AgronomAgent):
→ Запускает AgroDeterministicEngine:
   - computeSeedingRate(МТС=4.2г, всхожесть=92%, цель=0.6млн/га) → 2.74 кг/га
   - computeFertilizerDose(почва, цель=3.5т/га) → N:120, P:60, K:80, S:40 кг/га
   - computeGDDWindow(RegionProfile=MARITIME, гибрид="Лумина") → сев: 25 авг–10 сен
   - validateTankMix(фунгицид×ретардант) → COMPATIBLE
   - validateDAG(10 операций) → 0 Hard Stops
→ Генерирует DRAFT TechMap: 10 операций с BBCH-окнами и зависимостями

A_RAI (EconomistAgent, параллельно):
→ Бюджет по категориям: семена 18%, удобрения 28%, СЗР 15%, ГСМ 12%...
→ Итог: 34,800 руб/га ≤ Budget Cap 40,000 ✅
→ ROI при 3.5 т/га × 35,000 руб/т → маржа 17,200 руб/га
→ Вознаграждение RAI (Δ = 3.5 − 3.0 база) × 30% × 35,000 = 5,250 руб/га

ResponseComposer:
→ Виджет: DRAFT техкарты (10 операций, интерактивная таблица)
→ Экономика: план/заб/макс сценарии рядом

```

## Эволюция: как A_RAI становится умнее

```

                ↓
Institutional Memory: паттерны успешных решений → Knowledge Graph
                ↓
Следующий сезон: A_RAI точнее, быстрее, увереннее

После 5 сезонов: A_RAI знает о рапсе в Краснодарском крае больше,
чем любая другая система в мире — потому что это реальные данные реальных хозяйств.
```
