# A_RAI — Blueprint интерактивного общения пользователя и агента

Дата: 2026-03-07  
Статус: Implemented Canon / interaction blueprint closed in code  
Назначение: зафиксировать целевую модель удобного, многослойного и управляемого общения `пользователь <-> оркестратор <-> агент <-> рабочие окна` без засорения основного чата.

---

## 1. Зачем нужен этот blueprint

Текущий AI-контур уже умеет:

- держать постоянный AI Dock слева;
- вести основную историю сообщений;
- передавать `workspaceContext` вместе с запросом;
- открывать структурированный вывод поверх правой рабочей зоны;
- показывать сигналы и структурные виджеты;
- связывать ответ агента с правым контентом.

Этого уже достаточно, чтобы сказать: фундамент есть.

Но текущая композиция ещё не даёт по-настоящему удобного взаимодействия:

- чат перегружается служебными сообщениями;
- правая зона пока работает как один почти одинаковый overlay-паттерн;
- нет нормального деления по тяжести задачи;
- нет управляемого цикла добора контекста;
- нет настоящей мультиоконности, где агент может вести несколько рабочих контекстов одновременно.

Этот blueprint нужен, чтобы превратить уже существующий фундамент в зрелую продуктовую модель.

---

## 2. Главный принцип

### Чат = генеральная струна диалога

Чат не должен становиться свалкой из:

- служебных уточнений;
- технических ошибок;
- длинных списков недостающих полей;
- структурированного вывода, который лучше показать в отдельной рабочей форме.

Чат должен оставаться:

- основной нитью разговора;
- местом постановки задачи;
- местом коротких решений и подтверждений;
- местом высокоуровневого статуса.

### Правая рабочая зона = основная поверхность продукта

Правая зона не принадлежит агенту.

Это основная рабочая поверхность самого продукта, где пользователь работает с:

- меню;
- дашбордами;
- таблицами;
- картами;
- карточками контрагентов;
- карточками полей;
- документами;
- рабочими экранами домена.

Агент не должен забирать эту поверхность себе как “собственное полотно”.

Правильная модель другая:

- правая зона остаётся продуктовой рабочей поверхностью пользователя;
- агентные окна открываются поверх неё как оверлэй;
- фон продукта остаётся видимым, читаемым и узнаваемым;
- пользователь может одновременно работать с экраном продукта и с контекстом, который открыл агент.

Именно поверх этой зоны должны жить агентные действия:

- структурированный результат;
- добор контекста;
- сравнение вариантов;
- сценарные панели;
- подтверждение промежуточных шагов;
- связанные переходы по системе.

## 3. Что уже реализовано и на что можно опереться

По текущему коду уже подтверждено:

- слева есть постоянный AI Dock с регулируемой шириной;
- рабочий маршрут и page-context уже попадают в чатовый запрос;
- структурные виджеты уже отдаются как отдельный слой данных;
- справа уже есть overlay-вывод поверх рабочей зоны;
- есть механизм “сигнал -> открыть связанный фрагмент справа”;
- состояние чата и правого вывода уже централизованно хранится в store.
- для `Агроном-А -> tech_map_draft` уже есть managed clarification loop;
- для `Экономист-А -> compute_plan_fact` уже есть managed clarification loop;
- backend уже отдаёт `pendingClarification`, `workWindows[]`, `activeWindowId`;
- `context_acquisition` и `context_hint` уже существуют как реальные типы окон;
- окно умеет режимы `inline / panel / takeover`;
- окна уже можно сворачивать, восстанавливать и складывать в стек;
- у окон уже есть `category`, `priority`, `parentWindowId`, `relatedWindowIds`;
- completed resume-path уже подтверждён live HTTP smoke, а не только unit/spec;
- окна уже умеют post-result actions:
  - `Открыть результат`
  - `Открыть поле`
  - `Перейти к техкартам`
- окна уже показывают источник запроса строкой вида:
  - `Источник: “Составь черновик техкарты по озимому рапсу”`

Это значит, что не нужно изобретать архитектуру заново.

Нужно сделать следующее:

- перевести один тяжёлый overlay в типизированную систему окон;
- перестроить левый dock в более собранную композицию;
- ввести режимы вывода по тяжести задачи;
- добавить managed clarification loop;
- позволить агенту открывать несколько активных рабочих окон.

Дополнительное подтверждение после реализации второй части:

- левый `AI Dock` пересобран вокруг канона `header -> signals -> history -> composer`;
- тяжёлая отдельная карточка мини-инбокса убрана;
- `SignalsStrip` теперь живёт внутри dock как компактный обзор сигналов;
- legacy `widgets[]` больше не являются primary UX-парадигмой и автоматически мигрируются в `workWindows[]`;
- введены дополнительные first-class window families:
  - `structured_result`
  - `related_signals`
  - `comparison`
- `RaiOutputOverlay` больше не рендерит old-style widgets как основной путь;
- для `EconomistAgent` появился отдельный non-legacy rich-output path с `comparison` окном и companion signal-window.
- first-class contract layer больше не является только markdown-addendum:
  - `IntentRouter`
  - `Supervisor resume-path`
  - clarification composition
  уже используют общий backend contract source для:
  - `Focus`
  - `Intent`
  - `Required Context`
  - `UI Action Surface`

### 3.1 Что уже закрыто живыми slices

Первые эталонные slices уже доведены до живого состояния:

- пользователь просит техкарту;
- оркестратор выбирает `agronomist / tech_map_draft`;
- если не хватает `fieldRef` и `seasonRef`, чат получает короткий человеческий ответ;
- справа открывается `context_acquisition` окно;
- дополнительно приходит `context_hint`, которое уходит в стек;
- контекст может подтягиваться из `workspaceContext`;
- после добора контекста работает авто-продолжение без повторного ручного запроса;
- completed result возвращается в тот же thread;
- result path тоже уже multi-window:
  - главное окно результата;
  - hint-окно дальнейших действий;
- этот же platform-pattern уже работает и для `economist / compute_plan_fact`:
  - clarification overlay для missing `seasonId`;
  - auto-resume после добора сезона;
  - completed result windows;
  - route-action обратно в финансовый workflow;
- весь этот путь подтверждён:
  - unit/spec;
  - web tests;
  - live HTTP smoke.

Итог: blueprint больше не является только “идеей на будущее”. Он уже реально работает на всех 4 reference agent families:

- `Агроном-А / tech_map_draft`
- `Экономист-А / compute_plan_fact`
- `Знание-А / query_knowledge`
- `Мониторинг-А / emit_alerts`

## 4. Целевая модель взаимодействия

### 4.1 Один запрос, два канала ответа

Каждый запрос пользователя может рождать два типа реакции:

1. Короткая реплика в чат  
   Пример:
   `Чтобы продолжить, мне нужны поле и сезон. Я открыл панель добора контекста справа.`

2. Оверлэй-окно поверх правой зоны  
   Пример:
   `Панель добора контекста для техкарты`

Это и есть каноническая модель:

- в чат идёт короткий смысл;
- поверх правой зоны открывается рабочее действие агента.

### 4.2 Чат не дублирует правую зону

Чат не должен:

- повторять таблицу, уже открытую справа;
- перечислять 15 недостающих атрибутов;
- выдавать длинный structured payload;
- играть роль формы.

Если контент лучше воспринимается как действие, форма, сравнение, список или рабочая панель, он должен открываться как оверлэй поверх правой зоны.

## 5. Три режима вывода

Текущая проблема UI в том, что почти всё ощущается одинаково тяжёлым. Это нужно сломать.

### 5.1 `Inline insight`

Короткий режим.

Подходит для:

- рекомендации;
- предупреждения;
- одного найденного риска;
- короткого summary;
- маленькой подсказки по контексту.

Форма:

- компактная карточка-оверлэй в правой верхней части рабочей зоны;
- не перекрывает основную работу;
- легко исчезает;
- может быть закреплена в более крупное окно.

Примеры:

- `Найдено 3 отклонения`
- `Для техкарты не хватает сезона`
- `План-факт просел ниже порога`

### 5.2 `Work panel`

Основной режим.

Подходит для:

- структурированного ответа агента;
- добора контекста;
- показа связанного бэклога;
- просмотра рисков;
- показа таблицы среднего размера;
- промежуточной рабочей формы.

Форма:

- крупная оверлэй-панель поверх правой зоны;
- фон справа остаётся видимым и узнаваемым;
- ширина 40–60% рабочей зоны для аналитики;
- ширина 60–75% для таблиц, сравнений и контекстных форм.

Это должен быть канонический default-режим.

### 5.3 `Takeover mode`

Редкий режим.

Подходит для:

- пошагового анализа;
- мастера действий;
- сравнения вариантов;
- больших таблиц;
- построения решения из нескольких шагов.

Форма:

- почти полный захват правой зоны оверлэй-слоем;
- использовать только там, где человеку реально нужно сосредоточиться на рабочем сценарии.

Запрет:

- не превращать любой обычный ответ в takeover.

## 6. Мультиоконность агента

Это правильная идея, и она хорошо ложится на вашу архитектуру.

### 6.1 Что означает мультиоконность

Агент должен уметь открыть не одно абстрактное “окно вывода”, а несколько живых оверлэй-окон, каждое из которых связано:

- с конкретным запросом;
- с конкретной подтемой;
- с конкретным типом работы.

Примеры параллельных окон:

- `Техкарта: добор контекста`
- `Отклонения по полю 12`
- `План действий по сезону 2026`
- `Список связанных рисков`

Это не должно ломать базовую рабочую модель пользователя:

- пользователь продолжает видеть продуктовый экран;
- агент поверх него открывает нужные окна по контексту;
- окна можно сворачивать, возвращать и переключать;
- основная правая зона не превращается в “территорию агента”.

### 6.2 Какой должна быть модель окна

Каждое окно должно иметь:

- `windowId`
- `threadId`
- `originMessageId`
- `agentRole`
- `title`
- `type`
- `mode`
- `status`
- `routeContext`
- `payload`
- `actions`
- `isPinned`
- `isDirty`
- `openedAt`
- `updatedAt`

Минимальная форма:

```ts
type AiWorkWindowMode = "inline" | "panel" | "takeover";

type AiWorkWindowStatus =
  | "ready"
  | "needs_user_input"
  | "waiting_system_context"
  | "completed"
  | "dismissed";

interface AiWorkWindow {
  windowId: string;
  threadId: string;
  originMessageId: string;
  agentRole: string;
  title: string;
  type: string;
  mode: AiWorkWindowMode;
  status: AiWorkWindowStatus;
  routeContext?: string;
  payload: Record<string, unknown>;
  actions?: AiWindowAction[];
  isPinned: boolean;
  isDirty: boolean;
  openedAt: string;
  updatedAt: string;
}
```

### 6.3 Поведение мультиоконности

Окна должны:

- жить в рамках одного чата;
- быть привязаны к сообщению, из которого родились;
- уметь сосуществовать параллельно;
- иметь стек или список активных окон;
- не теряться при переключении между окнами;
- позволять пользователю вернуться к любому открытому контексту.

Это создаёт ощущение “агент ведёт несколько рабочих потоков”, а не просто отвечает сообщениями.

## 7. `Contextual Overlay Canvas`

Это должен быть не просто overlay, а управляемый canvas правой рабочей зоны.

### 7.1 Обязательные свойства

- рабочая зона остаётся видимой;
- overlay не убивает фон полностью;
- человек продолжает узнавать, в каком контексте он находится;
- окно привязано к конкретному запросу из чата;
- окно имеет собственный заголовок, статус и действия;
- окно может быть:
  - свернуто;
  - переведено в компактный стек активных окон;
  - закреплено;
  - переведено в takeover;
  - закрыто без потери основного треда.

### 7.2 Рабочая логика окна

Оверлэй-окно агента не должно быть отдельным “чужим приложением”.

Оно должно вести себя как умный слой поверх текущего продуктового экрана:

- пользователь видит, с каким экраном он сейчас работает;
- агент подмешивает нужный контекст поверх него;
- если нужно, окно ведёт пользователя в нужный раздел интерфейса;
- после этого пользователь может вернуться к окну и продолжить задачу;
- несколько окон могут жить параллельно и обслуживать разные подтемы одного треда.

### 7.3 Чего не должно быть

- один и тот же шаблон для всех ответов;
- тяжёлый overlay на любой чих;
- ощущение “открылось отдельное чужое модальное окно”;
- потеря понимания, из какого запроса это вообще родилось.

### 7.4 Канонический заголовок окна

У каждого рабочего окна должна быть шапка:

- имя агента;
- тип окна;
- краткий статус;
- связь с исходным запросом;
- действия: закрепить, развернуть, закрыть.

Пример:

- `Агроном-А`
- `Добор контекста для техкарты`
- `Нужно поле и сезон`
- `Источник: “Составь черновик техкарты по озимому рапсу”`

## 8. Новый цикл добора контекста

Это критичный продуктовый пробел, который нужно закрыть первым.

### 8.1 Проблема текущего состояния

Сейчас агент уже умеет понять, что контекста не хватает.

Но дальше система ещё слишком сырая:

- в чат уходит слишком техническая формулировка;
- нет нормального guided flow;
- человек вынужден догадываться, что делать дальше.

### 8.2 Целевая схема

Когда агенту не хватает данных:

1. Агент возвращает структурированное описание нехватки.
2. Оркестратор не спамит чат деталями.
3. Оркестратор открывает справа контекстное окно добора.
4. В чат идёт только короткая понятная реплика.
5. Пользователь добирает контекст через:
   - выбор;
   - переход;
   - подтверждение;
   - ввод;
   - использование данных текущего экрана.
6. После добора контекста система автоматически продолжает исходную задачу.

### 8.3 Минимальный контракт нехватки контекста

```ts
interface MissingContextItem {
  key: string;
  label: string;
  reason: string;
  required: boolean;
  expectedSource: "workspace" | "user" | "record";
  suggestedActions?: string[];
}

interface ClarificationNeed {
  kind: "missing_context";
  agentRole: string;
  summary: string;
  items: MissingContextItem[];
  recommendedMode: "inline" | "panel" | "takeover";
}
```

### 8.4 Как это должно звучать в чате

Не так:

- `Не хватает контекста: fieldRef, seasonRef`

А так:

- `Чтобы подготовить техкарту, мне нужны поле и сезон. Я открыл справа панель добора контекста.`

### 8.5 Как это должно выглядеть справа

Окно:

- `Что нужно для продолжения`
- `Поле`
- `Сезон`
- `Что уже найдено`
- `Что можно взять из текущего экрана`
- `Открыть карточку поля`
- `Выбрать сезон`
- `Продолжить`

## 9. Новая композиция левого AI Dock

Твоя идея здесь правильная: левый блок надо делать более собранным.

### 9.1 Каноническая композиция

Левый AI Dock должен иметь 3 зоны:

1. `Header`
   - название агента/ассистента;
   - статус;
   - быстрые действия;
   - компактный блок `Signals`.

2. `History`
   - основной поток сообщений;
   - только смысловые реплики;
   - короткие системные переходы;
   - без тяжёлых дублирующих карточек.

3. `Composer`
   - поле ввода;
   - attach / context / mode;
   - отправка;
   - быстрые уточняющие действия.

### 9.2 Что делать с мини-инбоксом

Отдельную тяжёлую карточку под чатом лучше убрать.

Вместо этого:

- встроить `Signals` в верхнюю часть чата;
- показывать там не больше 2–3 строк;
- давать переход в полный список или в соответствующее окно.

Это вернёт рабочее пространство и уберёт ощущение раздвоенного интерфейса.

При этом сам dock должен работать в паре с оверлэй-окнами:

- чат держит основную нить разговора;
- справа пользователь продолжает работать с продуктом;
- поверх правой зоны агент открывает ровно те окна, которые нужны по контексту.

## 10. Предлагаемая frontend-модель

### 10.1 Что есть сейчас

Сейчас уже есть store для:

- сообщений;
- режима панели;
- открытия виджетов;
- ширины чата;
- выбранной цели сигнала.

### 10.2 Что нужно добавить

Нужно расширить store до модели рабочих окон.

Минимально:

```ts
interface PendingClarificationState {
  active: boolean;
  agentRole: string;
  originMessageId: string;
  originalGoal: string;
  missingKeys: string[];
  collected: Record<string, unknown>;
  windowId?: string;
}

interface AiInteractionStore {
  workWindows: AiWorkWindow[];
  activeWindowId: string | null;
  pendingClarification: PendingClarificationState | null;
}
```

Нужны новые продуктовые сущности:

- `AiSignalsStrip`
- `AiWorkCanvas`
- `AiWindowTabs`
- `AiInlineInsight`
- `AiWorkPanel`
- `AiTakeoverPanel`
- `AiContextAcquisitionWindow`

При этом старые кирпичи не нужно выкидывать сразу. Их можно использовать как базу.

## 11. Предлагаемая backend-модель

### 11.1 Что не надо делать

Не надо делать так, чтобы фронт сам угадывал, как интерпретировать любой агентный текст.

### 11.2 Что надо делать

Backend должен отдавать:

- короткий текст для чата;
- структурированные рабочие окна;
- структурированные действия;
- явный тип режима вывода;
- сигнал о pending clarification.

Минимально в ответе агента/оркестратора должны появиться:

```ts
interface RaiChatResponse {
  text: string;
  widgets?: RaiChatWidget[];
  workWindows?: AiWorkWindow[];
  pendingClarification?: ClarificationNeed | null;
  activeWindowId?: string | null;
}
```

### 11.3 Кто решает, какой режим открытия нужен

Не сам UI.

Режим открытия должен идти из orchestration layer:

- `inline` для коротких предупреждений;
- `panel` для основного рабочего вывода;
- `takeover` для тяжёлых сценариев.

UI может только адаптировать размеры под экран.

---

## 12. Механика связывания окна с диалогом

Это принципиально важно.

Каждое рабочее окно должно быть привязано:

- к исходному сообщению пользователя;
- к агенту, который его породил;
- к конкретной задаче;
- к текущему треду.

Пользователь должен видеть:

- откуда окно появилось;
- связано ли оно с текущим сообщением;
- что произойдёт после подтверждения или заполнения.

Это снимает ощущение “магического окна из ниоткуда”.

Дополнительно пользователь должен видеть, что окно существует не само по себе, а как рабочий слой поверх текущего экрана продукта.

---

## 13. Канонические типы окон для первого релиза

Не надо сразу строить 20 видов. Достаточно первых 5.

### 13.1 `context_acquisition`

Для добора недостающего контекста.

Первый обязательный кандидат:

- `Агроном-А -> техкарта -> не хватает поле/сезон`

### 13.2 `structured_result`

Для основного структурированного ответа агента.

### 13.3 `related_signals`

Для связанных рисков, отклонений, бэклога, алертов.

### 13.4 `comparison`

Для сравнений и сценарного анализа.

### 13.5 `action_master`

Для длинных guided-сценариев.

## 14. Пошаговый rollout

### Фаза 1. Упорядочить текущий UX

- встроить `Signals` в header чата;
- убрать тяжёлую отдельную карточку мини-инбокса;
- оставить чат как 3-зонный dock;
- сохранить текущий overlay как временный `Work panel`.

### Фаза 2. Ввести режимы окон

- `inline`
- `panel`
- `takeover`

Статус: выполнено для первого slice.

### Фаза 3. Ввести `workWindows[]`

- хранение нескольких активных окон;
- переключение между окнами;
- pin/unpin;
- close;
- restore.

Статус: выполнено.

Что уже есть:

- `workWindows[]`
- `activeWindowId`
- стек свёрнутых окон
- restore
- close с graph-semantics:
  - закрытие основного окна закрывает и дочерние hint-окна
- выбор активного окна по `priority/category/mode`

Что уже закрыто дополнительно:

- `pin/unpin`
- общий менеджер для `context_*`, `structured_result`, `related_signals`, `comparison`

### Фаза 4. Реализовать clarification loop

Первый канонический сценарий:

- пользователь просит техкарту;
- не хватает поля и сезона;
- агент открывает `context_acquisition` окно;
- пользователь добирает контекст;
- выполнение продолжается автоматически.

Статус: выполнено для живых governed сценариев:

- `Агроном-А -> tech_map_draft`
- `Экономист-А -> compute_plan_fact`

### Фаза 5. Перевести существующие widgets в окна

Старые structured widgets не выкидывать.

Нужно:

- завернуть их в новую оконную модель;
- использовать их как payload рабочих окон.

Статус: выполнено.

### Фаза 6. Включить мультиоконность

- несколько окон на один тред;
- разные окна от разных агентов;
- быстрый возврат в открытый рабочий контекст.

Статус: выполнено.

Что уже есть:

- несколько окон в одном thread-response;
- родительско-дочерние связи через `parentWindowId` и `relatedWindowIds`;
- cross-window action `focus_window`;
- сворачивание в стек и восстановление;
- result-hint и clarification-hint как дочерние окна.

Что уже закрыто дополнительно:

- `comparison` как first-class analytical window;
- мультиоконная graph-semantics для clarification/result/hint flows;
- единый window manager для канонических product flows Stage 2.

## 15. Что не делать

- не строить всё как набор модалок;
- не превращать чат в форму;
- не отдавать выбор режима только фронту;
- не плодить дублирующие карточки одного и того же смысла;
- не начинать с универсального “супер-окна для всего”;
- не пытаться сразу переписать весь AI Dock и весь overlay-контур за один спринт.

---

## 16. Первый эталонный сценарий

Нужно выбрать один сценарий и довести его до идеала.

Лучший кандидат:

### `Агроном-А -> техкарта -> не хватает контекста`

Потому что здесь сразу видны все ключевые требования:

- выбор агента;
- обязательный контекст;
- уточнение без засорения чата;
- привязка к рабочему экрану;
- переход пользователя в карточку/поле/сезон;
- автоматическое продолжение выполнения после добора.

Если этот сценарий станет удобным, дальше паттерн можно размножать на:

- `Экономист-А`
- `Знание-А`
- future roles
- action masters
- multi-step panels

---

## 17. Критерии готовности

Blueprint можно считать реализованным не тогда, когда “красиво стало”, а когда одновременно выполняется следующее:

- чат остаётся читаемым и не перегруженным;
- правая зона продолжает быть основной рабочей поверхностью продукта;
- overlay не выглядит как тяжёлая случайная модалка;
- пользователь понимает, что делать дальше;
- агент может открыть больше одного рабочего окна;
- рабочие окна можно свернуть и вернуть без потери контекста;
- недостающий контекст добирается через guided flow;
- после добора система продолжает исходную задачу без ручного перезапуска;
- всё это не ломает текущий persistent shell и `workspaceContext`.

---

## 18. Сухой вывод

Правильная целевая модель для `RAI_EP`:

- слева: постоянный AI Dock как нить разговора;
- справа: основная рабочая поверхность продукта;
- поверх неё: `Contextual Overlay Canvas` как слой агентных окон;
- в этом слое: режимы `inline / panel / takeover`;
- над этим: мультиоконность и сворачиваемые окна;
- внутри orchestration: managed clarification loop.

Именно это даст ощущение не “чата с карточками”, а зрелой агентной операционной среды.
