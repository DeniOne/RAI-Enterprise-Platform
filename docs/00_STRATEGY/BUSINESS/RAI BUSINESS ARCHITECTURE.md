# Версия 1.0 — Целостная модель бизнеса и системы

## ЧАСТЬ I — СТРАТЕГИЧЕСКАЯ МОДЕЛЬ

### 1. Сущность бизнеса
RAI — это результатный агроконсалтинг с технологическим ядром, где:

- Продукт — достижение измеримого урожая.
- Монетизация — через прирост результата (Δ).
- Система (`RAI_EP`) — внутренний производственный инструмент.
- Дополнительная маржа — через поставки, комиссии и интеграцию финансовых сервисов.

### 2. Формула бизнеса
`Контекст хозяйства`
→ `План Урожая`
→ `Эталон (Техкарта)`
→ `Контроль исполнения`
→ `Управление отклонениями`
→ `Фактический результат`
→ `Δ (прирост)`
→ `Монетизация`
→ `Масштабирование знаний`

### 3. Структура холдинга
#### 3.1 РАИ Технологии
- Владение IP.
- Разработка `RAI_EP`.
- AI-алгоритмы.
- Память и аналитика.
- Лицензирование внутри холдинга.

#### 3.2 РАИ Консалтинг
- Заключение договоров.
- Управление клиентами.
- Ответственность за результат.
- Получение performance-дохода.

#### 3.3 Торговая компания "РАИ Торг"
- Поставка СЗР.
- Маржинальная наценка.
- Логистика.

#### 3.4 Производство (Грипил) "Про-Грипил"
- Контрактное производство.
- Контроль качества.
- Оптовая реализация.

### 4. Источники дохода
Основные:

- Фиксированный входной аудит.
- Performance-вознаграждение (`Δ × Доля`).

Дополнительные:

- Маржа на СЗР.
- Комиссии банков.
- Комиссии страховых.
- Продажа Грипил.

### 5. Два контура управления
#### Контур 1 — Back-Office (Управление компанией)
Функция: управление портфелем клиентов, экономикой и рисками.

Состав:

- CRM (контекст хозяйств).
- Экономика (`Δ`, маржа, ROI).
- Финансы (бюджет, трансфертные платежи).
- HR (найм, KPI, развитие).
- GR (субсидии, отчётность).
- Supply (склад, закупки).

#### Контур 2 — Front-Office (Работа с хозяйствами)
Функция: управление производственным циклом урожая.

Состав:

- План Урожая.
- Техкарта.
- Task Engine.
- Фото/гео-подтверждение.
- Алерты.
- Техсовет.
- Фиксация результата.

### 6. Ролевая модель
#### Менеджер РАИ
Зона ответственности:

- Контекст хозяйства.
- Сбор данных.
- Контроль исполнения.
- Коммуникация с клиентом.
- Не принимает агрономических решений.

#### Агроном-консультант РАИ
Зона ответственности:

- План Урожая.
- Разработка Техкарты.
- Анализ отклонений.
- Участие в техсовете.

#### Руководство РАИ
Зона ответственности:

- Экономика портфеля.
- Риск-контроль.
- Стратегия.
- Разрешение конфликтов.

#### ИИ (`RAI_EP`)
Функция:

- Симуляции.
- Проверка конфликтов.
- Алерты.
- Экономические прогнозы.
- Контроль инвариантов.

Ограничение: ИИ не утверждает решения, только предлагает.

### 7. Линии взаимодействия
#### Менеджер РАИ↔ Хозяйство
Канал:

- Telegram.
- Web-интерфейс.
- Телефон.
- Выезд.

#### Агроном РАИ ↔ Менеджер РАИ
Канал:

- Система.
- Техсовет.
- Внутренний чат.

#### Руководство РАИ ↔ Система РАИ
Канал:

- Управленческий дашборд.
- Риск-карта.
- Портфельная аналитика.

#### Back-Office РАИ ↔ Front-Office РАИ
Back-Office РАИ:

- Задаёт экономические рамки.
- Контролирует маржу.
- Управляет портфелем.

Front-Office РАИ:

- Генерирует результат.
- Создаёт экономический эффект.
- Поставляет фактические данные.

### 8. Поток денег
- Клиент → РАИ Консалтинг (фикс + % `Δ`).
- Клиент → Торговая компания РАИ (СЗР).
- Банк → Комиссия.
- Страховая → Комиссия.
- Консалтинг РАИ → РАИ Технологии (роялти).

### 9. Стратегический фокус
RAI — это:

- Не SaaS (возможно в будущем, отдельные модули системы).
- Не IT-компания (хотя есть IT подразделение).
- Не просто консалтинг.
- Интеллектуальный агро-оператор, управляющий урожаем как проектом.

## ЧАСТЬ II — ТЕХНИЧЕСКИЙ BLUEPRINT

### 1. Архитектурный принцип
`RAI_EP` = Операционная система достижения состояния.

Не контроль задач.
Контроль достижения производственных состояний.

### 2. Слои системы
#### Layer 1 — Business Core
- Identity & Access.
- Registry (Company, Field, Season).
- Task Engine.
- Economy Engine.
- Audit & Explainability.
- Governance Layer.

#### Layer 2 — RAI Domain
- Plan Yield (План Урожая).
- TechCard Engine.
- Risk Triage.
- Alert System.
- Economic Simulation.
- Engram Memory.
- Satellite Monitoring.

#### Layer 3 — Interaction Layer
- Telegram Bot.
- Сбор данных.
- Фото/гео.
- Быстрые действия.
- Web Interface.
- CRM.
- Планирование.
- Дашборды.
- Техсоветы.

### 3. Основные доменные сущности
#### 3.1 CRM Context
- Company.
- Field.
- Region.
- Resources.
- Risk Profile.

#### 3.2 Plan Yield
- Culture.
- Season.
- Base Yield.
- Min / Optimum / Max.
- Economic Limits.
- Risk Tolerance.

#### 3.3 TechCard
- Operations.
- Dependencies.
- Dosages.
- Control Points.
- Risk Flags.

#### 3.4 Execution
- Tasks.
- Events.
- GeoEvidence.
- PhotoEvidence.
- State Transitions.

#### 3.5 Result
- Actual Yield.
- Market Price.
- Cost.
- `Δ`.
- Performance Fee.

### 4. Контур данных
`CRM → PlanYield → TechCard → Tasks → Events → Result → Economy → Memory`

### 5. Контур контроля
`Execution Event`
→ `Risk Evaluation`
→ `Alert`
→ `Escalation`
→ `Tech Council`
→ `Decision`
→ `State Update`

### 6. Финансовый слой
Система должна учитывать:

- Доход по клиенту.
- Себестоимость сопровождения.
- Доход по поставкам.
- Комиссионные.
- Трансфертные платежи.
- Маржинальность по портфелю.

### 7. Управленческий слой
Руководство видит:

- EBITDA портфеля.
- Риск-профиль по хозяйствам.
- `Δ` по культурам.
- Сравнение сезонов.
- HR-показатели.

### 8. Критические инварианты
- Техкарта не существует без Плана Урожая.
- План Урожая не существует без CRM-контекста.
- `Δ` рассчитывается только после фиксации фактического урожая.
- ИИ не имеет права финального утверждения.
- Каждое экономическое решение должно быть объяснимо (Explainability).

### 9. Главный вывод
`RAI_EP` — это:

- Производственный инструмент холдинга.
- Система управления состояниями.
- Интеллектуальный контроль исполнения.
- Экономический калькулятор результата.
