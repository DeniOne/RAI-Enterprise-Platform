# FRONT_OFFICE_MASTER_PLAN

> Главный активный канон блока `Front-Office` в `RAI_EP`.

Этот документ собирает в одно целое:

- границы блока;
- business outcome;
- роли и каналы;
- информационную архитектуру;
- ключевые user flows;
- контрактные требования;
- roadmap реализации.

Все остальные документы блока считаются активными приложениями к этому master-plan, а не параллельными источниками истины.

---

## 1. Что такое Front-Office в RAI_EP

`Front-Office` это исполнительный интерфейс хозяйства для:

- фиксации факта;
- исполнения техкарты;
- эскалации отклонений;
- получения консультаций;
- пополнения контекста по хозяйству и урожаям.

`Front-Office` не является автономной системой принятия решений.  
Это операционный и сенсорный контур `Back-Office`, связанный с ним через события, evidence и управленческие реакции.

---

## 2. Зачем нужен этот блок

Для хозяйства `Front-Office` должен давать:

- дисциплину исполнения техкарты;
- быстрый сбор факта с поля;
- снижение приписок;
- ускорение реакции на отклонения;
- повышение урожайности через соблюдение технологии;
- улучшение экономики хозяйства и каждого урожая.

Для `RAI_EP` `Front-Office` должен давать:

- оперативность сопровождения;
- точность консультаций;
- прозрачность сопровождения;
- доказуемость работы;
- снижение потерь на коммуникации;
- масштабирование одного менеджера и агронома на большее число хозяйств.

---

## 3. Что Front-Office не должен делать

В `Front-Office` не входят:

- CRM;
- договоры;
- финансы;
- техсовет;
- финальное утверждение решений;
- аналитика руководства.

Также `Front-Office` не должен:

- менять план;
- закрывать отклонения окончательно;
- утверждать изменения технологии.

Все такие действия должны уходить в контуры, внешние по отношению к `Front-Office`.

---

## 4. Пользователи и каналы

### 4.1 Основные пользователи

- агроном хозяйства;
- менеджер хозяйства;
- директор хозяйства.

`Front-Office` проектируется не под одного человека, а под рабочую связку ролей внутри хозяйства.

### 4.2 Каналы

- `Telegram` — основной операционный интерфейс;
- `Web UI` — дополнительный интерфейс обзора, контроля и навигации.

### 4.3 Роль агента Front-Office

Агент `Front-Office` выступает посредником между:

- свободной коммуникацией пользователей;
- структурированными объектами системы;
- командой `RAI_EP`.

Он не принимает финальные управленческие решения, а:

- классифицирует входящие сообщения;
- выделяет задачу, observation, deviation, consultation или context update;
- сохраняет след исходного сообщения;
- маршрутизирует операционный поток в каноническую модель.

---

## 5. Базовые принципы блока

### 5.1 Event-first

Каждое действие должно порождать явное событие.

### 5.2 Object-anchored

Каждое действие должно быть привязано к:

- полю;
- сезону;
- техкарте;
- задаче;
- отклонению;
- консультации;
- контексту.

### 5.3 Evidence-driven

Любой полевой факт должен сопровождаться evidence или маркировкой его недостаточности.

### 5.4 Telegram-first

Полевое исполнение не должно зависеть от тяжелого web-интерфейса.

### 5.5 Role-aware

Все роли живут в одном объектном мире, но видят разный операционный фокус.

### 5.6 No free-floating chat

Свободная переписка допустима как вход, но не допустима как финальная форма хранения и управления.

---

## 6. Каноническая модель объектов

Основная цепочка `Front-Office`:

`Хозяйство -> Поле -> Сезон -> Техкарта -> Задача -> Observation / Evidence / Deviation -> Consultation / Escalation`

### Обязательные сущности первого контура

- хозяйство;
- поле;
- сезон;
- техкарта;
- задача;
- observation;
- deviation;
- evidence;
- context update.

### Важные расширяющие сущности

- консультация;
- история событий.

---

## 9. Канонические user flows

`Front-Office` должен в обязательном порядке поддерживать следующие core flows:

- получение задачи;
- старт задачи;
- завершение задачи;
- фиксация observation;
- фиксация deviation;
- запрос консультации;
- согласование техкарты без изменения содержания;
- передача context update;
- свободная переписка через агента с переводом в структурированную сущность.

Именно эти flows составляют ядро контурa.

---

## 10. Evidence policy

Для `Front-Office` evidence не факультативен, а обязателен.

Поддерживаемые типы:

- фото;
- гео;
- время;
- голос;
- таблица;
- документ;
- подтверждение контакта.

### Минимальные требования

- старт задачи: фото, гео, время, голос/таблица/документ по ситуации;
- завершение задачи: фото, гео, время, голос/таблица/документ по ситуации;
- фиксация отклонения: фото, гео, время, голос/таблица/документ;
- запрос консультации: фото, гео, время, голос/таблица/документ, подтверждение контакта при необходимости.

Если evidence неполный:

- событие не теряется;
- система маркирует `weak evidence`;
- управленческий риск возрастает.

## 11. API и интеграционный принцип

Контрактная поверхность `Front-Office` делится на 3 слоя:

### 11.1 Read layer

Для экранов и карточек:

- overview;
- list;
- detail;
- history;
- related objects.

### 11.2 Action layer

Для user flows:

- task start / complete / cancel;
- report deviation;
- request consultation;
- approve tech map;
- context update.

### 11.3 Channel adapter layer

Для `Telegram` и агентного ввода:

- intake message;
- classify message;
- map to canonical action;
- persist message trace.

## 12. Что уже реально есть в коде

На дату `2026-03-09` реально подтверждены:

- `fields`;
- `tasks`;
- `tech-map`;
- `agro-orchestrator`;
- `field-observation`;
- route `/front-office`;
- `Telegram`/agent-related контур в кодовой базе.

Это означает:

- `Front-Office` не нулевая идея;
- ядро контракта и доменных сущностей уже существует;
- главный разрыв сейчас в продуктовой сборке, а не в полном отсутствии backend-основы.

---

## 13. Главные gap'ы

Ключевые незакрытые разрывы:

- web-контур `/front-office` пока placeholder;
- нет role-aware shell для хозяйства;
- есть рассинхроны между frontend и backend контрактами;
- `season` не оформлен как канонический REST read-model;
- `deviation`, `consultation`, `context update` не собраны в единую product surface;
- свободная переписка еще не доведена до канонической agent-mediated модели.

## 15. Definition of Done для блока

`Front-Office` можно считать собранным, когда:

- хозяйство реально может работать через `Telegram`;
- `Web UI` реально пригоден для контроля и навигации;
- core user flows работают end-to-end;
- все действия привязаны к объектам и evidence;
- отклонения и консультации перестают жить "в чате";
- агент `Front-Office` превращает свободную коммуникацию в структурированную модель;
- нет битых маршрутов и ложных CTA.

## 17. Ключевой вывод

`Front-Office` в `RAI_EP` это не чат, не отдельный мини-продукт и не "ещё один фронт".  
Это операционный контур хозяйства, в котором:

- исполнение технологии становится управляемым;
- факты становятся доказуемыми;
- отклонения становятся объектами системы;
- консультации становятся контекстными;
- свободная коммуникация перестает быть шумом и превращается в структурированную operational truth.
