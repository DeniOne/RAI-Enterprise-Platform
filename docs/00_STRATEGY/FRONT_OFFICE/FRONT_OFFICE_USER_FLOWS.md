# FRONT_OFFICE_USER_FLOWS

## 1. Назначение документа

Этот документ фиксирует канонические пользовательские сценарии `Front-Office`.

Он нужен, чтобы:

- перевести `Scope` и `IA` в конкретные рабочие потоки;
- зафиксировать, какие действия реально допустимы в `Front-Office`;
- привязать действия к событиям, evidence и реакции `Back-Office`;
- подготовить основу для `API contracts`, UX-детализации и тестов.

Документ опирается на:

- [FRONT_OFFICE_SCOPE.md](./FRONT_OFFICE_SCOPE.md)
- [FRONT_OFFICE_IA.md](./FRONT_OFFICE_IA.md)
- `Front-Office Function Admission Rules`

---

## 2. Канонический принцип

Любой сценарий `Front-Office` должен замыкаться в цепочку:

`Trigger -> Event -> Evidence -> Backend Reaction -> State Impact`

Если сценарий не дает:

- явного события;
- объектной привязки;
- evidence или маркировки его отсутствия;
- управленческой реакции,

то такой сценарий не должен существовать как функция `Front-Office`.

---

## 3. Главные роли в сценариях

- агроном хозяйства;
- менеджер хозяйства;
- директор хозяйства;
- агент `Front-Office` как посредник и структуризатор входящих сообщений.

---

## 4. Каналы сценариев

### 4.1 Telegram-first сценарии

Основные операционные сценарии должны быть доступны через `Telegram`:

- получение задачи;
- старт задачи;
- завершение задачи;
- передача evidence;
- фиксация отклонения;
- запрос консультации;
- обновление контекста.

### 4.2 Web UI сценарии

Через `Web UI` должны выполняться сценарии:

- обзор статуса;
- просмотр списков и карточек;
- разбор истории;
- согласование техкарты в допустимых рамках;
- проверка completeness evidence;
- навигация по связанным сущностям.

---

## 5. Core User Flows

## 5.1 Flow A: Получение и принятие задачи

- **Роль:** агроном хозяйства / менеджер хозяйства
- **Канал:** Telegram, Web UI
- **Цель:** понять, какая задача назначена, в каком контексте и что требуется для исполнения

### Trigger

Пользователю назначена задача по техкарте.

### Steps

1. Пользователь получает уведомление о новой задаче.
2. Открывает карточку задачи.
3. Видит:
   - поле;
   - сезон;
   - техкарту;
   - срок;
   - инструкцию;
   - требования к подтверждению.
4. Подтверждает принятие задачи к исполнению или переходит к старту.

### Event

- `TaskViewed`
- `TaskAccepted` или переход к `TaskStarted`

### Backend Reaction

- фиксация просмотра и принятия;
- обновление статуса доступности задачи для контроля;
- запись в историю событий.

### State Impact

- ответственность пользователя становится явной;
- задача попадает в активный рабочий контур.

### Failure Mode

Если пользователь не открывает задачу или не подтверждает принятие:

- задача остается в списке ожидающих;
- система поднимает сигнал по дисциплине исполнения.

---

## 5.2 Flow B: Старт задачи

- **Роль:** агроном хозяйства / менеджер хозяйства
- **Канал:** Telegram-first, Web UI optional
- **Цель:** зафиксировать начало исполнения задачи в поле

### Trigger

Пользователь фактически приступает к операции по техкарте.

### Steps

1. Пользователь открывает задачу.
2. Нажимает `Старт`.
3. Прикладывает обязательные подтверждения:
   - фото;
   - гео;
   - время;
   - при необходимости голос / таблицу / документ.
4. Отправляет подтверждение старта.

### Event

- `TaskStarted`

### Backend Reaction

- модуль задач переводит задачу в `IN_PROGRESS`;
- evidence привязывается к задаче;
- событие отражается в истории;
- возможны проверки через `Integrity Gate`.

### State Impact

- задача официально запущена;
- появляется доказанный факт начала исполнения;
- обновляется операционный статус по полю и сезону.

### Failure Mode

Если evidence неполный:

- старт может быть принят как слабое доказательство;
- система маркирует риск;
- факт остается трассируемым.

---

## 5.3 Flow C: Завершение задачи

- **Роль:** агроном хозяйства / менеджер хозяйства
- **Канал:** Telegram-first, Web UI optional
- **Цель:** зафиксировать завершение операции

### Trigger

Работа по задаче фактически завершена.

### Steps

1. Пользователь открывает активную задачу.
2. Нажимает `Завершить`.
3. Прикладывает обязательные evidence:
   - фото;
   - гео;
   - время;
   - голос / таблицу / документ при необходимости.
4. Отправляет подтверждение завершения.

### Event

- `TaskCompleted`

### Backend Reaction

- модуль задач переводит задачу в `COMPLETED`;
- фиксируется доказанный факт исполнения;
- данные становятся доступными для последующего анализа и экономического учета;
- обновляется история сезона и поля.

### State Impact

- задача завершена;
- появляется подтвержденный факт выполнения;
- система обновляет картину исполнения техкарты.

### Failure Mode

Если evidence слабый или противоречивый:

- completion помечается как риск;
- возможна эскалация на проверку;
- задача не теряет след аудита.

---

## 5.4 Flow D: Фиксация наблюдения

- **Роль:** агроном хозяйства
- **Канал:** Telegram-first, Web UI secondary
- **Цель:** передать наблюдение по полю, культуре или операции

### Trigger

Пользователь замечает факт, важный для технологического или управленческого контекста.

### Steps

1. Пользователь выбирает поле / сезон / задачу.
2. Отправляет наблюдение:
   - текст;
   - фото;
   - голос;
   - документ / таблицу при необходимости.
3. Агент `Front-Office` помогает классифицировать сообщение.

### Event

- `ObservationRecorded`

### Backend Reaction

- создается observation;
- observation связывается с объектом;
- контекст хозяйства и сезона дополняется;
- при необходимости observation становится входом для risk/deviation logic.

### State Impact

- память системы обогащается новым фактом;
- повышается качество последующих консультаций и решений.

### Failure Mode

Если объектная привязка не определена:

- агент должен запросить уточнение;
- наблюдение не должно оставаться "сиротой".

---

## 5.5 Flow E: Фиксация отклонения

- **Роль:** агроном хозяйства / менеджер хозяйства
- **Канал:** Telegram-first, Web UI for review
- **Цель:** зафиксировать отклонение от техкарты или от допустимого хода сезона

### Trigger

Пользователь видит, что фактическая ситуация не соответствует техкарте, сроку, условию или ожиданию.

### Steps

1. Пользователь выбирает связанную задачу / поле / сезон.
2. Нажимает `Сообщить об отклонении`.
3. Описывает проблему.
4. Прикладывает evidence:
   - фото;
   - гео;
   - время;
   - голос / документ / таблицу.
5. Отправляет сообщение.

### Event

- `DeviationReported`

### Backend Reaction

- создается карточка отклонения;
- запускается причинная реакция `Back-Office`;
- отклонение попадает в контур review;
- инициируется риск и дальнейший разбор.

### State Impact

- проблема становится управляемым объектом;
- отклонение попадает в очередь дальнейшей обработки;
- начинается путь к техсовету и решению.

### Failure Mode

Если evidence неполный:

- отклонение все равно фиксируется;
- система маркирует его как слабое доказательство;
- повышается уровень риска и ответственности.

---

## 5.6 Flow F: Запрос консультации

- **Роль:** агроном хозяйства / менеджер хозяйства / директор хозяйства
- **Канал:** Telegram-first, Web UI as context view
- **Цель:** получить консультацию в контексте текущей ситуации

### Trigger

Пользователь сталкивается с вопросом по полю, задаче, отклонению или сезону.

### Steps

1. Пользователь инициирует `Запрос консультации`.
2. Выбирает или уточняет контекст:
   - поле;
   - задача;
   - сезон;
   - отклонение.
3. Отправляет вопрос и evidence:
   - фото;
   - гео;
   - время;
   - голос;
   - документ / таблицу;
   - подтверждение контакта при необходимости.
4. Агент `Front-Office` структурирует запрос.

### Event

- `ExpertConsultationRequested`

### Backend Reaction

- создается консультационный кейс;
- запрос связывается с предметом обсуждения;
- `RAI_EP` и связанная роль сопровождения получают полный контекст.

### State Impact

- вопрос становится управляемым объектом;
- консультация не теряется в чате;
- знания и история решения сохраняются.

### Failure Mode

Если пользователь описал вопрос свободным текстом без объекта:

- агент обязан запросить уточнение;
- система должна попытаться автоматически определить контекст по истории переписки.

---

## 5.7 Flow G: Согласование техкарты

- **Роль:** агроном хозяйства / директор хозяйства
- **Канал:** преимущественно Web UI, с уведомлением через Telegram
- **Цель:** подтвердить техкарту без изменения ее содержания

### Trigger

Техкарта подготовлена и готова к согласованию со стороны хозяйства.

### Steps

1. Пользователь открывает карточку техкарты.
2. Просматривает:
   - общую модель;
   - ключевые операции;
   - контрольные точки;
   - ограничения и допущения.
3. Подтверждает согласование.

### Event

- `TechMapApprovedByFarm`

### Backend Reaction

- статус согласования обновляется;
- история техкарты пополняется;
- карта становится ближе к рабочему исполнению.

### State Impact

- техкарта получает подтверждение со стороны хозяйства;
- возникает зафиксированная ответственность на исполнение.

### Failure Mode

Если пользователь не согласен из-за необходимости изменений:

- сценарий не должен замыкаться на простое `approve/reject`;
- должен запускаться внешний по отношению к `Front-Office` путь согласовательных изменений.

---

## 5.8 Flow H: Передача контекстного обновления

- **Роль:** агроном хозяйства / менеджер хозяйства / директор хозяйства
- **Канал:** Telegram-first, Web UI for review
- **Цель:** пополнить память системы новым значимым контекстом

### Trigger

Появилась новая информация о хозяйстве, поле, сезоне, ограничении, ресурсе или факте.

### Steps

1. Пользователь отправляет сообщение или документ.
2. Агент `Front-Office` определяет, что это не задача и не отклонение, а контекст.
3. Система уточняет объект привязки.
4. Контекст сохраняется в соответствующей сущности.

### Event

- `ContextUpdated`

### Backend Reaction

- обновляется карточка хозяйства / поля / сезона;
- контекст становится частью памяти системы;
- новые данные могут использоваться в дальнейших рекомендациях и сопровождении.

### State Impact

- `RAI_EP` работает на более полном контексте;
- снижается потеря знаний в переписке и документах.

### Failure Mode

Если контекст не классифицирован:

- агент переводит его в очередь уточнения;
- система не должна терять исходное сообщение.

---

## 5.9 Flow I: Свободная переписка с агентом Front-Office

- **Роль:** любая роль хозяйства
- **Канал:** Telegram
- **Цель:** дать пользователю естественный вход в систему без требования знать внутреннюю модель

### Trigger

Пользователь пишет свободное сообщение.

### Steps

1. Пользователь отправляет текст, голос, фото или документ.
2. Агент `Front-Office` анализирует сообщение.
3. Агент определяет, что это:
   - вопрос;
   - наблюдение;
   - отклонение;
   - задача;
   - контекст;
   - шум / неоперационный разговор.
4. Если нужно, агент задает уточняющий вопрос.
5. Итоговая сущность фиксируется в системе.

### Event

- зависит от результата классификации:
  - `ObservationRecorded`
  - `DeviationReported`
  - `ExpertConsultationRequested`
  - `ContextUpdated`
  - и т.д.

### Backend Reaction

- формируется структурированный объект;
- сохраняется трассируемая связь с исходным сообщением;
- управленческий контур получает валидный вход.

### State Impact

- свободное сообщение превращается в управляемую сущность;
- коммуникация не теряется и не живет вне модели.

### Failure Mode

Если сообщение невозможно классифицировать:

- агент обязан попросить уточнение;
- сырой input должен сохраниться как входной след.

## 6. Role-centric views of flows

### 6.1 Агроном хозяйства

Основные сценарии:

- получить задачу;
- стартовать задачу;
- завершить задачу;
- зафиксировать наблюдение;
- сообщить об отклонении;
- запросить консультацию;
- передать контекст.

### 6.2 Менеджер хозяйства

Основные сценарии:

- контролировать очередь задач;
- подтверждать фактическое выполнение;
- видеть проблемные места;
- фиксировать отклонения;
- поднимать консультации;
- удерживать дисциплину исполнения.

### 6.3 Директор хозяйства

Основные сценарии:

- видеть оперативную картину по хозяйству;
- согласовывать техкарту в допустимых рамках;
- видеть критические отклонения;
- подключаться к консультациям и эскалациям.

---

## 7. Что не является user flow Front-Office

Следующие сценарии не должны жить в `Front-Office`:

- финальное закрытие отклонения;
- изменение плана;
- техсовет;
- управление договорами;
- финансы;
- CRM;
- аналитика руководства.

Эти сценарии могут быть связаны с `Front-Office`, но исполняются за его границей.

---

## 8. Приоритет реализации сценариев

### Priority 1

- получение задачи;
- старт задачи;
- завершение задачи;
- фиксация отклонения;
- запрос консультации;
- передача контекста.

### Priority 2

- обзор сезонов;
- просмотр истории событий;
- просмотр evidence;
- согласование техкарты;
- карточки поля и сезона как рабочие хабы.

### Priority 3

- углубленная orchestration-навигация;
- агентная маршрутизация между ролями хозяйства;
- автоматическое извлечение структуры из свободной переписки.

---

## 9. Ключевой вывод

`Front-Office` должен поддерживать не десятки разрозненных действий, а ограниченный набор канонических потоков:

- исполнение;
- подтверждение;
- наблюдение;
- отклонение;
- консультация;
- пополнение контекста.

Именно эти потоки формируют доказуемую операционную работу хозяйства внутри `RAI_EP`.
