# ГРАНД-СИНТЕЗ: Технологическая Карта как Вычислительное Ядро FMS

> **Документ**: Технический синтез 6 независимых AI-исследований
> **Источники**: ChatGPT [Док1], ChatGPT#2 [Док2], CLUADE [Док3], COMET [Док4], GEMINI [Док5], GROK [Док6]
> **Цель**: Имплементация в систему RAI Enterprise Platform

## ЧАСТЬ 1. АРХИТЕКТУРНОЕ EXECUTIVE SUMMARY

### 1.1 Консенсус всех 6 источников: 7 фундаментальных аксиом

Все 6 исследований безусловно сходятся в следующих принципах:

| # | Аксиома | Суть | Источники |
|---|---------|------|-----------|
| 1 | **Техкарта ≠ таблица** | Техкарта — направленный ациклический граф (DAG) операций, а не плоский Excel-список. Узлы = операции, рёбра = зависимости (FS/SS/FF), веса = длительности и окна | [Док1–6] |
| 2 | **Урожай = гипотеза** | Y = f(G, E, M). Целевая урожайность — не факт, а тестируемая гипотеза с assumptions и constraints. Техкарта — формализация вектора M | [Док1–6] |
| 3 | **Двухконтурность** | Contract Core (подписываемый, иммутабельный, hash) + Execution Layer (изменяемый через ChangeOrder). Баланс юр.жёсткости и агрономической гибкости | [Док1–6] |
| 4 | **Provenance + Confidence** | Каждое критическое поле данных несёт мета-объект: откуда данные (source_type) + уровень доверия (0.0–1.0). Без этого — «мусор на входе» | [Док1–6] |
| 5 | **Триаж входных данных** | 3 слоя: Обязательные (блокируют создание) → Желательные (снижают confidence при отсутствии) → Вычисляемые/импутируемые (система генерирует сама) | [Док1–6] |
| 6 | **BBCH как универсальный язык** | Фенология кодируется исключительно через шкалу BBCH (0–99). Окна операций привязаны к фазам, а не к календарным датам | [Док1–6] |
| 7 | **Evidence-first исполнение** | Операция не может получить статус DONE без цифровых доказательств: GPS-треки, фото с геометками, акты списания, лаб.протоколы | [Док1–6] |

### 1.2 Жизненный цикл техкарты (состояния)

Консенсус по enum `TechMapStatus` [Док1,3,5]:

```
DRAFT → UNDER_REVIEW → APPROVED → IN_EXECUTION → CLOSED
                ↑                        |
                └── ChangeOrder ─────────┘
```

Переход `APPROVED → IN_EXECUTION` требует КЭП двух сторон [Док4,6]. Любой ChangeOrder, затрагивающий Contract Core, откатывает статус в `UNDER_REVIEW` [Док3,5].

## ЧАСТЬ 2. ЯДРО ДАННЫХ И МОДЕЛЬ ДАННЫХ (Блоки A + B)

### 2.1 Граф сущностей — синтезированная ER-модель

Все 6 источников описывают одну и ту же иерархию с минимальными расхождениями:

```
LegalEntity (1) ──→ (N) Farm (1) ──→ (N) Field
                                           │
                          Season ────┐     │
                          CropPlan ──┼─→ CropZone (= Field × Season × CropPlan)
                                     │         │
                                     │    TechMap (1:1 с CropZone)
                                     │         │
                                     │    ┌────┴────────────────┐
                                     │    │                     │
                                Operation (1:N)          BudgetLine (1:N)
                                     │                         │
                          ┌──────────┼──────────┐              │
                     Application  Machinery   Labor        KPI (1:N)
                     (Input ref)  Assignment              
                          │                               
                    Observation ──→ Evidence               
                          │                               
                    Risk ──→ AdaptiveRule                  
                          │                               
                    ChangeOrder ──→ Approval               
```

**Ключевая связь**: TechMap привязывается к CropZone (Field + Season + CropPlan), а НЕ напрямую к Field [Док1,3,5]. Это критически важно — одно поле может иметь разные техкарты в разные сезоны.

**Шкала Confidence** [Док3,5]:

- `0.7–1.0`: подтверждено документами, лабораторными измерениями, телеметрией
- Пример [Док5]: глубина посева с CAN-шины сеялки = 0.98, визуальная оценка практиканта = 0.40

## ЧАСТЬ 3. МЕТОДОЛОГИЯ РАСЧЁТА И АЛГОРИТМЫ (Блок C)

### 3.1 Гипотеза урожая — формализация

Формула [Док5]: **Y = f(G, E, M)** где:
- **G** (Genetics) = потенциал сорта/гибрида (фиксирован после выбора)
- **E** (Environment) = стохастический вектор (погода, почва) — не контролируется
- **M** (Management) = вектор операций техкарты — оптимизируется системой

Гипотеза включает [Док3,5]:
1. `target_yield_t_ha` — точка в диапазоне P10–P90 исторических данных поля
2. `target_quality` — масличность, влажность, эруковая кислота, глюкозинолаты
3. `assumptions[]` — «осадки в пределах нормы», «перезимовка с потерями ≤15%», «САТ ≥ X°C»
4. `constraints[]` — бюджетный лимит, экологические запреты, ресурсные ограничения

**Backward planning** [Док3]: от цели (урожайность) → назад к требуемой густоте → площади листьев → обеспеченности элементами → набору операций и норм.

### 3.2 Вычисляемые элементы — ключевые алгоритмы

#### 3.2.1 Расчёт нормы высева
```
НормаВесовая (кг/га) = (ЦелеваяГустота_млн/га × МассаТысячиСемян_г) / (ЛабВсхожесть% × ПолеваяВсхожесть% / 10000)
```
Источники: [Док1,3,5]. Диапазоны: 1.2–5.0 млн всхожих/га в зависимости от типа рапса и региона [Док6].
- Озимый (морской климат): норма снижается — длинная осень компенсирует ветвлением [Док5]
- Яровой (Сибирь): норма повышается для компенсации низкой полевой всхожести [Док5]

#### 3.2.2 Расчёт окон операций
```
ДатаОперации = f(РегиональныйПрофиль.GDD, ФактическаяПогода, МодельBBCH_гибрида)
```
НЕ календарные даты, а:
1. Базовые окна из `RegionProfile` [Док3,5,6]
2. Уточнение по сумме температур (GDD/САТ) и фенофазам BBCH [Док1–6]
3. Корректировка по доступности техники и людей [Док3]
4. Погодные ограничения: влажность почвы (%ППВ), скорость ветра, температура [Док5]

#### 3.2.3 Расчёт доз удобрений
```
ДозаN (кг/га) = (ВыносN × ЦелеваяУрожайность - ЗапасNпочвы × Кусвоения) / КусвоенияУдобрения
```
Входные данные: SoilProfile (N, P, K, S, B, pH), target_yield, региональные коэффициенты выноса [Док1,3,4].

#### 3.2.4 Экономический порог вредоносности (ЭПВ)
Динамический расчёт [Док3,5]:
```
ЭПВ = f(ФазаBBCH, Температура, ВидВредителя, СтоимостьОбработки, ПрогнозЦеныУрожая)
```
Примеры порогов [Док5]:
- Крестоцветная блошка: >1–3 жука/растение в фазе BBCH 10–13 → инсектицид
- Рапсовый цветоед: >3 шт/соцветие в фазе BBCH 51–59 → обработка

### 3.3 Двухконтурная архитектура

**Контур 1 — Базовый план**: Детерминированный набор Operation с нормами и окнами.

**Контур 2 — Адаптивные правила** (`AdaptiveRule`):
```json
{
  "id": "uuid",
  "trigger_type": "enum<WEATHER|NDVI|OBSERVATION|PRICE|PHENOLOGY>",
  "condition": "object",       // логическое выражение над данными
  "affected_operations": "array<operation_id>",
  "change_template": "object", // сдвиг окна, изменение нормы, добавление/отмена операции
  "requires_approval_role": "enum<ApproverRole>|null"
}
```

Примеры триггеров [Док3,5]:
- **WEATHER**: осадки >X мм за 10 дней → перенос внесения N на ≤3 дня
- **NDVI**: NDVI <0.6 в BBCH 30–39 → доп.осмотр или корректировка питания
- **PHENOLOGY** [Док5]: переход в BBCH 60 → открытие окна фунгицидной обработки, BBCH 69 → жёсткое закрытие
- **OBSERVATION**: >N личинок/м² → автогенерация ChangeOrder на инсектицид

### 3.4 Валидационные правила — что считается ошибкой

Консенсус [Док3,5,6]:

| Класс ошибки | Описание | Тип |
|--------------|----------|-----|
| Несовместимые смеси | В одном `tank_mix_group_id` — несовместимые Inputs | HARD STOP |
| Превышение норм | `planned_rate > max_rate` из регистрации/этикетки | HARD STOP |
| Нарушение фазового окна | Операция вне допустимых BBCH (напр. гербицид после BBCH 51) | HARD STOP |
| Пропуск критической операции | `is_critical=true` без факта до конца окна | CRITICAL WARNING |
| Регуляторные ограничения | СЗР в водоохранной/буферной зоне | HARD STOP |
| Ресурсный конфликт | Потребность в технике/людях > доступных мощностей | WARNING |
| Сингулярность норм | Превышение зарегистрированной дозы д.в./га | HARD STOP [Док5] |

**Hard Stop** [Док5] = алгоритмическая блокировка, операция не может быть запланирована/выполнена.

## ЧАСТЬ 4. ЮРИДИЧЕСКАЯ И ОПЕРАЦИОННАЯ МОДЕЛЬ (Блоки D + E)

### 4.1 Двухслойная контрактная архитектура

Абсолютный консенсус [Док1–6] — техкарта архитектурно разделена на два слоя:

**Contract Core (подписываемый базис)**:
- Идентификация сторон (юрлицо, консультант/RAI, подрядчики)
- Площадь обрабатываемых контуров (geometry)
- Целевая урожайность и качество
- Бюджетный лимит (Budget Cap, руб/га)
- Спецификация закупаемых ТМЦ (фиксация номенклатуры)
- Критические операции (обязательные к исполнению)
- Правила ответственности и распределения рисков
- Порядок ChangeOrder governance
- **Фиксируется**: КЭП двух сторон + SHA-256 hash [Док3,5]
- **Не изменяется** без переподписания

**Execution Layer (операционная часть)**:
- Полный список операций с календарями
- Конкретные нормы внесения (внутри подписанных диапазонов)
- Выбор конкретных продуктов (в рамках допустимых д.в.)
- Адаптивные правила и триггеры
- Технические детали по технике и людям
- **Изменяется**: через протокол ChangeOrder без переподписания (если не нарушает Contract Core)

### 4.2 Протокол ChangeOrder (ЗНИ)

Процесс [Док3,5]:

```
Триггер (Observation/Weather/KPI/AI)
    │
    ▼
Формирование ChangeOrder (автоматически)
    │─── change_type: SHIFT_DATE|CHANGE_INPUT|CHANGE_RATE|CANCEL_OP|ADD_OP
    │─── diff_payload: машиночитаемый diff
    │─── Delta Cost: прогноз финансового влияния
    │
    ▼
Маршрут согласования (по ролям)
    │─── Если Delta ≤ contingency_fund → Агроном утверждает
    │─── Если Delta > contingency_fund → Финансовый директор [Док5]
    │
    ▼
Применение → Новая версия TechMap (version++)
    │─── Аудитный след: кто/когда/почему/на основании чего
```

**Классификация ChangeOrder по масштабу** [Док5]:
- **Микро-корректировка** (внутри Execution Layer): сдвиг даты на ±3 дня, замена тягового класса → Агроном
- **Существенное изменение**: смена продукта СЗР, увеличение нормы >5% → Агроном + Директор
- **Критическое**: превышение Budget Cap, отмена критической операции → Переподписание Contract Core

### 4.3 Требования к Evidence (доказательства исполнения)

Для операций с `is_critical=true` [Док3,5,6]:

| Тип Evidence | Описание | Привязка |
|-------------|----------|----------|
| PHOTO + GEO_TRACK | Фото с координатами и временем в пределах контура поля | Operation |
| GEO_TRACK | GPS-треки агрегатов, связанные с operation_id | Operation |
| INVOICE | Накладные, подтверждающие закупку и списание Inputs | Input/Application |
| LAB_REPORT | Протоколы анализов: почва, растение, урожай (качество, остатки СЗР) | Observation |
| WEATHER_API | Слепки состояния погоды во время критических операций | Operation |
| CONTRACT | Электронные акты выполненных работ | Operation |

**Definition of Evidence** [Док5]: Операция НЕ МОЖЕТ получить статус DONE без:
1. Телеметрия подтверждает покрытие ≥98% площади контура
2. Складская система регистрирует акты списания материалов
3. Агроном прикрепляет фото пост-чека

**Гарантия целостности**: `Evidence.checksum` — контрольная сумма файла [Док3].

### 4.4 Развёртывание в операционный календарь

Техкарта → DAG → Сетевой график [Док3,5]:

```json
// ScheduleEntry — единица операционного календаря
{
  "date": "date",
  "operation_id": "string FK",
  "field_id": "string FK",
  "priority": "enum<HIGH|MEDIUM|LOW>",
  "planned_start": "datetime",
  "planned_end": "datetime",
  "resource_allocation": {
    "machinery_ids": "array<string>",
    "labor_ids": "array<string>"
  },
  "sla_hours": "number",              // целевой срок
  "time_tolerance_hours": "number"    // допуск по смещению
}
```

Алгоритм развёртывания [Док3]:
1. Построить DAG из Operation + dependencies
2. Рассчитать критический путь и буферы
3. Для каждого дня/недели — список задач с приоритизацией (по is_critical и близости закрытия окон)
4. Обнаружить ресурсные конфликты (техника/люди в один интервал)

### 4.5 Инженерия операции: Pre-checks → Execution → Post-checks

Каждая Operation содержит `OperationExecutionProtocol` [Док3,5]:

**Pre-checks (до выезда)**:
- Погодные условия в допустимом коридоре
- Техника исправна (ТО сделано)
- Подтверждённые остатки материалов на складе
- Нет несовместимых смесей в планируемом баке
- Калибровка оборудования (давление в шинах, высевающий аппарат) [Док5]

**Execution (контроль в процессе)**:
- Телеметрия: скорость, норма, покрытие площади
- Допуски: ±5% по норме высева, ±3 дня по дате [Док5,6]
- Периодические фото/видео

**Post-checks (после)**:
- Сверка фактических норм (телеметрия vs план)
- Фиксация выполненной площади и времени
- Физический контроль (раскопка рядков, оценка глубины) [Док5]
- Генерация Evidence
- Закрытие операции → статус DONE

### 4.6 Протокол «ИИ → Человек» — матрица делегирования

Синтез [Док3,5,6]:

| Уровень | Кто решает | Что |
|---------|-----------|-----|
| **Автономно (ИИ)** | Алгоритм | Расчёт окон, критического пути, базовых норм; маршрутизация техники; пересчёт GDD/BBCH; проверка совместимости смесей; генерация уведомлений; детекция валидационных ошибок |
| **С подтверждением агронома** | Агроном | Перенос сроков посева; корректировка нормы высева; старт инсектицидной обработки при ЭПВ; смена продуктов; любые изменения в критических операциях |
| **Эскалация на директора** | Директор/фин.контролёр | Любые ChangeOrder с увеличением себестоимости; превышение Budget Cap; отмена критических операций; нестандартные риски (форс-мажор, санкции) |
| **Полностью ручное** | Юрист + руководство | Споры о причинах недостижения урожайности; юридические разногласия; регуляторные изменения |

  "gdd_base_temp_c": "number 0–10",

### 5.3 Финансовая структура техкарты

#### 5.3.1 Структура бюджета

Сущность `BudgetLine` с категориями [Док3,5,6]:

| Категория (`BudgetCategory`) | Доля в бюджете [Док6] | Источник данных |
|-------------------------------|----------------------|-----------------|
| SEEDS | 15–20% | Склад ERP |
| FERTILIZERS | 25–30% | Склад ERP |
| PESTICIDES | 15% | Склад ERP |
| FUEL | 10% | Норматив л/га × цена ГСМ |
| LABOR | В составе 20% | Тарифная сетка × сложность операции |
| RENT | В составе 20% | Договоры аренды паёв |
| LOGISTICS | 5–10% | Транспортные плечи |
| ANALYSES | В составе 5–10% | Услуги лабораторий |

#### 5.3.2 Экономические KPI

| KPI | Формула | Единица |
|-----|---------|---------|
| Себестоимость/га (C_ha) | Σ BudgetLine.actual_cost / area_ha | руб/га |
| Себестоимость/т (C_t) | C_ha / actual_yield_t_ha | руб/т |
| Валовый доход/га | actual_yield × market_price | руб/га |
| Маржа/га | Доход/га – C_ha | руб/га |
| Маржинальность | Маржа / Доход × 100 | % |
| Маржа с учётом рисков [Док5] | (Урожайность × Цена) – Затраты × (1 + P_потери) | руб/га |
| Отклонение план/факт | (actual – planned) / planned × 100 | % |

#### 5.3.3 Правила контроля перерасходов

Синтез [Док3,5,6]:

1. **Порог согласования**: если `actual > planned × (1 + tolerance_pct)` → автогенерация ChangeOrder
   - tolerance_pct задаётся в Contract Core по категориям [Док3]
   - [Док6]: >10% — автоматический ChangeOrder; >5% по семенам
2. **Резервный фонд** (Contingency fund): 5–10% от базового бюджета [Док5]
   - Delta ≤ фонд → Агроном утверждает самостоятельно
   - Delta > фонд → Блокировка до КЭП финансового директора
3. **Нулевой допуск** [Док5]: Любое незапланированное увеличение затрат (даже +5% нормы вылива) алгоритмически считается перерасходом и требует обоснования

**Recalculation Engine** [Док5]: При каждом срабатывании триггера пересчитываются C_ha, C_t и Risk-Adjusted Margin в реальном времени.

## ЧАСТЬ 6. КАРТА ПРОТИВОРЕЧИЙ И НЕОПРЕДЕЛЁННОСТЕЙ

### 6.1 Противоречия между источниками

| # | Конфликт | Позиция A | Позиция B | Архитектурный вердикт |
|---|---------|-----------|-----------|----------------------|
| 1 | **CropZone vs прямая привязка TechMap** | [Док3,5]: TechMap → CropZone (Field × Season × CropPlan) — отдельная сущность CropZone | [Док6]: TechMap напрямую содержит fieldId + season + crop — без CropZone | **Принять CropZone [Док3]**. Отдельная сущность обеспечивает: множественные планы на одно поле, повторное использование полей между сезонами, чёткую нормализацию БД. В коде [Док6]-вариант — это денормализованный shortcut, допустимый только в API-ответе |
| 2 | **Confidence: 0–1 vs 0–100** | [Док3,5]: confidence: float 0.0–1.0 | [Док6]: confidence: integer 0–100 (процент) | **Принять 0–1 [Док3]**. Стандарт ML/data-science. В UI отображать как процент (×100). Хранить как float |
| 3 | **Норма высева: диапазон** | [Док5,6]: 1.2–1.5 млн/га (яровой Сибирь) | [Док6]: 2–5 млн/га (общий диапазон) | **Конфликта нет** — разная гранулярность. [Док6] даёт общий диапазон по всем регионам, [Док5] специфицирует для конкретного профиля. Архитектурно: диапазон хранится в `RegionProfile`, а не в коде |
| 4 | **Количество операций в примере** | [Док3]: 10 операций | [Док6]: 8 операций | [Док5]: 10 операций | **Число операций не фиксировано** — это параметр конкретной техкарты. В мини-примере: 10–12 операций как разумный базис для рапса. Граф должен поддерживать 5–30+ узлов |
| 5 | **Статусы TechMap** | [Док3]: DRAFT→UNDER_REVIEW→APPROVED→IN_EXECUTION→CLOSED | [Док5]: APPROVED_FOR_EXECUTION (один статус вместо двух) | **Принять 5-статусную модель [Док3]**. Разделение APPROVED и IN_EXECUTION даёт юридическую чёткость момента начала исполнения (подписание КЭП) |
| 6 | **Уровни Evidence** | [Док6]: 4 уровня доказательств (фото→акт→лаб.анализ→ГЛОНАСС) | [Док3,5]: Плоский enum типов без иерархии | **Синтез**: Ввести `evidence_weight` (число 1–10) для каждого типа Evidence. Фото+геотег = 3, акт+подпись = 5, лаб.анализ = 8, трек ГЛОНАСС = 9. Позволит алгоритмически оценивать «силу доказательства» |
| 7 | **Структура блоков техкарты** | [Док6]: 8 блоков (включая «Уход и мониторинг» как отдельный) | [Док1–5]: 6 основных разделов (Уход интегрирован в «Граф операций») | **Принять интегрированный подход** [Док1–5]. Уход и мониторинг = тип операции (SCOUTING) внутри графа, а не отдельный раздел документа. Упрощает обработку |

## ЧАСТЬ 7. ИНЖЕНЕРНЫЕ СЛЕПЫЕ ЗОНЫ

Критические пробелы, обнаруженные во всех 6 исследованиях. Без их проработки система будет неполной:

| # | Слепая зона | Почему критично | Что допроектировать |
|---|------------|-----------------|---------------------|
| 1 | **Мульти-полевая оптимизация** | Все источники рассматривают техкарту одного поля изолированно. В реальности хозяйство из 50+ полей — конкуренция за технику, людей, время, склад. Техника не может быть одновременно на поле №12 и №37 | Модуль `FleetScheduler` — оптимизация распределения ресурсов по всем полям хозяйства. Алгоритм: минимизация простоев + соблюдение SLA всех техкарт |
| 2 | **Складская интеграция** | Упоминается «связь с ERP/складом», но нет конкретного протокола. Как система узнаёт, что на складе осталось 2 тонны NPK вместо 5? | API-контракт с 1С:Склад или аналогом: `GET /stock/{input_id}` → `{available_qty, reserved_qty, unit_price}`. Блокировка операции при `available < planned` |
| 3 | **Модели GDD/BBCH для конкретных гибридов** | Все источники ссылаются на «тепловые модели гибридов», но ни один не описывает, откуда эти модели берутся и как калибруются | Справочник `HybridPhenologyModel`: для каждого гибрида — таблица GDD→BBCH переходов. Источник: данные производителя + калибровка по историческим наблюдениям хозяйства |
| 4 | **Справочник совместимости баковых смесей** | Валидация «несовместимые смеси» — Hard Stop, но справочник не описан | База данных `TankMixCompatibility`: матрица пар Input×Input → {COMPATIBLE, CAUTION, INCOMPATIBLE} + условия (температура, pH раствора). Источник: этикетки + производители |
| 5 | **Офлайн-режим** | Поля в России часто без мобильной связи. Как механизатор работает без интернета? | Локальный кэш текущих заданий на устройстве + буфер Evidence для синхронизации при появлении связи. Конфликт-резолюция по timestamp |
| 6 | **Мультиарендность (multi-tenancy)** | Ни один источник не описывает изоляцию данных между юрлицами в SaaS-модели | Tenant isolation на уровне `legal_entity_id`. Row-level security в PostgreSQL. Отдельные S3-бакеты для Evidence |
| 7 | **Версионирование справочников** | Регистрации СЗР, нормы, ГОСТ — меняются ежегодно. Как обновлять Input.legal_restrictions без инвалидации старых техкарт? | Версионирование справочников с `effective_from`/`effective_to`. Техкарта ссылается на конкретную версию справочника |

| 10 | **Результат-ориентированная компенсация** | [Док2,4] упоминают модель оплаты по результату (Δ урожайности), но нет формализации | Сущность `CompensationModel`: `{base_yield, actual_yield, delta, bonus_per_ton, penalty_per_ton, evidence_requirements}`. Привязка к TechMap + Contract Core |
