Работаем по кнопке/экрану в формате “док + сразу реализация”.

Контекст:
- Цель: пройти кнопку/экран end-to-end, зафиксировать в docs и сразу реализовать в коде.
- Папка документации: `docs/10_FRONTEND_MENU_IMPLEMENTATION`.
- Всегда обновляй `00_MASTER_MENU_MAP.md` (статус/комментарий) после выполнения этапа.
- Файл общего техдолга: `99_TECH_DEBT_CHECKLIST.md` (туда добавляются только индексированные задачи).

Обязательный порядок действий:
1) Создай/обнови отдельный файл документа для конкретной кнопки/экрана.
2) Сначала опиши в документе целевое поведение (UX, переходы, состояния, API-связки, критерии готовности).
3) Сразу после описания реализуй это в коде.
4) Проверь работоспособность (lint/tests/ручной сценарий).
5) Зафиксируй итог и остаточный долг в документе.
6) Обнови статус производства в личном файле кнопки/экрана.
7) Добавь/обнови задачи в `99_TECH_DEBT_CHECKLIST.md` только с `TD-ID`.

Структура документа (обязательно):
0. Статус производства кнопки/экрана:
   - Stage (`DISCOVERY | IMPLEMENTING | MVP_READY | HARDENING | DONE`)
   - Готовность (`0-100%`)
   - Дата последнего обновления
   - Следующий milestone
1. Название кнопки/экрана и бизнес-роль
2. Целевые маршруты (основной + подмаршруты)
3. Поведение при нажатии (куда ведет, что видит пользователь)
4. UI/UX-сценарий:
   - loading
   - empty
   - error
   - permission
5. Кликабельность блоков и действия, пути переходов
6. Smart routing контракт (`entity`/`severity`, подсветка, авто-скролл)
7. API-связки (какие endpoint используются)
8. Критерий готовности MVP
9. Production-ready checklist (обязательно для каждого документа):
   - [ ] заменить демо-данные на реальные API-метрики
   - [ ] унифицировать/почистить кодировку текстов (без кракозябр)
   - [ ] добавить e2e-сценарий “клик -> переход -> подсветка сущности”
10. Технический долг (обязательно отдельным блоком):
   - что не доделано
   - почему не сделано сейчас
   - приоритет (High/Medium/Low)
   - следующий конкретный шаг
11. Ссылки на TD-ID в `99_TECH_DEBT_CHECKLIST.md`:
   - перечислить только ID релевантных долгов (например: `TD-FARMS-DATA-001`).

Правило статусов и техдолга (обязательно):
- Статус производства хранится только в личном файле кнопки/экрана.
- В `99_TECH_DEBT_CHECKLIST.md` хранятся только задачи техдолга с индексами `TD-...`, без процентов готовности кнопки.
- Каждый новый долг из личного файла кнопки должен иметь ссылку на соответствующий `TD-ID`.

Правила предложений (обязательно соблюдать):
- Не используй формулировки “если хочешь/могу”.
- При любом предложении развития сразу предлагай конкретное действие к выполнению.
- Для каждого действия обязательно указывай ожидаемый эффект:
  - зачем это делается
  - что улучшится для пользователя/продукта/кода

Формат финального отчета по задаче:
1. Что сделано (док + код)
2. Какие файлы изменены
3. Что проверено (lint/tests/ручной проход)
4. Что осталось (техдолг + production-ready пункты)
5. Следующее действие (конкретное) + ожидаемый эффект
