# Consulting UI Contracts: Screen Passports
RAI Enterprise Platform

Назначение: Зафиксировать контракт для каждого экрана, предотвращая дублирование и несогласованные действия.

## 1. Управление Урожаем — ЯДРО

### 1.1 Обзор (Dashboard)

*   **Экран:** Dashboard консультанта
*   **Назначение:** Обзор ключевых показателей по всем хозяйствам, фокусировка на проблемах.
*   **Кто пользователь:** Ведущий консультант, Агроном-куратор.
*   **Входные данные:** Список активных хозяйств, агрегированные статусы рисков (Risk Score), критические отклонения (Count), P/L статус (Forecast vs Baseline).
*   **Выходные данные:** Переход к конкретному отклонению или хозяйству.
*   **Ключевые действия:** Просмотр сводки, фильтрация по уровню риска, переход в детализацию проблемы.
*   **Связанные сущности:** Account, HarvestPlan, RiskEvent, Deviation.

### 1.2 Хозяйства и Контрагенты (CRM)

#### 1.2.1 Реестр хозяйств
*   **Экран:** Список хозяйств (Accounts List)
*   **Назначение:** Навигация по клиентской базе.
*   **Кто пользователь:** Консультант, Менеджер по продажам.
*   **Входные данные:** Список Accounts, фильтры (Регион, Статус, Тип).
*   **Выходные данные:** Выбор хозяйства для просмотра.
*   **Ключевые действия:** Поиск, фильтрация, создание нового хозяйства.
*   **Связанные сущности:** Account.

#### 1.2.2 Карточка хозяйства
*   **Экран:** Детали хозяйства (Account Details)
*   **Назначение:** Полный контекст по конкретному клиенту.
*   **Кто пользователь:** Консультант, Агроном.
*   **Входные данные:** Данные Account, список связанных Fields, история сезонов (Past Seasons), текущие Harvest Plans.
*   **Выходные данные:** Обновленные данные хозяйства.
*   **Ключевые действия:** Редактирование профиля, управление полями, просмотр истории.
*   **Связанные сущности:** Account, Field, HarvestPlan.

### 1.3 Планы Урожая

#### 1.3.1 Список планов урожая
*   **Экран:** Реестр планов (Harvest Plans List)
*   **Назначение:** Управление портфелем планируемых урожаев.
*   **Кто пользователь:** Консультант.
*   **Входные данные:** Список Harvest Plans, статусы (Draft, Active, Completed), прогресс.
*   **Выходные данные:** Выбранный план.
*   **Ключевые действия:** Создание плана, фильтрация по статусу/культуре.
*   **Связанные сущности:** HarvestPlan, Account, Crop.

#### 1.3.2 Конструктор плана урожая
*   **Экран:** Редактор плана (Harvest Plan Designer)
*   **Назначение:** Постановка и согласование цели.
*   **Кто пользователь:** Консультант (редактирование), Клиент (просмотр/согласование).
*   **Входные данные:** Исторические данные (Yield History), ограничения (Constraints), экономические параметры (Price, Costs).
*   **Выходные данные:** Созданный или обновленный HarvestPlan.
*   **Ключевые действия:** Ввод Min/Opt/Max целей, определение метрик успеха, выбор культуры и полей.
*   **Связанные сущности:** HarvestPlan, PerformanceContract.

### 1.4 Техкарты Урожая

#### 1.4.1 Список Техкарт
*   **Экран:** Реестр техкарт (TechMaps List)
*   **Назначение:** Доступ к эталонам исполнения.
*   **Кто пользователь:** Агроном, Консультант.
*   **Входные данные:** Список TechMaps, привязка к HarvestPlans.
*   **Выходные данные:** Выбранная техкарта.
*   **Ключевые действия:** Создание техкарты на основе плана, клонирование (с осторожностью).
*   **Связанные сущности:** TechMap, HarvestPlan.

#### 1.4.2 Workbench проектирования
*   **Экран:** Редактор техкарты (TechMap Editor)
*   **Назначение:** Детальное проектирование агрономического процесса.
*   **Кто пользователь:** Агроном-технолог.
*   **Входные данные:** Стадии роста (Phenology), библиотека операций, доступные ресурсы.
*   **Выходные данные:** Утвержденная TechMap.
*   **Ключевые действия:** Добавление операций (Operations), определение зависимостей, расстановка контрольных точек (Checkpoints).
*   **Связанные сущности:** TechMap, Operation, Resource.

### 1.5 Исполнение Техкарт (Агроном)

#### 1.5.1 Активные техкарты
*   **Экран:** Монитор исполнения (Execution Monitor)
*   **Назначение:** Контроль соответствия факта плану.
*   **Кто пользователь:** Агроном.
*   **Входные данные:** TechMap vs Actual Events, данные мониторинга (Satellite, IoT, Scouting).
*   **Выходные данные:** Выявленные отклонения.
*   **Ключевые действия:** Сравнение план/факт, подтверждение выполненных операций, инициация отклонения.
*   **Связанные сущности:** TechMap, ExecutionEvent.

### 1.6 Исполнение Техкарт (Менеджер Хозяйства)

#### 1.6.1 Задачи по техкарте
*   **Экран:** Список задач (Task List)
*   **Назначение:** Оперативное управление работами в поле.
*   **Кто пользователь:** Менеджер хозяйства.
*   **Входные данные:** Список запланированных Operations (из TechMap) на ближайший период.
*   **Выходные данные:** Отметки о выполнении, комментарии.
*   **Ключевые действия:** Просмотр задач, маркировка выполнения (Done), запрос переноса, отчет о проблеме.
*   **Связанные сущности:** Task, Operation.

### 1.7 Отклонения и Решения

#### 1.7.1 Реестр отклонений
*   **Экран:** Журнал отклонений (Deviations Log)
*   **Назначение:** Управление инцидентами, влияющими на результат.
*   **Кто пользователь:** Консультант, Агроном, Менеджер.
*   **Входные данные:** Список Deviations, статусы, критичность.
*   **Выходные данные:** Выбранное отклонение для разбора.
*   **Ключевые действия:** Фильтрация, сортировка по убыванию риска.
*   **Связанные сущности:** Deviation.

#### 1.7.2 Карточка отклонения
*   **Экран:** Разбор отклонения (Deviation Review)
*   **Назначение:** Анализ причины и принятие решения.
*   **Кто пользователь:** Консультант (решение), Клиент (согласование критических).
*   **Входные данные:** Детали Deviation, контекст TechMap, прогноз влияния (Impact Analysis).
*   **Выходные данные:** DecisionRecord (принятое решение).
*   **Ключевые действия:** Оценка ущерба, выбор варианта действия (Mitigation Strategy), фиксация решения.
*   **Связанные сущности:** Deviation, DecisionRecord.

### 1.8 Результат и Эффект

#### 1.8.1 Фактический урожай и Эффект
*   **Экран:** Расчет эффективности (Performance Calculator)
*   **Назначение:** Финальная фиксация результата и расчет вознаграждения.
*   **Кто пользователь:** Консультант, Финансист.
*   **Входные данные:** Actual Yield (факт), Baseline, цены реализации, условия контракта.
*   **Выходные данные:** Performance Report, расчет Fee.
*   **Ключевые действия:** Ввод фактических данных, запуск расчета, генерация отчета.
*   **Связанные сущности:** PerformanceContract, YieldRecord.

## 2. Стратегия

### 2.1 Стратегический Dashboard
*   **Экран:** Портфельный обзор (Strategy Overview)
*   **Назначение:** Верхнеуровневый взгляд на бизнес и риски.
*   **Кто пользователь:** Топ-менеджмент, Руководитель практики.
*   **Входные данные:** Агрегированные данные по всем Harvest Plans, финансовый прогноз, тепловая карта рисков.
*   **Выходные данные:** Стратегические корректировки.
*   **Ключевые действия:** Анализ портфеля, drill-down до проблемных зон.
*   **Связанные сущности:** Portfolio, StrategyRisk.

## 3. Экономика

### 3.1 Экономика урожая
*   **Экран:** Экономический анализ (Crop Economics)
*   **Назначение:** Понимание рентабельности конкретного плана.
*   **Кто пользователь:** Экономист, Консультант.
*   **Входные данные:** Бюджет TechMap, фактические затраты, прогноз выручки.
*   **Выходные данные:** P&L прогноз.
*   **Ключевые действия:** Сценарный анализ (что-если), контроль маржинальности.
*   **Связанные сущности:** Budget, Transaction.

## 4. Финансы

### 4.1 Начисления performance-вознаграждений
*   **Экран:** Биллинг (Billing & Fees)
*   **Назначение:** Управление денежными потоками от консалтинга.
*   **Кто пользователь:** Финансовый менеджер.
*   **Входные данные:** Расчетные Performance Fees из модуля Результат.
*   **Выходные данные:** Инвойсы, акты.
*   **Ключевые действия:** Генерация счетов, контроль оплаты.
*   **Связанные сущности:** Invoice, Payment.

## 5. GR (Государственные отношения)

### 5.1 Реестр госорганов
*   **Экран:** Справочник регуляторов (Regulators Registry)
*   **Назначение:** База знаний по требованиям.
*   **Кто пользователь:** Юрист, GR-специалист.
*   **Входные данные:** Список органов, типы требований.
*   **Выходные данные:** Актуальные требования.
*   **Ключевые действия:** Поддержание актуальности требований.
*   **Связанные сущности:** Authority, Requirement.

### 5.2 Проверки и инспекции
*   **Экран:** Календарь проверок (Inspections Calendar)
*   **Назначение:** Управление рисками проверок.
*   **Кто пользователь:** Менеджер хозяйства, Юрист.
*   **Входные данные:** Плановые проверки, предписания.
*   **Выходные данные:** План подготовки.
*   **Ключевые действия:** Регистрация проверки, чек-лист подготовки, отчет о результатах.
*   **Связанные сущности:** Inspection, ComplianceTask.

## 6. Производство (Грипил)

### 6.1 Склады и хранение
*   **Экран:** Управление запасами (Inventory)
*   **Назначение:** Учет физического продукта.
*   **Кто пользователь:** Завсклад, Производственник.
*   **Входные данные:** Остатки сырья и готовой продукции.
*   **Выходные данные:** Движения ТМЦ.
*   **Ключевые действия:** Приемка, списание, инвентаризация.
*   **Связанные сущности:** StockItem, Warehouse.

## 7. Знания

### 7.1 База знаний
*   **Экран:** Навигатор знаний (Knowledge Explorer)
*   **Назначение:** Поиск решений и лучших практик.
*   **Кто пользователь:** Все роли.
*   **Входные данные:** Индексированные статьи, кейсы, техкарты прошлых лет.
*   **Выходные данные:** Найденное решение.
*   **Ключевые действия:** Поиск, просмотр статей, связывание с текущими задачами.
*   **Связанные сущности:** Article, CaseStudy.

## 8. Настройки

### 8.1 Пользователи и роли
*   **Экран:** Управление доступом (Access Control)
*   **Назначение:** Администрирование системы.
*   **Кто пользователь:** Администратор.
*   **Входные данные:** Список пользователей, ролевая модель.
*   **Выходные данные:** Права доступа.
*   **Ключевые действия:** Создание пользователей, назначение ролей.
*   **Связанные сущности:** User, Role, Permission.
