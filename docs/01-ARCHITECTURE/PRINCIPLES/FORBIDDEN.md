# FORBIDDEN RULES

## Статус документа
Данный документ определяет **КАТЕГОРИЧЕСКИЕ ЗАПРЕТЫ** в проекте.

- Является частью FOUNDATION
- Обязателен к исполнению всеми ролями
- Нарушение = немедленная остановка процесса

---

## 1. Запреты на мышление и интерпретацию

Категорически запрещено:

- Делать предположения при отсутствии данных
- Заполнять пробелы «логикой», «опытом» или «здравым смыслом»
- Интерпретировать неописанное поведение как разрешённое
- Считать молчание в документации свободой действий

Фраза  
> «Так логичнее»  
считается **архитектурным нарушением**.

---

## 2. Запреты на архитектуру

Запрещено:

- Создавать новые модули без явной фиксации
- Менять границы ответственности модулей
- Добавлять скрытые зависимости
- Объединять модули «для упрощения»
- Расширять архитектуру без Admission-процедуры

---

## 3. Запреты на реализацию

CODER запрещено:

- Писать код без утверждённой задачи
- Менять публичные интерфейсы без миграции
- Оптимизировать производительность без запроса
- Рефакторить «заодно»
- Вносить изменения, не отражённые в memory-bank

---

## 4. Запреты на процесс

Запрещено:

- USER → CODER взаимодействие
- Пропуск роли TECHLEAD
- Начинать кодинг до архитектурного CHECK
- Использовать чат как источник истины
- Переносить решения между сессиями без фиксации

---

## 5. Реакция на нарушение

При любом нарушении:

1. Процесс немедленно останавливается
2. TECHLEAD фиксирует нарушение
3. USER уведомляется
4. Решение не принимается независимо от результата

---

## 6. Ключевой принцип

> **Лучше остановить процесс,  
> чем принять неправильное решение.**
