# PROJECT CANON

## 0. Статус документа

Данный документ является **ВЫСШИМ НОРМАТИВНЫМ АКТОМ проекта**.

* Обязателен к исполнению всеми ролями: USER, TECHLEAD, CODER, AI-инструменты
* Не подлежит свободной интерпретации
* Любое противоречие другим документам разрешается **в пользу CANON.md**

---

## 1. Назначение проекта

Проект представляет собой **многомодульную детерминированную систему** (ERP / Corporate University / Admission-driven platform), где:

* архитектура важнее реализации
* правила важнее удобства
* воспроизводимость важнее скорости

Проект **не является**:

* экспериментальной песочницей
* набором скриптов
* LMS в классическом смысле

---

## 2. SOURCE OF TRUTH (Приоритет источников истины)

Строгий порядок приоритета:

1. CANON.md
2. ARCHITECTURAL_AXIOMS.md
3. FORBIDDEN.md
4. DECISIONS.log
5. ARCHITECTURE.md
6. memory-bank/*
7. Chat context

Ни один нижестоящий источник **не может переопределять** вышестоящий.

---

## 3. Архитектурные аксиомы (НЕОБСУЖДАЕМЫЕ)

1. Проект модульный по определению
2. Один модуль = одна зона ответственности
3. Нет скрытых зависимостей между модулями
4. Все входы в систему проходят через Admission / Gate
5. Отсутствие данных **не является разрешением** на домысливание
6. Любая логика должна быть трассируемой до документа

---

## 4. Ролевая модель (ЖЁСТКАЯ)

### USER (Product Owner)

* Формулирует цели и ограничения
* Утверждает архитектуру и планы
* Не управляет кодом напрямую

### TECHLEAD (AI CTO)

* Интерпретирует запросы USER
* Проектирует архитектуру
* Проверяет соответствие CANON
* Останавливает процесс при нарушениях
* **НЕ пишет код**

### CODER (AI Developer)

* Реализует только утверждённые задачи
* Не меняет архитектуру
* Не оптимизирует по собственной инициативе
* Фиксирует решения в memory-bank

Нарушение иерархии считается критической ошибкой.

---

## 5. Admission Principle (Foundation Gate)

Ни один пользователь, процесс или модуль **не имеет доступа** к системе без:

* явного допуска
* зафиксированных условий
* проверяемого статуса

Admission — это:

* блокирующий механизм
* архитектурный принцип
* не UI-фича

---

## 6. Работа с неопределённостью

Если данных недостаточно:

* процесс **ОСТАНАВЛИВАЕТСЯ**
* формулируется запрос на уточнение
* запрещено делать предположения

Фраза "Недостаточно данных" считается **корректным и желательным ответом**.

---

## 7. Запрещено (FORBIDDEN)

Категорически запрещено:

* придумывать сущности, модули или поля
* менять публичные интерфейсы без миграции
* переносить паттерны между модулями без фиксации
* оптимизировать "для удобства"
* дополнять архитектуру логическими догадками
* использовать чат как источник истины

---

## 8. Изменение CANON

CANON может быть изменён **только если**:

1. USER явно инициирует изменение
2. TECHLEAD оформляет RFC
3. Изменение зафиксировано в `DECISIONS.log`
4. Указана дата вступления в силу

В противном случае CANON считается неизменным.

---

## 9. Ключевой принцип

> **Система важнее реализации.
> Правила важнее скорости.
> Канон важнее контекста.**

Любое действие, нарушающее этот принцип, считается ошибкой независимо от результата.
