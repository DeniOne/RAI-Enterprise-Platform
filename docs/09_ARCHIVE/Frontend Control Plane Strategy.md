# RAI Enterprise — Frontend Control Plane Strategy

> [!WARNING]
> **STATUS: SUPERSEDED BY [Master Frontend Architecture](file:///f:/RAI_EP/docs/00_STRATEGY/Master%20Frontend%20Architecture.md)**
> Данный документ является архивным и оставлен для истории эволюции проекта.

Этот документ определяет стратегию интерфейса управления (Control Plane) для институциональных пользователей RAI Enterprise Platform.

---

### Executive Summary
Данная стратегия переводит фронтенд системы из категории классических ERP в категорию **AI Control Plane**. Основной фокус смещен с ввода данных на **мониторинг когнитивных процессов, валидацию решений ИИ и управление рисками** в реальном времени.

**Ключевые принципы:**
*   **Explainability-First**: Ни одно решение ИИ не выводится без контекста "Почему?".
*   **Ledger-Fact Alignment**: Все финансовые и производственные действия неразрывно связаны с транзакционным Ledger-ядром.
*   **Institutional Governance**: Поддержка mTLS, M-of-N гейтов и неизменяемого аудиторского следа (Merkle Tree).

---

🔷 СТРУКТУРА МЕНЮ (NAV-TREE)

Ниже представлена иерархия интерфейса управления, реализованная в бэк-офисном фронтенде:

1.  **УПРАВЛЕНИЕ УРОЖАЕМ**
    *   **Обзор**: Сводная панель (Dashboard)
    *   **Хозяйства и Контрагенты**: Реестр хозяйств, Контрагенты, Поля / Объекты, История сезонов
    *   **Планы урожая**: Черновики, Активные, Архив
    *   **Техкарты Урожая**: Проектирование, Активные, Замороженные, Архив
    *   **Исполнение Техкарт**: Контур Агронома (Back-office), Контур Менеджера Хозяйства (Back-office)
    *   **Отклонения и Решения**: Выявленные отклонения, Разбор, Принятые решения
    *   **Результат и Эффект**: Фактический урожай, Сравнение с планом, Performance-оплата

2.  **СТРАТЕГИЯ**
    *   Стратегический обзор, Портфель планов урожая, Карта рисков, Сценарное моделирование, Журнал стратегических решений

3.  **ЭКОНОМИКА**
    *   Экономика урожая, Агрегация эффекта, Юнит-экономика, Safety Net контроль, Прогноз экономики

4.  **ФИНАНСЫ**
    *   Денежные потоки, Performance-начисления, Счета и расчёты, Финансовая отчетность

5.  **GR (Governmental Relations)**
    *   Регуляторный контекст, Ограничения, Контракты, Комплаенс, Юридически значимые решения

6.  **ПРОИЗВОДСТВО (ГРИПИЛ)**
    *   Сырьё и закупки, Хранение, Производство, Контроль качества, Логистика, Производственная аналитика

7.  **ЗНАНИЯ**
    *   База знаний, Кейсы урожая, Паттерны решений, Эволюция техкарт

8.  **НАСТРОЙКИ**
    *   Пользователи и роли, Права доступа, Аудит, Интеграции, Системные параметры

### 1. ОБЗОР (Super-Dashboard)
**Назначение**: Глобальная когнитивная панель для мониторинга всей экосистемы консалтинга.
**Уровень**: Level F (Institutional).
**Тип**: Institutional / Strategic.

**Дерево экранов**:
```
/overview
  /global-consciousness (Состояние ИИ-ядра)
  /risk-heatmap (Тепловая карта рисков по всем контрактам)
  /alerts (Критические отклонения, требующие вмешательства бэк-офиса)
  /cross-tenant (Сводка по всем агрохолдингам для аудиторов)
```

**Структура (Dashboard)**:
- **A. Контекст**: Global Ecosystem Status, Active Contracts: N, Model: Oracle v2, Trace ID: Root.
- **B. Зона**: 
  - **Global Consciousness Gauge**: Визуализация "уверенности" системы в текущей стабильности всех техкарт.
  - **Risk Heatmap**: Матрица (Вероятность/Ущерб) по всем активным сезонам.
  - **Consulting Flow**: Живая лента событий (Инициация плана -> Изменение в Ledger -> Сигнал агронома).
- **C. Действия**: Переход к проблемному тенанту, Инициация экстренного аудита, Заморозка рискованных контрактов.

**AI-Компоненты**:
- `GlobalConsciousnessGauge`: Анимированный индикатор осознанности AI.
- `RiskHeatmap`: Интерактивная матрица рисков.
- `AlertStream`: Лента аномалий с мгновенным доступом к Explainability.

### 2. УПРАВЛЕНИЕ УРОЖАЕМ

#### 2.1 Хозяйства и Контрагенты
**Назначение**: Управление мастер-данными объектов консалтинга. Это фундамент для Ledger-изоляции.
**Уровень**: Level A-B.
**Тип**: Operational.

**Дерево экранов**:
```
/harvest/registry
  /farms
     /:id (Карточка хозяйства)
  /contractors
  /fields
     /:id/history (История по конкретному полю)
```

**Структура (Карточка Поля)**:
- **A. Контекст**: Field ID, Region, Soil Status, Season 2026.
- **B. Зона**: 
  - **Interactive Map**: Спутниковая подложка с границами.
  - **Integrity Badge**: Статус проверки границ через Merkle Tree (Level F).
- **C. Действия**: Редактирование границ (требует подтверждения риск-менеджером), Привязка к техкарте.

---

#### 2.2 Планы урожая
**Назначение**: Проектирование целевых показателей сезона. Переход от "хотелок" к "математической модели".
**Уровень**: Level B (Deterministic).
**Тип**: Strategic.

**Дерево экранов**:
```
/harvest/plans
  /drafts
     /:id/optimizer (AI-оптимизатор параметров плана)
  /active
  /archive
```

**Структура (Оптимизатор Плана)**:
- **A. Контекст**: Plan ID, Crop, Target Yield, Model v1.2.
- **B. Зона**: 
  - **Decision Panel**: Слайдеры параметров (Удобрения vs Риск vs ROI).
  - **Simulated Outcome**: График вероятности достижения урожая (Monte Carlo).
  - **Why this plan?**: Explainability блок — почему AI рекомендует именно эти нормы.
- **C. Действия**: Launch Simulation, Commit to Ledger (Gate: M-of-N Approval), Export to PDF.

**API Contract (Recommendation)**:
```json
{
  "recommendation": "Increase NPK by 15%",
  "why": "Low soil moisture predicted for July, requiring early root development",
  "factors": ["Weather forecast", "Soil sensors", "Historical yield"],
  "confidence": 0.89,
  "traceId": "trc_99281"
}
```

#### 2.3 Техкарты Урожая
**Назначение**: Технологический "чертеж" сезона. Превращение стратегии в последовательность операций.
**Уровень**: Level C (Contradiction Engine).
**Тип**: Operational / Governance.

**Дерево экранов**:
```
/harvest/techmaps
  /design (Визуальный конструктор операций)
  /active (Карты в работе)
  /frozen (Замороженные из-за рисков)
  /:id/explainability (Почему выбрана эта химия/норма)
```

**Структура (Дизайн техкарты)**:
- **A. Контекст**: TechMap ID, Crop, Soil Type, Model Version, Trace ID.
- **B. Зона**: 
  - **Operation Timeline**: Ганта-подобная лента работ.
  - **Agronomic Rationale**: Блок обоснования каждой операции (Почему именно этот гербицид?).
  - **Risk Guardrails**: Визуализация ограничений (например, "Не более 200кг NPK").
- **C. Действия**: Добавление операции, Валидация через Conflict Engine, Публикация (Gate: mTLS Signature).

---

#### 2.4 Исполнение Техкарт (Контур бэк-офиса)
**Назначение**: Мониторинг реальности. Это НЕ интерфейс хозяйства, это интерфейс контроля для штатных сотрудников RAI.
**Уровень**: Level D (Industrial Hardening).
**Тип**: Operational.

**Экраны**:
1. **Контур Агронома-консультанта (Back-office)**:
   - Поток данных с датчиков и спутников.
   - Валидация агрономических допущений.
   - Фиксация аномалий (болезни, вредители).
2. **Контур Менеджера сопровождения (Back-office)**:
   - GPS-трекинг техники (физическое присутствие в поле).
   - Фото-фиксация работ (артефакты исполнения).
   - Сверка фактических затрат ресурсов с нормами в Ledger.

---

#### 2.5 Отклонения и Решения
**Назначение**: Управление конфликтами между планом и фактом. 
**Уровень**: Level C-E (Regenerative).
**Тип**: Governance.

**Дерево экранов**:
```
/harvest/deviations
  /active (Список выявленных аномалий)
  /triage (Разбор: ИИ-анализ причин)
  /decisions (Принятые решения и их след в Ledger)
```

**Структура (Экран Разбора)**:
- **A. Контекст**: Deviation ID, Type: Under-fertilized, Severity: High, Trace ID.
- **B. Зона**: 
  - **Explainability Panel**: Что пошло не так (Data Drift vs Human Error).
  - **Counterfactual**: "Если мы не вмешаемся, потери составят -15% урожая".
  - **Override Modal**: Если агроном не согласен с AI — выбор своего пути с Liability Tag.
- **C. Действия**: Accept AI Decision, Override (Sign with mTLS), Escalate to M-of-N.

#### 2.6 Результат и Эффект
**Назначение**: Финализация сезона. Расчет "бонуса за успех".
**Уровень**: Level E-F.
**Тип**: Governance / Institutional.

**Дерево экранов**:
```
/harvest/results
  /actual (Фактическая урожайность по данным элеваторов)
  /comparison (План-факт анализ)
  /performance-fee (Расчет оплаты по результату)
```

### 4. ЭКОНОМИКА
**Назначение**: Управление когнитивными моделями доходности и устойчивости. Мир "прогнозов и симуляций".
**Уровень**: Level E (Regenerative).
**Тип**: Governance.

**Дерево экранов**:
```
/economy/harvest-economics (Экономический профиль урожая)
/economy/safety-net (Контроль "защитной сетки" — лимиты убытков)
/economy/unit-economics (Разбор прибыльности каждого гектара)
/economy/projections (Прогноз финансового результата сезона)
```

**Структура (Safety Net)**:
- **A. Контекст**: Entity, Risk Threshold, Model: SafetyNet v2.1, Trace ID.
- **B. Зона**: 
  - **Risk Corridor**: График допустимых отклонений затрат.
  - **Drift Alerts**: Список операций, которые "уводят" экономику в красную зону.
  - **Regeneration Score**: Как текущие решения влияют на SRI (устойчивость).
- **C. Действия**: Корректировка лимитов, Инициация ре-симуляции сезона.

---

### 5. ФИНАНСЫ
**Назначение**: Управление финансовыми фактами. Мир "реестров, счетов и Ledger-записей".
**Уровень**: Level D-F (Ledger Hardening).
**Тип**: Operational.

**Дерево экранов**:
```
/finance/cash-flow (Реальное движение ДС)
/finance/accruals (Расчет Performance-оплаты на основе Ledger)
/finance/invoicing (CRM-flow: Выставление счетов и контроль оплат)
/finance/reporting (Финансовые отчеты для бэк-офиса)
```

**Интерфейс CRM-Менеджера (Invoicing)**:
- **A. Контекст**: Client ID, Contract ID, Outstanding Balance, Trust Status.
- **B. Зона**: 
  - **Receivables Table**: Дебиторка с разбивкой по этапам техкарт.
  - **Transaction History**: Список проводок в Ledger с хешами.
- **C. Действия**: Инициирование транзакции (Generate Invoice), Маркировка оплаты (Gate: Verification), Связь с CMR.

---

### 6. GR (Governmental Relations)
**Назначение**: Комплаенс и юридическая чистота.
**Уровень**: Level F (Institutional).
**Тип**: Institutional.

**Дерево экранов**:
```
/gr/regulatory-feed (Актуальные ограничения и законы)
/gr/contracts (Цифровые контракты и цифровые подписи)
/gr/compliance (Статус соответствия регуляторным грейдам)
/gr/juridical-decisions (Журнал юридически значимых решений)
```

**Структура (Contracts)**:
- **A. Контекст**: Agreement ID, Parties, mTLS Auth Status, Trace ID.
- **B. Зона**: 
  - **Immutable Document View**: Отображение текста контракта с проверкой целостности.
  - **Interaction Log**: Кто и когда подписывал/редактировал (Auditable).
- **C. Действия**: Электронная подпись (KMS/mTLS), Выгрузка архива для регулятора.

**Уровень**: Level D (Industrial Hardening).
**Тип**: Operational.

**Дерево экранов**:
```
/production/sourcing (Сырье и закупки)

```

**Структура (Логистика)**:
- **A. Контекст**: Batch ID, Carrier, Destination, Status: In-Transit, Trace ID.
- **B. Зона**: 
  - **Live Tracking Map**: Гео-позиция грузов.
  - **Certificate of Origin**: Цифровой паспорт качества (аттестован AI).
- **C. Действия**: Инициация новой отгрузки, Блокировка партии (если нарушен температурный режим).


**Структура (Audit Explorer)**:

- **C. Действия**: Экспорт аудиторского следа, Проверка целостности базы.

---

🔷 ЗАКЛЮЧЕНИЕ И КОГНИТИВНЫЕ ПРИНЦИПЫ

1. **Единая точка истины**: Любое изменение в Control Plane (особенно в модуле "Финансы" и "Отклонения") моментально генерирует Trace ID и запись в Ledger.
2. **Explainability-first**: Пользователь (агроном-консультант или CFO) никогда не видит решение AI без кнопки "Почему?".
3. **Бесшовный Flow**: 
   - Выявили отклонение в **Исполнении** -> Создали разбор в **Отклонениях** -> Обновили риски в **Стратегии** -> Отразили потенциальный убыток в **Экономике**.
4. **Безопасность**: Все действия в модулях GR и Настройки требуют mTLS-авторизации и (опционально) M-of-N подтверждения.
