# RAI Enterprise Platform
## Institutional UI Audit — Final Consolidated Report

---

## 1. Информационная Архитектура (IA)

### 1.1. Карточка контрагента — структурный уровень
> [!CAUTION]
> **🔴 Критическая проблема №1: Неверная модель H1**

**Сейчас:**
* В H1 выводится полное юридическое наименование.
* Оно обрезается.
* Дублируется.
* Перегружает шапку.

**Почему это плохо:**
* Пользователь работает с кратким именем.
* Обрезанный H1 выглядит как баг.
* Длинное юр. имя ломает композицию.

**Решение:**
* **H1** → Краткое имя (ООО «ЧЕРЕМШАНАГРО»)
* **Подзаголовок** → Полное юридическое наименование
* **Справа** → ИНН / Юрисдикция
* *ИНН и статус должны быть в верхней зоне быстрого сканирования.*

**Приоритет:** Критический

---

### 1.2. Смешение уровней данных в шапке
**Сейчас в одной строке:**
`Тип субъекта` | `Статус` | `Этап проверки` | `Проверка СБ` | `Холдинг` | `Governance-информация`

**Это разные уровни, которые нужно разделить визуально:**

| Слой | Что туда относится |
| :--- | :--- |
| **Identity** | Тип, юрисдикция, ИНН |
| **Governance** | В реестре холдинга |
| **Compliance** | Проверка СБ |
| **Process** | Этап верификации |

---

### 1.3. Синдром «вечной формы» (Критическая UX-проблема)
Все данные выглядят как input-поля. Это создаёт:
* ложный аффорданс
* ощущение edit-mode
* визуальный шум
* снижение доверия

**Требуется ввести явный toggle:**
* 🔘 **Редактировать**

**View mode:**
* Label
* Value (без рамок)

**Edit mode:**
* Input
* Save / Cancel

*Это ключевой шаг к institutional UX.*

**Приоритет:** Критический

---

### 1.4. Масштабируемость (Структурный риск)
**Сейчас:**
* Банковские счета — форма одного счета
* ЛОПР — форма одного лица
* Связанные активы — форма добавления

*При 10+ записей UI сломается.*

**Правильная модель:**
В каждом табе:
1. Таблица существующих записей
2. Кнопка “Добавить”
3. Inline-редактирование или drawer

*Без этого система не масштабируется.*

**Приоритет:** Критический

---

### 1.5. “Предпросмотр иерархии” — эффект незавершённости
**Текст:** `INTERACTIVE GRAPH COMING SOON` — это staging-артефакт. В продакшене так быть не должно.

**Решение (Empty state):**
* Иконка графа
* Текст: «Иерархия не построена»
* CTA: «Построить структуру»

**Приоритет:** Высокий

---

### 1.6. Cross-linking
Связанные активы не кликабельны. Любая связанная сущность должна вести в свой модуль.

*Это принцип связанной системы.*

**Приоритет:** Высокий

## 2. Визуальная система

### 2.1. Слишком мягкий SaaS-стиль
**Сейчас:**
* большие радиусы
* мягкие тени
* карточки в карточках
* бледная палитра

**RAI — control system. Нужна визуальная дисциплина:**
* Radius ≤ 12px
* Минимум теней
* Чёткие границы
* Строгая grid

**Приоритет:** Высокий

---

### 2.2. ALL CAPS — shouty UI
`«КОРПОРАТИВНАЯ СТРУКТУРА»` — это утомляет.

**Решение:**
* Sentence case
* SemiBold (600)
* spacing вместо Caps

---

### 2.3. Контраст
**Проблема:**
* Labels слишком светлые
* Мелкий кегль

**Исправление:**
* Цвет: `#6B7280`
* Размер: `13px+`
* Вес: `weight 500`
* *Проверить WCAG AA.*

---

### 2.4. Конфликт CTA
Primary (чёрный) используется и для `Save`, и для `“+ Новый счет”`. Иерархия действий ломается.

**Правильная система:**
1. **Primary** → Save
2. **Secondary** → Add
3. **Ghost** → Minor

---

### 2.5. Пропорции полей
БИК и название банка одинаковой ширины. Поле должно визуально подсказывать длину данных.

*Использовать grid-col-span.*

---

### 2.6. Иконки
* Разная толщина линий
* Неконсистентность
* Возможные retina-артефакты

*Нужен единый SVG-набор.*

---

## 3. UX и Взаимодействие

### 3.1. Кликабельность строк
Строки таблиц не выглядят интерактивными.
**Добавить:**
* hover фон
* chevron
* pointer

---

### 3.2. Hit-area
Breadcrumb стрелка, иконки, Date-picker: *Минимум 44×44 px.*

---

### 3.3. График “План vs Факт”
Без осей — не аналитика.
**Добавить:**
* оси
* сетку
* легенду
* tooltip

---

### 3.4. Debug-информация
`SYSTEM CANONICAL`, `Simulate: CEO` — это staging-артефакты. Спрятать за Dev-mode.
