# ДИАГНОСТИКА СБОЕВ И ПРОБЛЕМАТИКА ВЗАИМОДЕЙСТВИЯ CRM-АГЕНТА

## 1. Назначение

Этот документ фиксирует проблематику взаимодействия `CRM-агента` на реальном пользовательском сценарии:

- регистрация нового контрагента по ИНН;
- работа с уже существующим контрагентом;
- прохождение CRM write-path через AI-чат.

Документ нужен не для описания happy-path, а для понимания:

- что именно сломалось;
- почему проблема была двойной;
- какие соседние риски ещё могут проявиться;
- как такие сбои диагностировать дальше.

Routing и ownership-ожидания для этой диагностики нужно сверять с:

- [INSTRUCTION_ORCHESTRATOR_ROUTING_AND_AGENT_SELECTION.md](./INSTRUCTION_ORCHESTRATOR_ROUTING_AND_AGENT_SELECTION.md)
- [INSTRUCTION_AGENT_PROFILE_CRM_AGENT.md](./AGENT_PROFILES/INSTRUCTION_AGENT_PROFILE_CRM_AGENT.md)

---

## 2. Когда применять

Применять всегда, когда:

- `CRM-агент` отвечает не тем действием, которое ожидал пользователь;
- по CRM-операции в UI показывается ложный успех;
- карточка контрагента или CRM-объекта создаётся с пустыми полями;
- запрос пользователя уходит в `PendingAction`, хотя действие было инициировано человеком напрямую;
- есть подозрение, что `DADATA`, router, risk policy, composer или domain services сработали несогласованно.

---

## 3. Краткое описание инцидента

Реальный сценарий был таким:

- пользователь написал в чат: зарегистрировать контрагента по ИНН;
- система классифицировала сценарий как CRM-регистрацию;
- UI показал экран в духе `Контрагент зарегистрирован`;
- при этом в окне результата оказались `undefined` вместо имени контрагента и `partyId`;
- пользователь визуально получил сигнал об успехе, хотя корректного завершения операции не произошло.

Это опасный тип сбоя, потому что:

- ошибка не выглядит как ошибка;
- пользователь видит позитивный исход;
- в системе может не быть реально созданного объекта;
- в CRM-контуре возникает рассинхронизация между ожиданием и фактом.

---

## 4. Как должен работать правильный сценарий

Для запроса вроде:

- `зарегистрируй контрагента по ИНН 2610000615`

правильный путь такой:

1. router определяет `crm_agent / register_counterparty`;
2. из сообщения извлекается ИНН;
3. CRM-инструмент сначала проверяет, нет ли уже существующего контрагента с этим ИНН;
4. если дубля нет, идёт lookup через `DADATA`;
5. из lookup берутся реквизиты и наименование;
6. собирается `registrationData`;
7. создаётся карточка `party`;
8. в UI возвращаются реальные поля:
   - `partyId`
   - `legalName`
   - `inn`
   - `source`
   - `lookupStatus`
9. если контрагент уже существует, дубль не создаётся, а возвращается existing result.

---

## 5. Что произошло фактически

На инциденте сломались сразу два слоя.

### 5.1 Первая проблема: write-path был заблокирован, хотя действие уже запросил человек

По старому поведению:

- любой `CRM WRITE` считался требующим дополнительного подтверждения;
- прямой запрос пользователя из чата не трактовался как уже подтверждённое действие;
- вместо исполнения создавался `PendingAction`.

То есть система вела себя так, как будто:

- пользователь попросил зарегистрировать контрагента;
- но это всё ещё не считается достаточным основанием для выполнения;
- нужен ещё один отдельный этап подтверждения.

Проблема в этом месте была не в `DADATA`, а в том, что write-path тормозился раньше фактического доменного действия.

### 5.2 Вторая проблема: UI-композер интерпретировал заблокированный результат как успешный

После блокировки:

- runtime возвращал технический результат вида `riskPolicyBlocked`;
- composer не распознавал его как blocked state;
- CRM-окно строилось так, будто пришёл нормальный `RegisterCounterpartyResult`;
- поля `legalName` и `partyId` читались из payload, где их не было;
- пользователь видел ложный экран успеха и `undefined`.

Это была более опасная часть сбоя, потому что она искажала реальное состояние системы.

---

## 6. Почему это именно двойная проблема

Проблема была не в одном месте.

Если бы сломался только первый слой:

- пользователь увидел бы честный статус `требуется подтверждение`;
- операция не выполнилась бы, но хотя бы без обмана.

Если бы сломался только второй слой:

- данные могли бы выполниться, но рендер был бы плохим.

Фактически сломались оба слоя одновременно:

1. исполнение не завершилось как полноценный CRM-action;
2. интерфейс всё равно показал результат как будто всё прошло успешно.

Именно поэтому инцидент выглядел особенно раздражающим:

- не было ни корректного выполнения;
- ни корректного отказа;
- ни корректного промежуточного статуса.

---

## 7. Корневые причины по слоям

### 7.1 Router и intent layer

На момент инцидента routing уже умел распознать CRM-сценарий.

То есть проблема была не в том, что запрос ушёл в совсем чужой агент.
Проблема была дальше по цепочке исполнения.

### 7.2 Risk policy layer

Риск-политика считала:

- `CRM WRITE` требует подтверждения;
- прямой пользовательский чат-запрос сам по себе не является достаточным подтверждением.

Это приводило к созданию `PendingAction` даже там, где пользователь явно и однозначно инициировал действие вручную.

### 7.3 Actor context layer

В исполняющий контур не прокидывался полноценный смысл пользовательского действия:

- кто именно инициировал действие;
- является ли это прямой ручной командой пользователя;
- нужно ли повторное human confirmation.

Из-за этого write-path не отличал:

- автономное действие агента;
- прямое действие человека через чат.

### 7.4 Response composition layer

Composer не различал:

- успешный CRM-result;
- blocked result;
- config-blocked result.

Из-за этого blocked payload попадал в CRM rich output как будто он успешный.

### 7.5 UX-trust layer

UI получил результат, который выглядел как окончательный успех.

Это уже не просто техническая ошибка, а риск доверия к системе:

- пользователь перестаёт понимать, можно ли верить статусам;
- критичные write-сценарии становятся недостоверными.

---

## 8. Что было исправлено в этом проходе

В текущем проходе были внесены такие исправления:

1. прямой пользовательский `CRM WRITE` больше не уходит в лишний `PendingAction`, если это ручной живой запрос;
2. в `CRM-агент` и runtime теперь прокидывается пользовательский контекст;
3. blocked result больше не отображается как успешная регистрация;
4. вместо ложного успеха UI теперь должен показывать честный статус:
   - `Требуется подтверждение`
   - или `Выполнение заблокировано`
5. добавлены целевые тесты на:
   - прямой пользовательский CRM write;
   - корректную реакцию composer на blocked result.

Важно:

- это исправляет конкретный обнаруженный дефект;
- но не закрывает всю проблематику CRM-взаимодействия целиком.

---

## 9. Возможные дальнейшие проблемы

Ниже перечислены риски, которые ещё могут проявиться даже после текущего исправления.

### 9.1 Ошибки на границе `DADATA`

Риски:

- `DADATA` вернула `NOT_FOUND`;
- `DADATA` вернула неполный профиль;
- `DADATA` временно недоступна;
- `DADATA` вернула данные, не совпадающие с ожидаемой схемой;
- контрагент существует, но у него нет достаточного набора реквизитов для автосоздания.

Что это означает:

- агент должен уметь честно деградировать;
- нельзя превращать такой сценарий в фальшивый успех;
- должен быть понятный путь ручного добора данных.

### 9.2 Дубли и коллизии по ИНН

Риски:

- контрагент уже есть в `party`, но оформлен с другим форматом `registrationData`;
- в системе есть несколько записей с одним и тем же ИНН;
- existing lookup нашёл не тот объект;
- есть `party`, но нет соответствующего CRM account;
- есть CRM account, но нет корректной связи с `party`.

Это отдельная проблема согласованности доменных сущностей.

### 9.3 Разрыв между `party` и CRM-слоем

Для агроконсалтинга CRM-контур связан не только с классическим аккаунтом клиента, но и с:

- агрохолдингами;
- хозяйствами;
- арендаторами техники и оборудования;
- поставщиками `СЗР`, семян, удобрений;
- сервисными организациями;
- контрагентами по логистике и сопровождению.

Риски:

- `party` создаётся, а CRM account не создаётся;
- CRM account создаётся без связи с `party`;
- тип контрагента определён неверно;
- downstream сценарии по аренде, продажам и консалтингу не понимают новый объект.

### 9.4 Недостаточная типизация intent-ов

Риски:

- запрос `зарегистрируй контрагента` и `обнови реквизиты контрагента` могут пересекаться;
- запросы на создание контакта, аккаунта, обязательства и связи могут ошибочно матчиться друг с другом;
- смежные формулировки пользователя могут приводить к неверному tool call.

То есть сам `CRM-агент` может быть подключён, но слой естественного языка ещё не достаточно устойчив.

### 9.5 Несогласованность risk policy

Текущая логика для CRM write-path должна быть строго выверена.

Оставшиеся риски:

- слишком широкий bypass write-блокировок;
- наоборот, лишние блокировки на ручных пользовательских действиях;
- неодинаковое поведение в legacy runtime и `agent-first-hybrid`;
- разные правила для одинаковых действий в чате и в классическом UI.

### 9.6 Ложные статусы в UI в других CRM-сценариях

Текущий инцидент пойман на регистрации контрагента.
Но такая же проблема принципиально может повториться для:

- создания связи контрагентов;
- создания CRM account;
- создания контакта;
- создания взаимодействия;
- создания обязательства;
- update/delete операций.

Нужно считать это не единичной аномалией, а целым классом риска:

- blocked result или degraded result не должен интерпретироваться как completed.

### 9.7 Нехватка clarification для CRM create/update

Сейчас у CRM-контура всё ещё ограниченный слой добора контекста.

Риски:

- агент будет пытаться исполнять операцию без полного набора полей;
- пользователю не будет понятно, чего именно не хватает;
- часть запросов станет зависать между `FAILED` и ручной догадкой.

### 9.8 Специфика аграрного бизнеса

Для нашего домена обычный CRM слишком узок.

Надо учитывать, что контрагентами являются не только абстрактные “клиенты”, а, например:

- хозяйства;
- агрохолдинги;
- производители;
- арендаторы техники;
- поставщики `СЗР`;
- партнёры по консалтингу;
- подрядчики по полевым работам;
- перевозчики;
- лаборатории;
- дилеры оборудования.

Риск:

- если agent contracts и CRM data model будут ориентированы только на generic CRM, то реальные доменные сценарии агробизнеса начнут ломать классификацию и UI.

---

## 10. Что нужно проверять в следующих прогонах

На следующих тестовых проходах нужно обязательно прогонять минимум такие сценарии:

1. регистрация нового контрагента по ИНН с успешным `DADATA lookup`;
2. регистрация уже существующего контрагента по ИНН без создания дубля;
3. `DADATA NOT_FOUND` с честным сообщением и без ложного успеха;
4. `DADATA ERROR` с корректной деградацией;
5. создание связи между существующими контрагентами;
6. создание CRM account на базе существующего `party`;
7. создание контакта и обязательства по существующему аккаунту;
8. проверка blocked/config-blocked сценариев без ложного `COMPLETED`;
9. прогон одного и того же сценария в:
   - legacy runtime;
   - `agent-first-hybrid`.

---

## 11. Минимальный набор последующих улучшений

Если продолжать развитие правильно, следующие шаги должны быть такими:

1. расширить golden-set для CRM intent-ов;
2. добавить диагностику blocked/degraded/completed состояний по всем CRM write-tools;
3. усилить clarification flow для CRM create/update сценариев;
4. проверить связность `party` ↔ `CRM account` ↔ `contacts` ↔ `obligations`;
5. выделить отдельные тестовые сценарии для:
   - поставщиков `СЗР`;
   - арендодателей техники;
   - клиентов консалтинга;
   - аффилированных структур и холдингов;
6. убедиться, что UI не показывает позитивный статус без подтверждённого domain result.

---

## 12. Критические ошибки и запреты

Запрещено:

- считать blocked result успешным;
- показывать `COMPLETED`, если доменное действие не выполнено;
- создавать дубль контрагента без проверки existing registry;
- скрывать ошибки `DADATA` за общей фразой “готово”;
- смешивать `party` и `CRM account` как будто это одна и та же сущность;
- считать generic CRM-модель достаточной для аграрного домена без дополнительных проверок.

## 14. Что должно получиться на выходе

После чтения этого документа инженер должен ясно понимать:

- почему инцидент с регистрацией контрагента был не одной, а двумя ошибками сразу;
- почему доверие к UI в write-сценариях критично;
- какие соседние зоны риска у `CRM-агента` ещё не закрыты;
- какие сценарии обязательно прогонять дальше, чтобы довести CRM-контур до надёжного production-уровня.
