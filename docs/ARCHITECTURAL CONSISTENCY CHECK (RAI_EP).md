# CONTEXT (НЕ ИЗМЕНЯТЬ)
Проект: RAI Enterprise Platform (RAI_EP)

Стратегические аксиомы:
- RAI_EP ≠ CRM / ERP / BI / Бухгалтерия
- RAI_EP = Business Core / Brain Layer / Orchestration Layer
- Система строится вокруг доменов, процессов и жизненных циклов
- Фронт ОФС — только для руководства (тактика, стратегия, визуализация)
- Рядовые пользователи взаимодействуют ТОЛЬКО через Telegram-бот
- Service = IO (исполнение)
- Orchestrator = Brain (принятие решений)
- Domain ≠ Infrastructure
- Никакой «магии», только явные правила и состояния

# ТВОЯ ЗАДАЧА
Провести **архитектурную и системную проверку**, ответив СТРОГО по структуре ниже.

---

## 1. КЛАССИФИКАЦИЯ
Определи:
- Тип компонента: (Domain / Orchestrator / Service / Interface / Infrastructure / Integration)
- Его роль в системе (1–2 предложения)
- Является ли он:
  - Core
  - Supporting
  - Edge

---

## 2. ПРОВЕРКА КАНОНА (CHECKLIST)
Ответь **ДА / НЕТ / ЧАСТИЧНО** с кратким пояснением:

- Service = IO (без принятия решений)
- Orchestrator = Brain (централизует логику)
- Бизнес-правила НЕ находятся в контроллерах
- Доменная логика НЕ размазана по сервисам
- Компонент не знает лишних доменов
- Владение состоянием явно определено
- Статусы и переходы имеют бизнес-смысл
- **Multi-tenancy**: Права и владение данными (`companyId`) явно разграничены
- **Infrastructure Leak**: Домен не зависит напрямую от БД/Redis (используются адаптеры/интерфейсы)

---

## 3. ЦЕЛОСТНОСТЬ ПРОИЗВОДСТВА (PRODUCTION INTEGRITY)
Ответь **ДА / НЕТ / ЧАСТИЧНО**:

- **Backend Integrity**: Все ли компоненты (Service, Controller, Resolver, DTO, Repository) физически реализованы, зарегистрированы в модулях и соответствуют схеме БД?
- **Frontend Integrity**: Реализованы ли страницы, UI-компоненты и API-hooks в `apps/web`? Соответствуют ли они UI Design Canon?
- **Telegram Connectivity**: Реализована ли связка с ботом (Handlers, Commands, Templates)? Может ли конечный пользователь (полевой сотрудник) взаимодействовать с этой фичей?
- **Inter-module Cross-talk**: Реализованы ли физические связи между модулями (Integration services, Event listeners)? Нет ли изолированных "островов" логики?

---

## 4. АРХИТЕКТУРНЫЕ РИСКИ
Перечисли:
- Явные нарушения (если есть)
- Скрытые риски (что сломается при росте системы)
- Где возможна деградация через 3–6 месяцев

❗ Не предлагай решений — только фиксация рисков.

---

## 5. СТРАТЕГИЧЕСКАЯ СОВМЕСТИМОСТЬ
Ответь:
- Усиливает ли модуль стратегическую цель RAI_EP?
- Не превращает ли он систему в CRM / ERP-подобную?
- Не смещает ли центр тяжести из Business Core в “операционку”?

---

## 6. КРАСНЫЕ ФЛАГИ 🚨
Отметь, если применимо:
- Сервис принимает решения
- Отсутствует явный Orchestrator
- Логика «работает, но непонятно почему»
- Статусы используются как флаги
- Компонент «удобный сейчас, опасный потом»
